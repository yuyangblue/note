# 贪心

## 大白话引入

今天中午去食堂打饭，窗口排了三条队。你扫一眼：A 队 5 个人但每个人只打一个菜，B 队 3 个人但每个人都在犹豫点啥，C 队 4 个人速度中等。你选哪个队？

大概率选 A 队——虽然人最多，但每个人走得快，你等的时间最短。这就**贪心**：不看全局最优（也许 B 队阿姨突然加了个窗口变快了？），只看眼前最有利的选择。

贪心的核心就一句话：**每一步都选当前看起来最好的，最终得到全局最优**。

听起来很美好对吧？但问题来了：**为什么这样能行？** 大部分时候你随便贪心是会翻车的。这章我们不光学贪心怎么用，更要学**怎么判断一道题能不能贪心**。

---

## 思维启发：从"直觉"到"证明"

### 场景 1：找零钱

你收银，要给顾客找 36 块钱。手上有这些面额的硬币：**25、10、5、1**。

正常人怎么找？先拿一个 25，剩 11；再拿一个 10，剩 1；再拿一个 1。搞定，3 枚硬币。

**直觉**：每次拿能拿的最大面额，用的硬币数就最少。

**为什么对的？** 因为大面额是小面额的倍数（25=5×5, 10=2×5, 5=5×1），用大面额替换小面额永远不亏。这个性质叫**贪心选择性质**。

### 场景 2：换一套硬币

现在面额换成 **25、20、10、5、1**。找 40 块钱。

贪心：25 + 10 + 5 = 3 枚。但最优解是：20 + 20 = 2 枚。**贪心翻车了！**

**为什么这次不行？** 因为 20 和 25 没有倍数关系了。你选了最大的 25，反而错过了两个 20 的方案。

**关键洞察**：贪心不是无脑选最大/最小，是要确认**当前的最优选择不会堵死后面的路**。简单说就是：**这个选择不会反悔**。

---

### 怎么判断一道题能贪心？

两个性质缺一不可：

| 性质 | 什么意思 | 怎么检查 |
|------|----------|----------|
| **贪心选择性质** | 你这一步的最优选择，最终一定出现在全局最优解里 | 交换论证：把最优解换成贪心选择的方案，不会变差 |
| **最优子结构** | 做完当前选择后，剩下的子问题仍然可以独立地用同样的贪心策略解决 | 子问题和原问题是同类问题，只是规模小了 |

实战中的判断捷径：**如果一道题让你选一些东西，而且顺序不影响可行性（选完一批再想下一批），大概率能贪心。**

## 一、经典模型：区间调度（活动选择）

### 问题描述

你有 n 个活动，每个活动有开始时间 `s[i]` 和结束时间 `t[i]`。同一时间只能参加一个活动，问最多能参加几个。

### 思考流程（重点）

**第 1 步：看到题的第一反应**

"让我选最多的活动……暴力？把 n 个活动的所有子集枚举一遍？2^n 种，n=100 直接爆炸。"

**第 2 步：找贪心的直觉**

"怎么样参加的活动最多？想一下现实里我怎么安排——肯定选结束早的，因为结束早能空出更多时间给后面的活动。"

有三种直觉候选：
1. 选**开始最早**的 → 但一个活动从 0 开到 100，后面全堵死了
2. 选**持续时间最短**的 → 但一个活动可能跨在两个短活动之间，选了它反而丢了两个
3. 选**结束最早**的 → 好像有道理，早点空出时间来

**第 3 步：验证直觉**

直觉 3（结束最早）最有希望。怎么证明？

**交换论证法**：
- 假设最优解里第一个活动不是结束最早的
- 把最优解的第一个活动换成结束最早的那个
- 新的结束时间 ≤ 老的结束时间，所以后面的活动全都能塞进去
- 替换后解不会变差 → 所以贪心选结束最早一定在某个最优解里

**第 4 步：抽象成算法**

1. 按结束时间排序
2. 选第一个结束的活动
3. 跳过所有与它冲突的（开始时间 < 它的结束时间）
4. 对剩下的重复这个过程

### C++ 代码

```cpp
#include <bits/stdc++.h>
using namespace std;

struct Activity {
    int start, end; // 活动开始时间和结束时间
};

int main() {
    int n;
    cin >> n;
    vector<Activity> acts(n);
    for (int i = 0; i < n; i++) {
        cin >> acts[i].start >> acts[i].end;
    }

    // 关键：按结束时间从小到大排序
    // 为什么按结束时间？因为结束越早，留给后面的时间越多
    // 这是贪心选择——每次选的是「对后续约束最小」的
    sort(acts.begin(), acts.end(), [](const Activity &a, const Activity &b) {
        return a.end < b.end;
    });

    int cnt = 0;        // 已选活动数
    int lastEnd = 0;    // 上一个选中的活动的结束时间

    for (const auto &act : acts) {
        // 如果当前活动的开始时间 >= 上一个活动的结束时间，说明不冲突
        // 注意这里是 >=，因为活动结束的时刻和下一个开始的时刻重合不算冲突
        if (act.start >= lastEnd) {
            cnt++;              // 选它！
            lastEnd = act.end;  // 更新当前最后时间
        }
        // 冲突的话直接跳过——这就是贪心的「不做后悔选择」：放掉它不影响后续
    }

    cout << cnt << "\n";
    return 0;
}
```

### 为什么不能按开始时间排序？

你试想：活动 A [0, 100]，活动 B [1, 2]，活动 C [3, 4]。按开始时间排序会先选 A，然后 B 和 C 都冲突，答案=1。但最优解是选 B 和 C，答案是 2。**按开始时间贪心没有「反悔机制」，所以翻车。**

## 二、邻项交换法

### 问题：国王游戏（Luogu P1080）

国王和大臣们站成一排，每个人左手上写一个数 `a`，右手写一个数 `b`。每个人获得的奖赏是他**之前所有人左手数的乘积**除以他**右手数**（向下取整）。问如何排队使**最大奖赏最小**。

### 思考流程

**第 1 步：读题反应**

"最大值最小化——这个味道很二分答案。" 但仔细一想，二分的 check 函数不好写，因为你没法快速判断"能不能让最大奖赏 ≤ X"。

**第 2 步：放弃二分，想想排序能不能解决问题**

"队伍顺序由我决定，那本质上是个排序问题。但排序的依据是什么？不是简单按 a 或按 b。"

**第 3 步：考虑相邻两个人**

假设队伍里两个人 i 和 j 相邻。考虑它们交换位置会不会更好。

"我不需要知道全局最大的奖赏怎么算，我只需要比较 i 在前还是 j 在前，能让**这两个人里较大的奖赏**更小。"

- i 在前：奖赏_i = product / b_i, 奖赏_j = product × a_i / b_j
- j 在前：奖赏_j = product / b_j, 奖赏_i = product × a_j / b_i

要比较这两个方案中较大值。小推导后得到比较条件：`a_i × b_i < a_j × b_j`——按这个排序！

**第 4 步：抽象成算法**

这不只是本题的结论——这是一个通用技巧：**邻项交换法**。

当你需要对一组元素排序使某种代价最优，但不知道排序的依据时：
1. 只看相邻两个元素
2. 写出它们在两种顺序下的代价
3. 推导出谁应该在前面的条件
4. 这个条件就是排序的 cmp 函数

### C++ 代码

```cpp
#include <bits/stdc++.h>
using namespace std;

struct Minister {
    int a, b; // 左手、右手上的数
};

int main() {
    int n;
    cin >> n;
    vector<Minister> v(n);
    int kingA; // 国王的左手（题目中会给出，乘积从国王开始算）
    cin >> kingA; // 国王的右手用不到
    for (int i = 0; i < n; i++) {
        cin >> v[i].a >> v[i].b;
    }

    // 邻项交换法的关键：按 a * b 从小到大排序
    // 为什么？因为可以证明，如果 a_i*b_i > a_j*b_j，交换 i 和 j 不会让答案变差
    // 大白话：a*b 小的排前面，能让两个人的奖赏都尽量小
    sort(v.begin(), v.end(), [](const Minister &x, const Minister &y) {
        return x.a * x.b < y.a * y.b;
    });

    // 需要高精度（乘积太大），这里是示意图，用 Python 大整数或自己实现
    // C++ 标准写法需要手写高精度乘法+除法

    long long product = kingA;  // 累积乘积（从国王的左手开始）
    long long maxReward = 0;    // 最大奖赏

    for (const auto &m : v) {
        // 当前大臣的奖赏 = 前面所有人左手乘积 / 自己的右手
        long long reward = product / m.b;
        maxReward = max(maxReward, reward);
        product *= m.a; // 更新乘积给下一位大臣用
    }
    cout << maxReward << "\n";
    return 0;
}
// 注意：本题 n 较大时乘积会极大超出 long long，需要高精度。
// 上述代码只展示贪心思路，实际提交需用 Python 或手写高精度。
```

## 三、经典模型：哈夫曼编码（合并果子）

### 问题描述

有 n 堆果子，每堆重量 `a[i]`。每次可以合并任意两堆，消耗的体力等于两堆重量之和。问最小总体力消耗。

### 思考流程

**第 1 步：直觉**

"每次合并体力等于两堆之和……那肯定每次挑最小的两堆合并啊！"

为什么？因为越早合并的堆，被加的次数越多。如果你让大堆早早参与合并，它的重量会被反复累加，体力就大了。反过来，让小堆先合并，大堆最后出场，总体力最小。

**第 2 步：数据结构的选取**

"每次取最小和次小，合并完又放回去，再来一次……这不就是个优先队列（最小堆）吗？"

**第 3 步：怎么证明贪心是对的？**

还是**交换论证**：假设最优合并方案中，第一次合并的不是最小的两堆。你把那一次换成最小的两堆，总结果是变小或不变。所以贪心策略一定在某个最优解里。

### C++ 代码

```cpp
#include <bits/stdc++.h>
using namespace std;

int main() {
    int n;
    cin >> n;

    // 最小堆（小顶堆）：每次 O(log n) 取出最小值
    // 为什么用堆？因为每次合并后要放回去重新取最小，堆能快速维护动态最小值
    priority_queue<int, vector<int>, greater<int>> pq;

    for (int i = 0; i < n; i++) {
        int x;
        cin >> x;
        pq.push(x);
    }

    long long ans = 0; // 总体力消耗

    // 直到只剩一堆：合并 n-1 次
    while (pq.size() > 1) {
        // 取出最小的两堆——贪心选择：每次合并重量最小的
        int a = pq.top(); pq.pop();
        int b = pq.top(); pq.pop();

        int cost = a + b;   // 合并这两堆消耗的体力
        ans += cost;        // 累加到总答案

        pq.push(cost);      // 合并后的新堆放回去
    }

    cout << ans << "\n";
    return 0;
}
```

### 为什么不能用排序然后依次合并？

不对。排序后从前往后加，比如 `[1, 2, 3, 4]`：
- 排序法：1+2=3 → 3+3=6 → 6+4=10，总和 = 3+6+10 = 19 ❌
- 堆法：1+2=3 → 3+3=6 → 4+6=10，总和也是 19……咦这次碰巧一样

但换个数据 `[1, 1, 1, 1]`：
- 排序法：1+1=2 → 2+1=3 → 3+1=4，总和 = 2+3+4 = 9
- 堆法：1+1=2 → 1+1=2 → 2+2=4，总和 = 2+2+4 = 8 ✅

为什么堆法更优？因为排序法**固定了谁先被合并**（只能从左往右），而堆法让**最小的两个随时合并**，不管它们原来在什么位置。堆的灵活性让贪心更强。

## 四、贪心+排序的常见套路

大量贪心题都是：**先排序，再扫描一遍**。

| 排序依据 | 经典问题 | 思路 |
|----------|----------|------|
| 按结束时间排序 | 活动选择 / 区间调度 | 每次选结束最早的 |
| 按右端点排序 | 区间选点 / 区间覆盖 | 点选在右端点最优 |
| 按 a+b 排序 | 国王游戏 / 刷题顺序 | 邻项交换法推导 |
| 按价值/重量比排序 | 分数背包（不是 0/1 背包！） | 性价比高的优先 |
| 按左端点排序 | 合并区间 | 能合并就合并，贪左边界 |

## 五、贪心「翻车」预警

| 问题类型 | 容易误用贪心的地方 | 正确方法 |
|----------|-------------------|----------|
| **0/1 背包** | 按性价比排序 → ❌ 因为物品不可分割 | 动态规划 |
| **找零钱（非倍数面额）** | 每次选最大 → ❌ 可能错过最优组合 | DP 或搜索 |
| **最短路带负权** | Dijkstra → ❌ 贪心选最小距离的结点不成立了 | Bellman-Ford / SPFA |
| **哈密顿路径** | 每次选最近的点 → ❌ 局部最优≠全局最优 | 状压 DP / 搜索 |

**金句**：贪心是用在**你确信「这一步不会让你后悔」**的时候。如果拿不准，先想一下能不能举出反例。能举出反例就别贪，换 DP 或搜索。
