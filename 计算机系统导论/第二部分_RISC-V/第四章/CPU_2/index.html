    一、流水线图类型¶

    （一）多周期流水线图¶

    ![[幻灯片59.png]]

    展示内容：多条指令在多个时钟周期内的执行过程

    示例：5条指令在9个时钟周期的执行

    指令 IF ID EX MEM WB
    ld x10, 40(x1) CC1 CC2 CC3 CC4 CC5
    sub x11, x2, x3 CC2 CC3 CC4 CC5 CC6

    ld x13, 48(x1) CC4 CC5 CC6 CC7 CC8

    关键观察： - 每个时钟周期都有一条新指令进入流水线 - 每个时钟周期都有一条指令完成（理想情况） - 吞吐量 = 1条指令/时钟周期

    （二）单周期流水线图¶

    ![[幻灯片60.png]]

    展示内容：某个特定时钟周期内，流水线各阶段正在执行的指令

    图中的关键信息： - 顶部横轴标注五个阶段：Instruction fetch → Instruction decode → Execution → Memory → Write-back - 每个阶段上方标注当前正在执行的指令 - 浅蓝色高亮的长条形模块就是流水线寄存器（IF/ID、ID/EX、EX/MEM、MEM/WB）

    1. 为什么需要流水线寄存器？¶

    问题场景：假设没有流水线寄存器

    1
2
3
    时钟周期1：指令A在IF阶段，产生指令内容
时钟周期2：指令A进入ID阶段，指令B进入IF阶段
问题：指令A的指令内容去哪了？被指令B覆盖了！

    流水线寄存器的作用：

    作用 说明
    保存前一阶段的输出 防止被后续指令覆盖
    提供后一阶段的输入 在正确的时钟周期提供正确的数据
    实现流水线并行 让多条指令同时处于不同阶段

    打个比方：

    想象一个工厂流水线： - 没有中间仓库：工人做完产品直接传给下一个工人，但如果下个工人还在忙，产品就丢了 - 有中间仓库（流水线寄存器）：工人把产品放到仓库，下一个工人有空了再取走

    2. 四个流水线寄存器保存的内容¶

    寄存器 位置 保存什么 为什么需要
    IF/ID IF与ID之间 指令内容、PC+4 ID阶段需要知道这条指令是什么，才能译码
    ID/EX ID与EX之间 操作数、立即数、控制信号、目标寄存器编号、PC+4 EX阶段需要操作数才能运算
    EX/MEM EX与MEM之间 ALU结果、写数据、控制信号 MEM阶段需要地址才能访问内存
    MEM/WB MEM与WB之间 内存读取数据/ALU结果、目标寄存器编号 WB阶段需要知道写入哪个寄存器、写入什么值

    二、流水线控制¶

    （一）简化的流水线控制架构¶

    ![[幻灯片61.png]]

    核心原理： - 控制信号由指令的opcode字段产生 - 与单周期实现类似，但控制信号需要随流水线传递

    这张图展示了什么？

    从图可以看到： 1. 控制单元（Control）：接收指令的opcode，输出各种控制信号 2. 蓝色线条：控制信号的传输路径 3. 控制信号的作用位置：寄存器堆、ALU、数据内存、多路选择器

    控制信号的具体作用：

    控制信号 控制哪个部件 作用
    RegWrite 寄存器堆 是否允许写寄存器
    MemRead/MemWrite 数据内存 读/写内存
    ALUSrc ALU的Mux ALU第二个操作数来源
    ALUOp ALU控制单元 ALU执行什么运算
    MemtoReg WB阶段的Mux 写回数据来自内存还是ALU
    Branch PC的Mux 是否分支跳转

    （二）控制信号的传递¶

    ![[幻灯片62.png]]

    关键问题：控制信号在ID阶段产生，但什么时候使用？

    控制信号 产生阶段 使用阶段 需要传递吗？
    ALUSrc ID EX ✅ 需要传到ID/EX
    ALUOp ID EX ✅ 需要传到ID/EX
    MemRead/MemWrite ID MEM ✅ 需要传到ID/EX，再传到EX/MEM
    MemtoReg ID WB ✅ 需要一路传到MEM/WB
    RegWrite ID WB ✅ 需要一路传到MEM/WB

    传递规则：

    1
2
3
4
5
6
7
    控制信号的生命周期：
ID阶段产生 → ID/EX保存 → EX/MEM保存 → MEM/WB保存

每个流水线寄存器只保存"后续阶段还需要"的信号：
- ID/EX：保存 EX、MEM、WB 三类信号
- EX/MEM：保存 MEM、WB 两类信号（EX阶段已用完的不用传）
- MEM/WB：只保存 WB 信号

    为什么这样设计？

    原因 说明
    避免信号丢失 下一个时钟周期，新的指令进入ID阶段，控制信号会被覆盖
    匹配指令进度 控制信号要"跟着指令走"，每到一个阶段就用对应的信号
    减少硬件开销 不需要每个阶段都重新解码，只解码一次然后传递

    三、数据冒险¶

    （一）什么是数据冒险¶

    ![[幻灯片64.png]]

    问题场景：

    1
2
3
4

    sub  x2, x1, x3    # x2 被写入

    问题本质：后续指令需要使用前序指令的结果，但结果还未写入寄存器。

    时序分析：

    1
2
3

    时钟周期：    CC1     CC2     CC3     CC4     CC5
sub x2...   [IF] → [ID] → [EX] →[MEM] → [WB]
                         ↑ x2在这里计算出来
and x12...         [IF] → [ID] → [EX] → ...
                          ↑ 这里就需要x2，但x2还没写入！

    图中的关键信息： - 左侧 x2 以浅灰色高亮显示，表示这是"问题寄存器" - 5条指令都涉及x2，形成一条"数据依赖链" - 右下角写着"We can resolve hazards with forwarding"——这就是解决方案

    （二）解决方案：前向转发（Forwarding）¶

    ![[幻灯片65.png]]

    核心思想：不需要等待结果写回寄存器，直接从ALU输出端"转发"给需要它的指令。

    这张图展示了什么？

    从图中可以看到两条数据通路： - 红色线：原本的数据流——从寄存器堆读取（但此时数据还没写入） - 蓝色线：转发路径——从ALU输出或MEM阶段直接传递

    时间轴上的观察：

    图中上半部分是时间轴（CC1-CC9），下半部分是流水线执行图： - 第一行显示 x2 的值在不同时钟周期的变化：CC1-CC4 是旧值(10)，CC5 开始是新值(-20) - 蓝色转发线从 sub 的 EX/MEM 阶段连接到 and 的 EX 阶段 - 这意味着：sub 在EX阶段算出的结果，不经过寄存器堆，直接送给 and 的ALU

    转发的本质：

    传统方式 转发方式
    结果写回寄存器 → 下条指令从寄存器读 结果直接送到下条指令的ALU输入
    需要等待WB阶段完成 EX阶段结束后立即可用
    可能产生停顿 无停顿

    （三）如何检测需要转发¶

    ![[幻灯片66.png]]

    核心问题：如何知道"当前指令需要的寄存器正好是前序指令要写的寄存器"？

    检测方法：比较寄存器编号

    必要前提：寄存器编号需要随流水线传递

    MEM/WB.RegisterRd MEM/WB寄存器 刚完成MEM阶段指令的目标寄存器编号

    （四）转发的额外条件¶

    ![[幻灯片67.png]]

    问题：不是所有"寄存器编号匹配"都需要转发！

    条件1：该指令确实要写寄存器

    图中的代码：EX/MEM.RegWrite, MEM/WB.RegWrite

    为什么需要？ - 如果前序指令是 Store 指令，它不写寄存器（RegWrite=0） - 此时 Rd 字段的值是"无意义"的，不应该用它来转发

    条件2：目标寄存器不是x0

    图中的代码：EX/MEM.RegisterRd ≠ 0, MEM/WB.RegisterRd ≠ 0

    为什么需要？ - x0 是硬连线到0的寄存器，写入x0不会改变任何值 - 很多指令用 x0 作为"丢弃结果"的方式，比如 add x0, x1, x2 只是为了设置条件码 - 此时不需要转发

    七、Load-Use 冒险¶

    （一）什么是 Load-Use 冒险¶

    ![[幻灯片73.png]]

    为什么 Load-Use 冒险不能用转发解决？

    普通的数据冒险（如 sub 后跟 and）可以用转发解决，因为： - sub 在 EX 阶段就计算出了结果 - 可以在同一个周期内转发给下一条指令的 EX 阶段

    但 Load 指令不一样：

    1
2
3
4
5
6
    时钟周期：   CC1     CC2     CC3     CC4     CC5
ld x2,20(x1)  [IF] → [ID] → [EX] →[MEM] → [WB]
                              ↑ 计算地址
                                    ↑ 数据从这里才能读取！
and x4,x2,x5         [IF] → [ID] → [EX] → ...
                                   ↑ 这里就需要x2，但数据还在内存中！

    问题本质： - Load 指令要到 MEM 阶段 才能从内存读取数据 - 下一条指令的 EX 阶段 就需要这个数据 - 即使转发，也需要"未来"的数据（不可能实现）

    检测条件（在 ID 阶段检测）：

    图中给出检测条件：

    1

且 (ID/EX.RegisterRd = IF/ID.RegisterRs1             ← 当前指令的源寄存器1 = Load的目标寄存器
    或 ID/ID.RegisterRd = IF/ID.RegisterRs2)         ← 或源寄存器2 = Load的目标寄存器

    图中关键信息： - 检测在 ID 阶段 进行（"Check when using instruction is decoded in ID stage"） - 需要比较 Load 指令的目标寄存器和当前指令的源寄存器 - 如果检测到冒险，"stall and insert bubble"——暂停并插入气泡

    ![[幻灯片74.png]]

    暂停流水线需要做什么？

    图中列出四个关键操作：

    操作 效果 为什么这样做
    Force control values in ID/EX register to 0 EX、MEM、WB 阶段执行 nop（空操作） 让当前指令"暂停"在ID阶段，不继续执行

    （四）动态分支预测¶

    ![[幻灯片82.png]]

    为什么需要动态分支预测？

    图中第一行说明："In deeper and superscalar pipelines, branch penalty is more significant"

    处理器类型 流水线深度 分支惩罚
    简单流水线 5 级 冲刷 1-3 条指令
    深度流水线 10-20 级 冲刷 10+ 条指令
    超标量处理器 多发射 每周期完成多条指令，惩罚更大

    动态预测的核心思想：

    "Use dynamic prediction" —— 根据历史行为预测分支走向

    Branch History Table（分支历史表）：

    组件 功能
    存储结构 一个小型缓存/表
    索引方式 用分支指令地址作为索引
    存储内容 taken（跳转）或 not taken（不跳转）

    工作流程：

    图中详细列出了三个步骤：

    步骤 操作 说明
    1. Check table 查询该分支的历史预测 分支指令地址作为索引查表
    2. Start fetching 根据预测从目标或顺序地址取指 预测 taken → 取目标地址；预测 not taken → 顺序取指
    3. If wrong 冲刷流水线，翻转预测 预测错误时，更新历史表

    预测准确率：

    现代 CPU 的分支预测准确率可达 95% 以上，意味着： - 95% 的分支判断正确 → 无惩罚 - 5% 预测错误 → 需要冲刷流水线

    与静态预测对比：

    预测方式 策略 准确率
    静态预测 总是预测跳转/不跳转 50-70%
    动态预测 根据历史预测 95%+

    十二、动态分支预测详解¶

    （一）为什么需要动态预测？¶

    核心问题：在更深的流水线和超标量处理器中，分支惩罚更严重。

    处理器类型 流水线深度 分支惩罚
    简单流水线 5级 冲刷1-3条指令
    深度流水线 10-20级 冲刷10+条指令
    超标量处理器 多发射 每周期多条指令，惩罚翻倍

    动态预测的核心思想：根据分支的历史行为来预测未来行为。

    分支历史表（Branch History Table）：

    特性 说明
    存储结构 小型缓存/表
    索引方式 用分支指令地址作为索引
    存储内容 taken（跳转）或 not taken（不跳转）

    工作流程：

    步骤 操作
    1. Check table 用分支指令地址查表，获取历史预测
    2. Start fetching 根据预测从目标地址或顺序地址取指
    3. If wrong 冲刷流水线，翻转预测值

    打个比方：

    想象你在猜一个人明天会不会带伞： - 静态预测：不管三七二十一，总是猜"会带"或"不会带" - 动态预测：看他过去一周带伞的记录，发现他晴天不带、雨天带，然后预测

    十三、异常和中断¶

    （一）什么是异常和中断¶

    定义：需要改变控制流的"意外"事件。

    异常 vs 中断：

    类型 来源 例子
    异常（Exception） CPU内部产生 未定义操作码、系统调用、除零错误
    中断（Interrupt） 外部I/O控制器产生 键盘输入、定时器中断、网络数据到达

    为什么要处理？

    • 用户按了Ctrl+C → 需要终止程序
    • 程序访问非法地址 → 需要报错
    • 定时器到期 → 需要切换进程

    挑战：在不牺牲性能的情况下处理它们。

    （二）如何处理异常¶

    三个关键步骤：

    步骤 操作 RISC-V中的实现
    1. 保存PC 保存引发异常（或被中断）的指令地址 SEPC（Supervisor Exception Program Counter）
    2. 保存原因 记录是什么问题 SCAUSE（Supervisor Exception Cause Register）
    3. 跳转到处理程序 转移控制权 默认地址：0x1C090000

    SCAUSE寄存器结构：

    1

    ┌────────────────────────────────────────────────────────────┐
│ 63        ...         8  │  7  │  6  │  5  │  4  │ 3-0    │
│       未使用             │  Interrupt  │  Exception Code  │
└────────────────────────────────────────────────────────────┘

    常见的异常代码：

    异常代码 含义
    2 未定义操作码
    12 硬件故障
    8 环境调用（系统调用）

    （五）流水线中的异常¶

    ![[幻灯片91.png]]

    核心观点：异常是另一种形式的控制冒险。

    示例场景：

    1

    处理步骤：

    步骤 操作

    可复用硬件 - ✓

    硬件复用：

  MEM: add x1  ← 异常在这里检测到！
  WB: or x13   ← 前序指令，正常完成

CC7 时（冲刷后）：
  IF: sd x26   ← 异常处理程序第一条指令
  ID: bubble   ← 原来的 sub x15 被冲刷
  EX: bubble   ← 原来的 ld x16 被冲刷
  MEM: bubble  ← 原来的空被填充
  WB: or x13   ← 前序指令已完成

    与分支预测错误的对比：

    特性 分支预测错误 异常处理
    触发条件 分支方向预测错误 硬件故障/非法操作
    冲刷范围 预测错误路径的指令 异常指令及后续所有指令
    PC更新 分支目标地址 异常处理程序地址
    额外操作 无 设置 SEPC、SCAUSE
    可恢复性 不可恢复（预测错误） 可能可恢复（如缺页异常）

    一句话总结：异常发生时，硬件自动冲刷流水线中异常指令之后的所有指令，插入 bubble，并跳转到异常处理程序执行

    （四）双发射中的冒险¶

    核心问题：一个周期两条指令 → 同一时刻竞争资源 + 依赖关系翻倍

    1. 包内冒险¶

    问题：同一个发射包里的两条指令有依赖

    1
2
3
4
5
    发射包：
┌─────────────────┬─────────────────┐
│  add x1, x2, x3 │  sub x4, x1, x5 │  ← x1 有依赖！
└─────────────────┴─────────────────┘
        槽1                槽2

    解决：编译器保证包内无依赖，或者拆成两个包

    2. 包间冒险¶

    问题：当前包依赖上一个包的结果

    1
2
    周期1：add x1, x2, x3  │  nop
周期2：sub x4, x1, x5  │  ld x6, 0(x4)  ← x1、x4 都依赖上周期

    解决：转发机制仍然有效，但需要更复杂的检测逻辑

    3. Load-Use 冒险更严重¶

    问题：Load 指令要到 MEM 阶段才能读到数据

    1
2
3
4
5
6
7
    周期1：nop           │  ld x1, 0(x10)
周期2：add x2, x1, x3 │  nop           ← x1 还没读到！

必须拆成：
周期1：nop           │  ld x1, 0(x10)
周期2：nop           │  nop           ← 强制等一周期
周期3：add x2, x1, x3 │  ...

    代价：双发射下 Load-Use 冒险浪费的是2个发射槽（相当于浪费2条指令的吞吐量）

    4. 双发射冒险总结¶

    冒险类型 原因 解决方案 代价
    包内数据冒险 同包指令有依赖 编译器调度避免 无（编译期解决）
    包间数据冒险 跨包依赖 转发机制 无（硬件解决）
    Load-Use冒险 数据还没读到 插入气泡 浪费2个槽
    结构冒险 两个ALU冲突 复制硬件资源 硬件成本

    5. 异常处理的精确性¶

    问题：乱序执行时，如何保证异常的正确处理？

    1
2
3
    ld x1, 0(x10)      # 可能发生异常（非法地址）
add x2, x1, x3     # 依赖x1
div x4, x5, x0     # 除零异常！

    时序： - div 先执行（没有依赖） - ld 后执行（需要访存） - 但 ld 在程序顺序上更早

    不精确异常的问题： - 如果 div 先触发异常，处理程序会认为 div 是第一条出错的指令 - 但实际上 ld 可能也会出错，且 ld 更早

    ROB 保证精确异常：

    1
2
3
4
5
6
7
8
9
    执行顺序：div → ld → add
异常检测：div 先检测到除零错误

ROB 中的处理：
1. div 的异常标记写入ROB
2. 不立即触发异常
3. 等待 div 成为 ROB 头部
4. 提交时检查到异常，触发异常处理
5. 此时所有之前的指令都已提交

    关键：异常只在提交阶段触发，保证精确性。

    6. 预约站 vs ROB 的分工¶
    组件 职责 数据流向
    预约站 缓冲指令，等待操作数 寄存器/ROB → 预约站
    ROB 缓存结果，保证顺序提交 功能单元 → ROB → 寄存器

    类比： - 预约站 = 餐厅的"等位区"，顾客（指令）等待食材（操作数） - ROB = 餐厅的"取餐区"，菜品（结果）按顺序给顾客

    ROB 64-256条目 ~10% ~15%
    寄存器重命名表 64-128条目 ~3% ~5%

    复杂度：乱序执行的硬件开销是顺序执行的 3-5倍。

    场景 顺序执行CPI 乱序执行CPI 提升
    无依赖 1.0 1.0 无

    1. 为什么 Cache Miss 是"性能杀手"？¶

    时间尺度对比：

    操作 延迟（周期数） 类比

    L1 Cache命中 3-4 周期 3-4秒

    5. 内存墙（Memory Wall）问题¶

    定义：CPU速度增长 >> 内存速度增长 → CPU大部分时间在等内存。

    历史趋势：

    年份 CPU频率 内存延迟（周期）
    1980 5 MHz ~10 周期
    1990 50 MHz ~20 周期
    2000 1 GHz ~100 周期
    2020 4 GHz ~200-300 周期

    核心矛盾： - CPU每年性能提升 ~50% - 内存延迟每年只改善 ~7% - 差距越来越大

    对流水线的影响：

    1
2
3
    流水线深度增加 → 分支惩罚增加
发射宽度增加   → Cache Miss 代价翻倍
CPU速度提升   → 内存延迟（周期数）增加

    技术1：非阻塞Cache（Non-blocking Cache）

    技术2：预取（Prefetching）

    预约站大小 缓冲更多等待的指令 面积、功耗增加
    重排序缓冲大小 支持更多in-flight指令 复杂度增加

    技术4：多线程（SMT/Hyper-Threading）

    内存停顿的计算：

    \[CPI_{内存} = \text{Miss次数/指令} \times \text{Miss惩罚}\]

    常见面试题¶

    Q1: 为什么Load-Use冒险不能只用转发解决？

    A: 因为Load指令要到MEM阶段才能从内存读到数据，而需要使用该数据的指令在EX阶段就需要它。即使转发，也需要"未来"的数据，物理上不可能。

    Q2: 2位预测器为什么比1位预测器好？

    A: 对于嵌套循环，1位预测器每次循环结束和下次循环开始都会预测错误。2位预测器需要连续两次错误才改变预测，避免了这个问题。

    Q3: 异常处理和分支预测错误有什么共同点？

    A: 都需要： 1. 冲刷流水线 2. 保存当前状态（PC、原因） 3. 跳转到新地址 4. 可以复用大部分硬件

    Q4: 为什么说流水线提高了吞吐量但不减少延迟？

    A: - 吞吐量：每周期完成的指令数增加（理想情况1条/周期） - 延迟：单条指令从开始到完成的时间不变（5个周期） - 流水线是"并行处理多条指令"，不是"加速单条指令"

    Q5: 什么是精确异常？为什么重要？

    A: - 精确异常：异常发生时，之前的指令都已完成，之后的指令都未执行 - 重要性：简化异常处理程序的逻辑，保证程序语义正确 - 实现难度：乱序执行、多发射等优化使得精确异常难以维护
