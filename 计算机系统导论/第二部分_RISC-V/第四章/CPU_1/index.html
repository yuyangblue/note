    CPU 核心知识整理¶

    一、CPU 性能三要素（经典公式）¶

    CPU 性能由执行时间决定，核心公式为：

    CPU执行时间=指令数×CPI×时钟周期

    • 指令数（Instruction count）：程序运行的总指令条数，由指令集架构（ISA和编译器优化共同决定（比如 RISC-V 精简指令集通常比 x86 指令数更少，编译器优化可减少冗余指令）。
    • CPI（每条指令平均时钟周期数）：平均每条指令需要的时钟周期，由 CPU 硬件设计决定（单周期 CPU CPI=1，流水线 CPU CPI≈1，复杂超标量 CPU CPI<1）。
    • 时钟周期（Cycle time）：CPU 主频的倒数，由硬件工艺、电路设计决定（时钟周期越短，主频越高）。

    二、课程内容规划¶

    本次课程将围绕两种 RISC-V 处理器实现展开：

    • 简化版单周期处理器：用于理解基础数据通路、控制逻辑，适合入门学习。
    • 真实流水线处理器：讲解 5 级流水线、冒险处理、性能优化，贴近实际芯片设计。

    三、核心指令子集（教学用精简集）¶

    为了简化教学，选取了覆盖三类核心操作的指令子集，覆盖处理器设计的绝大多数关键场景：

    指令类型 代表指令 功能说明
    内存访问 ld（load doubleword）、sd（store doubleword） 完成内存与寄存器之间的数据读写，是访存阶段的核心指令
    算术逻辑 add、sub、and、or 完成加减、与或等基础运算，对应执行阶段的 ALU 操作
    控制转移 beq（branch if equal） 条件分支指令，用于实现循环、判断，是流水线控制冒险的核心来源

    四、指令执行流程¶

    ![[Pasted image 20260408175633.png]]

    （一）三个核心阶段¶

    1. 取指阶段¶

    • PC → instruction memory, fetch instruction
      • 程序计数器（PC）存储下一条要执行指令的内存地址
      • 从指令内存中读取该地址对应的指令内容
      • 这是冯·诺依曼架构的核心"取指-执行"循环的第一步

    2. 译码阶段¶

    • Register numbers → register file, read registers
      • 解析指令中的寄存器编号字段
      • 从寄存器堆（register file）中读取对应寄存器的数据
      • 为后续运算准备操作数

    1. 算术逻辑运算指令¶

    • 核心部件：算术逻辑单元（ALU）
    • 功能：
      • 执行加减乘除、逻辑与或非等运算

    2. 访存指令¶

    • Access data memory for load/store
      • Load指令：从数据内存中读取数据到寄存器
      • Store指令：将寄存器中的数据写入数据内存
      • 这是CPU与内存之间的数据交互通道

    3. 控制转移指令¶

    • PC ← target address or PC + 4
      • 无条件跳转：直接将目标地址写入PC
      • 条件分支：根据ALU的比较结果决定是否跳转
      • 顺序执行：默认情况下PC自动加4（指向4字节对齐的下一条指令）

    五、计算机CPU的工作原理与指令执行机制¶

    ![[Pasted image 20260408175747.png]]

    （一）整体知识体系框架¶

    1. CPU整体架构概述¶

    • 核心定位：这是整个CPU工作原理的总览图
    • 关键组件：
      • PC（程序计数器） ：指挥棒，控制指令执行顺序
      • Instruction Memory（指令内存） ：存储程序代码
      • Registers（寄存器堆） ：高速数据缓存
      • ALU（算术逻辑单元） ：运算核心
      • Data Memory（数据内存） ：存储数据

    2. 指令执行流程¶

    • 核心定位：拆解CPU的"取指-译码-执行"核心流程
    • 三个阶段：
      1. 取指：PC指向指令内存，读取指令
      2. 译码：解析指令中的寄存器编号，读取操作数
      3. 执行：根据指令类型分支处理（ALU运算/访存/控制转移）

    （二）两大关键技术模块¶

    ![[Pasted image 20260408175851.png]]

    1. 多路复用器的作用¶

    • 核心问题：解决多个数据源竞争同一数据通路的冲突

    2. 控制单元的功能¶

    • 核心定位：整个系统的控制大脑
    • 关键组件：
      • Control Module（控制单元） ：通过蓝色连线向各模块输出控制信号
      • Mux（多路复用器） ：根据控制信号选择数据通路
      • 与门电路：实现分支条件判断

    六、数字逻辑基础¶

    （一）数字逻辑的三大基石¶

    1. 二进制信息编码：数字世界的通用语言¶

    • 物理本质：用电路的高低电平表示二进制的0和1
    • 工程实现：
      • 单根导线只能传输1比特信息（0或1）
      • 多根导线组成的总线可以同时传输多个比特（如32位总线一次传输4字节数据）
    • 哲学意义：这是"一切计算皆可简化为二进制操作"的物理基础

    2. 组合逻辑元件：无记忆的运算单元¶

    • 核心特性：输出仅取决于当前输入，没有记忆功能
    • 典型元件：
      • 逻辑门（与门、或门、非门等）
      • 算术逻辑单元（ALU）
      • 多路复用器（Mux）
    • 数学模型：可以用布尔代数完全描述其行为

    3. 时序逻辑元件：有记忆的存储单元¶

    • 核心特性：输出不仅取决于当前输入，还取决于历史状态（有记忆功能）
    • 典型元件：
      • 触发器（Flip-flop）
      • 寄存器（Register）
      • 内存（Memory）
    • 时钟同步：需要时钟信号来控制状态更新的时机
    • 工程意义：这是实现复杂计算任务的基础，因为需要存储中间结果和程序状态

    （二）组合逻辑元件¶

    ![[Pasted image 20260408180034.png]]

    1. 与门（AND-gate）¶

    • 逻辑表达式：Y = A & B
    • 电路符号：带圆角的矩形，左侧两个输入端口A、B，右侧一个输出端口Y
    • 功能：只有当所有输入为1时，输出才为1
    • 应用场景：用于条件判断、信号同步等

    2. 加法器（Adder）¶

    • 逻辑表达式：Y = A + B
    • 电路符号：带切角的多边形，内部印有"+"标识，左侧上下两个输入端口A、B，右侧一个输出端口Y
    • 功能：执行二进制加法运算
    • 扩展：可以级联构成多位加法器，如32位、64位加法器

    3. 多路选择器（Multiplexer）¶

    • 逻辑表达式：Y = S ? I1 : I0
    • 电路符号：竖长的圆角矩形，内部印有"Mux"字样，左侧上下两个输入端口I0、I1，底部一个选择控制端口S，右侧一个输出端口Y
    • 功能：根据选择信号S的值，选择其中一个输入输出到Y
    • 应用场景：用于数据通路选择、总线仲裁等

    4. 算术逻辑单元（Arithmetic/Logic Unit, ALU）¶

    • 逻辑表达式：Y = F(A, B)
    • 电路符号：带切角的多边形，内部印有"ALU"字样，左侧上下两个输入端口A、B，底部一个功能控制端口F，右侧一个输出端口Y
    • 功能：根据控制信号F的不同，执行算术运算（加、减、乘、除等）或逻辑运算（与、或、非、移位等）
    • 地位：这是CPU的核心运算部件，所有的数据处理最终都由ALU完成

    （三）时序逻辑元件¶

    1. 寄存器的核心概念¶

    • 核心功能：在数字电路中存储数据
    • 与组合逻辑的区别：寄存器是时序逻辑元件，具有记忆功能，可以保存数据状态
    • 应用场景：
      • CPU中的通用寄存器、程序计数器（PC）、指令寄存器（IR）等
      • 内存中的存储单元
      • 各种接口电路中的数据缓冲器

    2. 时钟同步机制¶

    ![[Pasted image 20260408180230.png]] - 核心特性：寄存器的状态更新需要时钟信号的控制 - 作用：确保整个数字系统中的各个部件同步工作，避免数据竞争和时序混乱 - 时钟信号：通常是一个周期性的方波信号，由专门的时钟发生器产生

    3. 边沿触发方式¶

    ![[Pasted image 20260408180244.png]] - 触发机制：寄存器只在时钟信号的边沿（上升沿或下降沿）时更新其存储的值 - 上升沿触发：当时钟信号从0变为1（低电平到高电平）时更新 - 下降沿触发：当时钟信号从1变为0（高电平到低电平）时更新 - 优势：相比电平触发，边沿触发具有更好的抗干扰能力和更高的工作频率

    4. 带写入控制的寄存器¶

    ![[Pasted image 20260408180258.png]] - 核心改进：在基础寄存器的基础上增加了写入控制功能 - 与基础寄存器的区别：基础寄存器在每个时钟边沿都会更新，而带写入控制的寄存器只有当写入控制输入（Write）为1，并且在时钟边沿（通常是上升沿）时，才会将输入数据（D）写入寄存器 - 应用场景： - CPU中的通用寄存器组 - 各种接口电路中的数据寄存器 - 需要选择性更新的存储单元

    七、CPU设计核心内容¶

    （一）数据通路的设计方法¶

    1. 数据通路的定义¶

    • 核心功能：在CPU中传输和处理数据的硬件路径
    • 组成元件：
      • 寄存器：存储临时数据
      • ALU：执行算术逻辑运算
      • 多路复用器（Mux） ：选择数据通路
      • 内存：存储指令和数据
      • 总线：连接各个元件的通信线路
    • 与控制单元的关系：数据通路负责数据的传输和处理，控制单元负责指挥数据通路的操作

    2. 数据通路的设计方法¶

    • 增量式设计：从简单到复杂，逐步构建完整的数据通路
    • 模块化设计：将数据通路划分为多个功能模块，如取指模块、译码模块、执行模块、访存模块、写回模块等
    • RISC-V架构：采用精简指令集计算机（RISC）架构，具有指令简单、执行高效的特点

    （二）RISC-V数据通路的设计流程¶

    1. 总体设计阶段¶

    • 设计目标：确定数据通路的总体架构和功能模块
    • 关键决策：
      • 选择指令集架构（如RISC-V）
      • 确定寄存器堆的大小和结构
      • 选择ALU的功能和性能
      • 确定内存的类型和访问方式
    • 设计输出：数据通路的总体框图和功能描述

    2. 模块设计阶段¶

    • 设计目标：详细设计每个功能模块的电路结构和时序特性
    • 关键模块：
      • 取指模块：从内存中读取指令
      • 译码模块：解析指令的操作码和操作数
      • 执行模块：执行指令的运算操作
      • 访存模块：访问数据内存
      • 写回模块：将运算结果写回寄存器堆
    • 设计输出：每个模块的详细电路图和时序分析报告

    3. 集成验证阶段¶

    • 设计目标：将各个功能模块集成在一起，验证整个数据通路的功能和性能
    • 关键步骤：
      • 连接各个模块的信号和数据通路
      • 编写测试程序，验证数据通路的功能正确性
      • 进行时序分析，验证数据通路的性能指标
      • 优化数据通路的设计，提高性能和可靠性
    • 设计输出：完整的数据通路设计文档和验证报告

    （三）指令取指¶

    ![[Pasted image 20260408180842.png]]

    1. 程序计数器（PC）¶

    • 核心功能：存储下一条要执行指令的内存地址
    • 寄存器类型：通常是一个64位的寄存器（在64位架构中）
    • 更新机制：
      • 顺序执行：PC自动加4（指向4字节对齐的下一条指令）
      • 分支跳转：根据分支指令的结果，将目标地址写入PC
    • 地位：程序计数器是CPU的核心寄存器之一，它决定了程序的执行顺序

    2. 指令内存（Instruction Memory）¶

    • 核心功能：存储程序的指令代码
    • 访问方式：只读访问（CPU只能从指令内存中读取指令，不能写入）
    • 地址来源：指令内存的地址输入来自程序计数器（PC）
    • 输出结果：输出对应地址的指令内容

    3. 加法器（Add）¶

    • 核心功能：计算下一条指令的地址
    • 输入信号：
      • 程序计数器（PC）的当前值
      • 增量值（通常是4，因为指令是4字节对齐的）
    • 输出结果：下一条指令的地址（PC + 4）
    • 反馈回路：加法器的输出反馈回程序计数器（PC）的输入，实现PC的自动更新

    （四）R型指令¶

    ![[Pasted image 20260408181039.png]]

    1. R型指令的定义¶

    • 指令格式：R型指令是RISC-V架构中的一种指令格式，主要用于算术逻辑运算
    • 指令字段：
      • opcode：操作码，用于识别指令类型
      • rd：目标寄存器，存储运算结果
      • funct3：功能码，用于指定具体的运算类型
      • rs1：源寄存器1，提供第一个操作数
      • rs2：源寄存器2，提供第二个操作数
      • funct7：功能码，用于进一步指定运算类型
    • 指令长度：所有R型指令都是32位固定长度

    2. R型指令的执行流程¶

    • 取指阶段：从指令内存中读取R型指令
    • 译码阶段：解析指令的各个字段，读取源寄存器rs1和rs2的值
    • 执行阶段：在ALU中执行指定的算术逻辑运算
    • 写回阶段：将运算结果写入目标寄存器rd

    （五）Load/Store指令¶

    ![[Pasted image 20260408181155.png]]

    1. Load/Store指令的定义¶

    • 指令格式：Load/Store指令是RISC-V架构中的一种指令格式，主要用于CPU与内存之间的数据交互
    • 指令类型：
      • Load指令：从内存中读取数据到寄存器
      • Store指令：将寄存器中的数据写入内存
    • 指令字段：
      • opcode：操作码，用于识别指令类型
      • rd：目标寄存器（仅Load指令使用）
      • funct3：功能码，用于指定数据类型（如字节、半字、字、双字）
      • rs1：源寄存器，提供基地址
      • imm：12位偏移量，用于计算有效地址
    • 指令长度：所有Load/Store指令都是32位固定长度

    2. Load/Store指令的执行流程¶

    • 取指阶段：从指令内存中读取Load/Store指令
    • 译码阶段：解析指令的各个字段，读取源寄存器rs1的值
    • 执行阶段：使用ALU计算有效地址（基地址 + 符号扩展后的偏移量）
    • 访存阶段：
      • Load指令：从内存中读取有效地址对应的数据
      • Store指令：将源寄存器rs2的值写入内存的有效地址
    • 写回阶段：
      • Load指令：将读取的数据写入目标寄存器rd
      • Store指令：没有写回操作

    （六）Branch指令¶

    ![[Pasted image 20260408181958.png]]

    1. Branch指令的定义¶

    • 指令格式：Branch指令是RISC-V架构中的一种指令格式，主要用于实现程序的分支跳转
    • 指令类型：
      • 条件分支指令：根据比较结果决定是否跳转（如beq、bne、blt、bge等）
      • 无条件分支指令：直接跳转到目标地址（如j、jal等）
    • 指令字段：
      • opcode：操作码，用于识别指令类型
      • funct3：功能码，用于指定比较类型（如相等、不等、小于、大于等于等）
      • rs1：源寄存器1，提供第一个比较操作数
      • rs2：源寄存器2，提供第二个比较操作数
      • imm：12位位移量，用于计算目标地址
    • 指令长度：所有Branch指令都是32位固定长度

    2. Branch指令的执行流程¶

    • 取指阶段：从指令内存中读取Branch指令
    • 译码阶段：解析指令的各个字段，读取源寄存器rs1和rs2的值
    • 执行阶段：
      • 使用ALU比较源寄存器rs1和rs2的值（通常是相减并检查Zero标志位）
      • 计算目标地址（当前PC值 + 符号扩展并左移1位后的位移量）
    • 访存阶段：没有访存操作
    • 写回阶段：没有写回操作
    • 更新PC：
      • 如果条件满足：将目标地址写入PC
      • 如果条件不满足：PC自动加4（指向4字节对齐的下一条指令）

    八、控制信号的选择性激活机制¶

    （一）控制信号的分类与默认值¶

    ![[Pasted image 20260408182159.png]]

    1. 写使能信号¶

    • 典型信号：RegWrite（寄存器写使能）、MemWrite（内存写使能）
    • 默认值：0（禁止写入）
    • 原因：
      • 防止误写入寄存器或内存，确保数据安全
      • 只有当指令明确需要写操作时，才将对应的写使能信号设置为1
    • 示例：
      • R型指令：RegWrite=1，MemWrite=0
      • Load指令：RegWrite=1，MemWrite=0
      • Store指令：RegWrite=0，MemWrite=1
      • 分支指令：RegWrite=0，MemWrite=0

    2. 选择信号¶

    • 典型信号：ALUSrc（ALU源选择）、MemtoReg（内存到寄存器选择）
    • 默认值：0（选择第一个输入）
    • 原因：
      • 大多数指令（如R型指令）使用第一个输入（寄存器值）
      • 只有当指令需要使用第二个输入（立即数或内存读取数据）时，才将对应的选择信号设置为1
    • 示例：
      • R型指令：ALUSrc=0，MemtoReg=0
      • Load指令：ALUSrc=1，MemtoReg=1
      • Store指令：ALUSrc=1，MemtoReg=0（无意义，因为RegWrite=0）

    3. 操作控制信号¶

    • 典型信号：ALU operation（ALU操作控制）
    • 默认值：无固定默认值，需根据指令类型设置
    • 原因：
      • 不同指令需要ALU执行不同的运算操作
      • 控制单元会根据指令的操作码和功能码，设置对应的ALU操作控制信号
    • 示例：
      • add指令：ALU operation=000（加法）
      • sub指令：ALU operation=001（减法）
      • and指令：ALU operation=010（与运算）
      • or指令：ALU operation=011（或运算）

    （二）控制信号的生成机制¶

    1. 控制单元的作用¶

    • 核心功能：根据指令的操作码和功能码，生成对应的控制信号
    • 输入信号：
      • opcode：指令的操作码
      • funct3：指令的功能码（部分指令有）
      • funct7：指令的功能码（部分指令有）
    • 输出信号：
      • 所有的控制信号，如RegWrite、MemWrite、ALUSrc、MemtoReg、ALU operation等
    • 实现方式：
      • 硬连线控制单元：使用组合逻辑电路直接生成控制信号
      • 微程序控制单元：使用微程序存储控制信号的生成逻辑

    2. 控制信号的真值表¶

    指令类型 RegWrite MemWrite ALUSrc MemtoReg ALU operation
    R型 1 0 0 0 由funct3和funct7决定
    Load 1 0 1 1 000（加法）
    Store 0 1 1 0 000（加法）
    分支 0 0 0 0 001（减法）
    跳转 1 0 1 0 000（加法）

    （三）未使用控制信号的处理原则¶

    1. 安全原则¶

    • 对于写使能信号，未使用时必须设置为0，以防止误写入寄存器或内存
    • 对于选择信号，未使用时设置为默认值（通常是0），以确保数据通路的正确操作

    2. 性能原则¶

    • 控制信号的生成应尽可能快，以减少指令执行的延迟
    • 硬连线控制单元的生成速度通常比微程序控制单元快，但灵活性较差

    3. 可扩展性原则¶

    • 控制单元的设计应考虑未来指令集的扩展，以便轻松添加新的指令类型
    • 微程序控制单元的可扩展性通常比硬连线控制单元好，但生成速度较慢

    九、完整数据通路设计¶

    ![[Pasted image 20260515205716.png]]

    （一）PC更新路径¶

    • 核心组件：
      • PC：程序计数器，存储下一条要执行指令的内存地址
      • Add：加法器，计算下一条指令的地址（PC + 4）或分支目标地址
      • Mux：多路复用器，选择下一条指令的地址（当前PC + 4 或 分支目标地址）
    • 数据流向：
      • PC → 加法器1（输入当前地址）
      • 加法器1 → Mux → PC（输入下一条指令的地址，当前PC + 4）
      • PC → 加法器2（输入当前地址）
      • 立即数生成单元 → 左移1位单元 → 加法器2（输入位移量）
      • 加法器2 → Mux → PC（输入分支目标地址，仅分支指令使用）
    • 控制信号：
      • PCSrc：PC源选择控制信号，0表示选择当前PC + 4，1表示选择分支目标地址

    （二）指令取指路径¶

    • 核心组件：
      • PC：程序计数器，存储下一条要执行指令的内存地址
      • Instruction memory：指令内存，存储和读取指令
    • 数据流向：
      • PC → 指令内存（输入地址）
      • 指令内存 → 指令输出（输出指令内容）
    • 控制信号：无专门的控制信号，指令取指是每个指令都必须执行的操作

    （三）寄存器访问路径¶

    • 核心组件：
      • Instruction：指令输入，来自指令内存
      • Registers：寄存器堆，存储和读取寄存器操作数
    • 数据流向：
      • 指令 → 寄存器堆（输入寄存器编号：Read register 1、Read register 2、Write register）
      • 寄存器堆 → ALU（输出操作数：Read data 1、Read data 2）
      • 寄存器堆 → 数据内存（输出操作数：Read data 2，仅Store指令使用）
    • 控制信号：
      • RegWrite：寄存器写控制信号，1表示允许写入，0表示禁止写入

    （四）ALU运算路径¶

    • 核心组件：
      • ALU：算术逻辑单元，执行算术逻辑运算和地址计算
      • Imm Gen：立即数生成单元，将指令中的立即数扩展为64位
      • ALUSrc：ALU源选择器，选择ALU的第二个操作数（寄存器值或立即数）
    • 数据流向：
      • 寄存器堆 → ALU（输入第一个操作数：Read data 1）
      • 寄存器堆 → ALUSrc → ALU（输入第二个操作数：Read data 2，仅R型指令使用）
      • 立即数生成单元 → ALUSrc → ALU（输入第二个操作数：立即数，除R型指令外的所有指令使用）
      • ALU → 数据内存（输出地址：ALU result，仅Load和Store指令使用）
      • ALU → MemtoReg → 寄存器堆（输出运算结果：ALU result，R型、I型、U型、J型指令使用）
    • 控制信号：
      • ALU operation：ALU操作控制信号，指定ALU执行的运算类型
      • ALUSrc：ALU源选择控制信号，0表示选择寄存器值，1表示选择立即数

    （五）内存访问路径¶

    • 核心组件：
      • Data memory：数据内存，存储和读取数据
      • MemtoReg：内存到寄存器选择器，选择写回寄存器的数据（ALU结果或内存读取数据）
    • 数据流向：
      • ALU → 数据内存（输入地址：ALU result）
      • 寄存器堆 → 数据内存（输入数据：Read data 2，仅Store指令使用）
      • 数据内存 → MemtoReg → 寄存器堆（输出数据：Read data，仅Load指令使用）
    • 控制信号：
      • MemWrite：内存写控制信号，1表示允许写入，0表示禁止写入（仅Store指令使用）
      • MemRead：内存读控制信号，1表示允许读取，0表示禁止读取（仅Load指令使用）
      • MemtoReg：内存到寄存器选择控制信号，0表示选择ALU结果，1表示选择内存读取数据（仅Load指令使用）

    （六）分支控制路径¶

    • 核心组件：
      • ALU：算术逻辑单元，执行操作数比较
      • Control：控制单元，生成分支控制信号
    • 数据流向：
      • ALU → 控制单元（输入Zero标志位）
      • 控制单元 → Mux（输入PCSrc控制信号）
    • 控制信号：
      • Branch：分支控制信号，1表示分支条件满足，0表示分支条件不满足
      • PCSrc：PC源选择控制信号，0表示选择当前PC + 4，1表示选择分支目标地址

    十、RISC-V处理器控制单元¶

    ![[Pasted image 20260408184432.png]]

    （一）ALU控制¶

    • 核心定位：讲解ALU控制信号与运算功能的对应关系
    • 关键内容：
      • ALU的四类核心运算：AND、OR、add、subtract
      • 对应的四位控制编码：0000、0001、0010、0110
      • 不同指令类型对ALU的需求：
        • Load/Store指令：使用add运算计算内存地址
        • Branch指令：使用subtract运算进行条件比较
        • R-type指令：根据opcode字段选择不同运算

    （二）指令格式与控制信号的对应关系¶

    1. 四种指令格式¶

    ![[Pasted image 20260408184343.png]] - R-type指令：用于算术逻辑运算，包含funct7、rs2、rs1、funct3、rd、opcode字段 - I-type指令：用于立即数运算和加载指令，包含imm12、rs1、funct3、rd、opcode字段 - S-type指令：用于存储指令，包含imm12[11:5]、rs2、rs1、funct3、imm12[4:0]、opcode字段 - SB-type指令：用于分支指令，包含imm12[12]、imm12[10:5]、rs2、rs1、funct3、imm12[4:1]、imm12[11]、opcode字段

    2. 控制信号的生成逻辑¶

    • ALUOp：由指令的opcode字段决定，用于指定ALU的运算类型
    • Funct7 field：仅R-type指令有，用于进一步指定ALU的运算类型
    • Funct3 field：用于指定具体的运算类型或数据类型
    • Operation：最终的ALU运算控制信号，由ALUOp、Funct7 field、Funct3 field共同决定

    十一、控制信号功能¶

    （一）控制信号的核心作用¶

    1. 协调硬件模块¶

    • 核心功能：指挥寄存器堆、ALU、内存等硬件模块协同工作
    • 示例：
      • RegWrite信号控制寄存器堆的写操作
      • MemWrite信号控制数据内存的写操作
      • ALU operation信号控制ALU执行的运算类型

    2. 选择数据通路¶

    • 核心功能：通过多路复用器选择不同的数据通路，实现不同指令类型的执行
    • 示例：
      • ALUSrc信号选择ALU的第二个操作数（寄存器值或立即数）
      • MemtoReg信号选择写回寄存器的数据（ALU结果或内存读取数据）
      • PCSrc信号选择下一条指令的地址（当前PC + 4 或 分支目标地址）

    3. 控制操作时序¶

    • 核心功能：确保各个硬件模块在正确的时间执行正确的操作
    • 示例：
      • 时钟信号控制寄存器的更新时机
      • 控制信号的生成和传输时序必须与指令执行的各个阶段严格同步

    4. 实现指令功能¶

    • 核心功能：不同的控制信号组合实现不同的指令功能
    • 示例：
      • R型指令：RegWrite=1，MemWrite=0，ALUSrc=0，MemtoReg=0
      • Load指令：RegWrite=1，MemWrite=0，ALUSrc=1，MemtoReg=1
      • Store指令：RegWrite=0，MemWrite=1，ALUSrc=1，MemtoReg=0

    （二）具体控制信号解析¶

    1. 主控制单元生成的控制信号¶

    （三）控制信号的生成与传输路径¶

    ![[Pasted image 20260408185028.png]]

    1. 控制信号的生成¶

    • 核心组件：Control控制单元和ALU controlALU控制单元
    • 输入信号：
      • Instruction [6-0]：指令的opcode字段
      • Instruction [30,14-12]：指令的funct7和funct3字段
    • 输出信号：
      • Control控制单元输出7种控制信号：Branch、MemRead、MemtoReg、ALUOp、MemWrite、ALUSrc、RegWrite

    2. 控制信号的传输路径¶

    • 从控制单元到硬件模块：控制信号通过蓝色连线从控制单元传输到对应的硬件模块
    • 从硬件模块到控制单元：ALU的Zero标志位通过黑色连线传输到控制单元，用于分支条件判断
    • 同步时序：所有控制信号的生成和传输必须与时钟信号严格同步，确保指令执行的正确性

    十二、CPU性能优化¶

    （一）单周期数据通路的性能瓶颈¶

    ![[Pasted image 20260408185225.png]]

    1. 时钟周期的限制¶

    • 核心问题：单周期数据通路的时钟周期必须大于最长指令的执行时间
    • 关键路径：加载指令（Load instruction）的执行路径是最长的，包括：
      • 指令内存读取
      • 寄存器堆读取
      • ALU计算内存地址
      • 数据内存读取
      • 寄存器堆写回
    • 性能影响：时钟周期较长，系统性能较低，硬件资源利用率较低

    2. 时钟周期的固定性¶

    • 核心问题：为不同指令改变时钟周期不可行，因为：
      • 时钟信号是全局同步的，不能为不同指令设置不同的时钟周期
      • 指令的执行时间是变化的，很难动态调整时钟周期
    • 性能影响：所有指令都必须等待最长指令的执行时间，导致硬件资源利用率较低

    （二）设计原则的违反¶

    1. "让常见情况更快"原则¶

    • 设计原则：优化常见情况的性能，而不是平均情况或最坏情况
    • 违反原因：单周期数据通路中，所有指令都必须等待最长指令的执行时间，即使大多数指令的执行时间较短
    • 性能影响：常见情况的性能没有得到优化，系统整体性能较低

    2. 性能优化的方向¶

    • 核心思路：通过流水线（Pipelining）将指令的执行过程划分为多个阶段，每个阶段在不同的硬件模块中并行执行
    • 流水线的优势：
      • 提高硬件资源利用率
      • 缩短时钟周期，提高系统性能
      • 保持指令的串行执行语义，但实现硬件的并行操作
    • 流水线的挑战：
      • 数据冒险（Data Hazard）：指令之间的数据依赖导致的流水线停顿
      • 控制冒险（Control Hazard）：分支指令导致的流水线停顿
      • 结构冒险（Structural Hazard）：硬件资源竞争导致的流水线停顿

    （三）流水线的基本原理¶

    1. 流水线的阶段划分¶

    • 典型阶段：
      • 取指（IF） ：从指令内存中读取指令
      • 译码（ID） ：解析指令的操作码和操作数，读取寄存器值
      • 执行（EX） ：执行ALU运算或地址计算
      • 访存（MEM） ：访问数据内存（仅Load和Store指令）
      • 写回（WB） ：将运算结果写回寄存器（仅R型、Load、I型、U型、J型指令）
    • 并行执行：不同指令的不同阶段在不同的硬件模块中并行执行
    • 性能提升：理想情况下，流水线可以将系统性能提高N倍（N为流水线的阶段数）

    2. 流水线的性能分析¶

    • 时钟周期：流水线的时钟周期等于最长阶段的执行时间
    • 指令吞吐量：单位时间内执行的指令数，等于时钟频率
    • 指令延迟：一条指令从取指到写回的总时间，等于时钟周期乘以流水线的阶段数
    • 加速比：流水线系统的性能与单周期系统的性能之比，理想情况下等于流水线的阶段数

    （一）RISC-V流水线的基本原理¶

    1. 五级流水线¶

    • 阶段划分：IF（取指）、ID（译码）、EX（执行）、MEM（访存）、WB（写回）
    • 每个阶段的功能：
      • IF：从指令内存中读取指令
      • ID：解析指令的操作码和操作数，读取寄存器值
      • EX：执行ALU运算或地址计算

      • WB：将运算结果写回寄存器（仅R型、Load、I型、U型、J型指令）
    • 并行执行机制：不同指令的不同阶段在不同的硬件模块中并行执行
    • 性能提升原理：理想情况下，流水线可以将系统性能提高N倍（N为流水线的阶段数）

    2. 流水线的洗衣店类比¶

    • 关键内容：
      • 洗衣的四个阶段：洗衣、烘干、折叠、存储
      • 流水线式洗衣的任务重叠执行状态
      • 流水线的加速比计算
      • 流水线的性能提升原理

    （二）流水线的并行执行机制¶

    • 并行执行：不同指令的不同阶段在不同的硬件模块中并行执行
    • 示例：当指令1在执行阶段时，指令2在译码阶段，指令3在取指阶段
    • 性能提升：理想情况下，流水线可以将系统性能提高N倍（N为流水线的阶段数）

    （三）流水线的加速比计算¶

    • 加速比公式：Speedup = 非流水线执行时间 / 流水线执行时间
    • 示例：
      • 四批衣服的非流水线执行时间为8小时，流水线执行时间为3.5小时，加速比为8/3.5≈2.3
      • 连续执行n批衣服的非流水线执行时间为2n小时，流水线执行时间为0.5n+1.5小时，加速比约为4（等于流水线的阶段数）

    十四、流水线冒险（Hazards）¶

    （一）冒险的定义与分类¶

    流水线冒险是指阻止下一条指令在下一个时钟周期开始执行的情况。冒险会破坏流水线的并行执行效率，导致性能下降。

    冒险类型 英文名称 核心问题 典型场景
    结构冒险 Structure Hazard 所需资源被占用 多条指令同时访问同一内存
    数据冒险 Data Hazard 等待前序指令完成数据读写 add x1, x2, x3 后紧跟 sub x4, x1, x5
    控制冒险 Control Hazard 控制决策依赖前序指令 分支指令后不知道下一条指令地址

    （二）结构冒险（Structure Hazard）¶

    1. 问题本质¶

    资源竞争：多条指令在同一时钟周期内竞争同一硬件资源。

    2. 典型场景¶

    在RISC-V流水线中，如果只有一块内存： - Load/Store指令需要访问数据内存 - 同时需要取指下一条指令 - 两者冲突，必须停顿一个周期（产生"气泡"bubble）

    3. 解决方案¶

    解决方案 实现方式 效果
    分离内存 独立的指令内存和数据内存 根本解决
    分离缓存 I-Cache 和 D-Cache 现代CPU主流方案

    结论：流水线数据通路需要独立的指令内存和数据内存，或独立的指令/数据缓存。

    （三）数据冒险（Data Hazard）¶

    1. 问题本质¶

    数据依赖：一条指令需要等待前序指令完成数据读写操作。

    2. 典型场景¶

    1
2
    add x19, x0, x1    # x19 = x0 + x1
sub x2, x19, x3    # 需要等待x19的值

    ![[Pasted image 20260408190628.png]] - add指令在EX阶段计算结果 - sub指令在ID阶段就需要读取x19 - 此时x19的值还未写入寄存器堆

    3. 解决方案一：转发（Forwarding/Bypassing）¶

    核心思想：不需要等待结果写入寄存器，直接从ALU输出端"转发"给下一条指令。

    ![[Pasted image 20260408190705.png]]

    代价：需要在数据通路中增加额外的连接线路。

    4. 解决方案二：Load-Use 冒险的特殊处理¶

    问题：Load指令的数据要到MEM阶段才能从内存读出，无法通过转发完全避免停顿。

    1
2
    ld x1, 0(x2)      # 从内存加载到x1
sub x4, x1, x5    # 需要x1的值，但此时x1还在MEM阶段

    解决方案： 1. 插入气泡：在Load指令后插入一个停顿周期 ![[Pasted image 20260408190729.png]] 2. 代码调度：编译器重排指令，避免紧跟使用Load结果 ![[Pasted image 20260408190746.png]]

    （四）控制冒险（Control Hazard）¶

    1. 问题本质¶

    分支决策依赖：分支指令决定了程序的控制流，但分支结果要等到流水线后期才能确定。

    2. 典型场景¶

    • 分支判断需要在EX阶段比较寄存器
    • 分支目标地址需要在ID阶段计算
    • 此时流水线已经取指了后续指令，但不确定是否正确

    3. 解决方案一：分支停顿（Stall on Branch）¶

    ![[Pasted image 20260408190937.png]] 方法：等待分支结果确定后再取下一条指令。

    代价：每个分支指令导致2个时钟周期的停顿（气泡）。

    4. 解决方案二：静态分支预测¶

    预测策略 规则 适用场景
    预测不跳转 总是取下一条顺序指令 简单实现
    后向跳转预测跳转 循环通常需要跳转回去 循环优化
    前向跳转预测不跳转 if语句通常不跳转 条件判断

    5. 解决方案三：动态分支预测¶

    核心思想：硬件记录每个分支的历史行为，预测未来行为。

    • 分支历史表（BHT）：记录每个分支最近的跳转/不跳转历史
    • 预测逻辑：假设未来行为会延续历史趋势
    • 错误恢复：预测错误时停顿并重新取指，同时更新历史

    现代CPU的预测准确率可达95%以上，显著降低控制冒险的性能损失。

    （五）冒险检测与处理总结¶

    冒险类型 检测方法 处理策略 性能影响
    结构冒险 资源冲突检测 分离指令/数据内存 已基本解决
    数据冒险 寄存器编号比较 转发 + 停顿 + 代码调度 1-2周期/冒险
    控制冒险 分支指令识别 预测 + 停顿 2周期/预测错误

    十六、流水线性能分析¶

    （一）理想情况下的性能¶

    1. 假设条件¶

    • 各阶段执行时间相等
    • 没有冒险发生
    • 流水线可以持续充满

    2. 性能指标¶

    指标 单周期CPU 流水线CPU
    时钟周期 800ps（最长指令） 200ps（最长阶段）
    指令延迟 800ps 1000ps（5阶段×200ps）
    吞吐量 1/800ps 1/200ps
    加速比 1× 4×

    结论：流水线提高了吞吐量（每秒完成更多指令），但不降低延迟（单条指令执行时间不变甚至更长）。

    （二）实际性能分析¶

    1. 性能损失来源¶

    来源 原因 损失周期数
    数据冒险 Load-Use冒险 1周期/冒险
    控制冒险 分支预测错误 2周期/错误
    结构冒险 资源冲突 已基本消除

    2. CPI计算¶

    \[\text{CPI}_{\text{流水线}} = 1 + \text{停顿周期数}/\text{指令数}\]

    示例：假设程序中20%是分支指令，分支预测错误率为10%，Load指令占20%，其中30%产生Load-Use冒险。

    \[\text{CPI} = 1 + 0.2 \times 0.1 \times 2 + 0.2 \times 0.3 \times 1 = 1 + 0.04 + 0.06 = 1.1\]

    （三）RISC-V架构的流水线友好设计¶

    设计特性 对流水线的好处
    固定32位指令长度 取指和解码在一个周期内完成（对比x86的1-17字节可变长度）
    少量且规整的指令格式 译码和读寄存器可在一周期完成
    Load/Store架构 地址计算在第3阶段，内存访问在第4阶段，职责清晰
    大量寄存器 减少内存访问，降低数据冒险概率

    十七、流水线总结¶

    （一）核心要点¶

    1. 流水线通过重叠执行提高指令吞吐量
    2. 多条指令同时处于不同执行阶段

    3. 理想加速比 = 流水线阶段数

    4. 三类冒险是流水线的主要挑战

    5. 结构冒险：资源冲突 → 分离内存/缓存
    6. 数据冒险：数据依赖 → 转发 + 停顿 + 代码调度

    7. 控制冒险：分支决策 → 预测 + 停顿

    8. 指令集设计影响流水线实现复杂度

    9. RISC-V的设计天然适合流水线
    10. 固定长度、规整格式、Load/Store架构

    （二）流水线与单周期对比¶

    特性 单周期CPU 流水线CPU
    时钟周期 最长指令时间 最长阶段时间
    硬件资源利用率 低（大部分时间闲置） 高（并行执行）
    吞吐量 低 高（理想4倍）
    设计复杂度 简单 复杂（需处理冒险）
