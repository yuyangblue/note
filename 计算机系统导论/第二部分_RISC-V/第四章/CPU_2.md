### （一）流水线寄存器

#### 1. 必要性

流水线各阶段之间需要**寄存器**来保存前一个周期产生的信息，供下一个周期使用。

![[Pasted image 20260408191236.png]]
#### 2. 四个流水线寄存器

![[Pasted image 20260408191302.png]]

| 寄存器        | 存储内容                    | 作用        |
| ---------- | ----------------------- | --------- |
| **IF/ID**  | 取指的指令、PC+4              | 为译码阶段提供指令 |
| **ID/EX**  | 操作数、立即数、控制信号、目标寄存器、PC+4 | 为执行阶段提供数据 |
| **EX/MEM** | ALU结果、写数据、控制信号          | 为访存阶段提供数据 |
| **MEM/WB** | 内存读取数据/ALU结果、控制信号       | 为写回阶段提供数据 |

## 二、流水线控制

### （一）简化的流水线控制架构

![[幻灯片61.png]]

**核心原理**：
- 控制信号由指令的opcode字段产生
- 与单周期实现类似，但控制信号需要**随流水线传递**

**这张图展示了什么？**

从图可以看到：
1. **控制单元（Control）**：接收指令的opcode，输出各种控制信号
2. **蓝色线条**：控制信号的传输路径
3. **控制信号的作用位置**：寄存器堆、ALU、数据内存、多路选择器

**控制信号的具体作用**：

| 控制信号 | 控制哪个部件 | 作用 |
|---------|-------------|------|
| `RegWrite` | 寄存器堆 | 是否允许写寄存器 |
| `MemRead`/`MemWrite` | 数据内存 | 读/写内存 |
| `ALUSrc` | ALU的Mux | ALU第二个操作数来源 |
| `ALUOp` | ALU控制单元 | ALU执行什么运算 |
| `MemtoReg` | WB阶段的Mux | 写回数据来自内存还是ALU |
| `Branch` | PC的Mux | 是否分支跳转 |

### （二）控制信号的传递

![[幻灯片62.png]]

**关键问题**：控制信号在ID阶段产生，但什么时候使用？

| 控制信号 | 产生阶段 | 使用阶段 | 需要传递吗？ |
|---------|---------|---------|-------------|
| `ALUSrc` | ID | EX | ✅ 需要传到ID/EX |
| `ALUOp` | ID | EX | ✅ 需要传到ID/EX |
| `MemRead`/`MemWrite` | ID | MEM | ✅ 需要传到ID/EX，再传到EX/MEM |
| `MemtoReg` | ID | WB | ✅ 需要一路传到MEM/WB |
| `RegWrite` | ID | WB | ✅ 需要一路传到MEM/WB |

**传递规则**：

```
控制信号的生命周期：
ID阶段产生 → ID/EX保存 → EX/MEM保存 → MEM/WB保存

每个流水线寄存器只保存"后续阶段还需要"的信号：
- ID/EX：保存 EX、MEM、WB 三类信号
- EX/MEM：保存 MEM、WB 两类信号（EX阶段已用完的不用传）
- MEM/WB：只保存 WB 信号
```

**为什么这样设计？**

| 原因 | 说明 |
|-----|------|
| **避免信号丢失** | 下一个时钟周期，新的指令进入ID阶段，控制信号会被覆盖 |
| **匹配指令进度** | 控制信号要"跟着指令走"，每到一个阶段就用对应的信号 |
| **减少硬件开销** | 不需要每个阶段都重新解码，只解码一次然后传递 |

## 三、数据冒险

### （一）什么是数据冒险

![[幻灯片64.png]]

**问题场景**：

```assembly
sub  x2, x1, x3    # x2 被写入
and  x12, x2, x5   # 需要读取 x2
or   x13, x6, x2   # 需要读取 x2
add  x14, x2, x2   # 需要读取 x2
sd   x15, 100(x2)  # 需要 x2 作为地址
```

**问题本质**：后续指令需要使用前序指令的结果，但结果还未写入寄存器。

**时序分析**：

```
时钟周期：    CC1     CC2     CC3     CC4     CC5
sub x2...   [IF] → [ID] → [EX] →[MEM] → [WB]
                         ↑ x2在这里计算出来
and x12...         [IF] → [ID] → [EX] → ...
                          ↑ 这里就需要x2，但x2还没写入！
```

**图中的关键信息**：
- 左侧 `x2` 以浅灰色高亮显示，表示这是"问题寄存器"
- 5条指令都涉及x2，形成一条"数据依赖链"
- 右下角写着"We can resolve hazards with forwarding"——这就是解决方案

---

### （二）解决方案：前向转发（Forwarding）

![[幻灯片65.png]]

**核心思想**：不需要等待结果写回寄存器，直接从ALU输出端"转发"给需要它的指令。

**这张图展示了什么？**

从图中可以看到两条数据通路：
- **红色线**：原本的数据流——从寄存器堆读取（但此时数据还没写入）
- **蓝色线**：转发路径——从ALU输出或MEM阶段直接传递

**时间轴上的观察**：

图中上半部分是时间轴（CC1-CC9），下半部分是流水线执行图：
- 第一行显示 `x2` 的值在不同时钟周期的变化：CC1-CC4 是旧值(10)，CC5 开始是新值(-20)
- 蓝色转发线从 `sub` 的 EX/MEM 阶段连接到 `and` 的 EX 阶段
- 这意味着：`sub` 在EX阶段算出的结果，**不经过寄存器堆**，直接送给 `and` 的ALU

**转发的本质**：

| 传统方式 | 转发方式 |
|---------|---------|
| 结果写回寄存器 → 下条指令从寄存器读 | 结果直接送到下条指令的ALU输入 |
| 需要等待WB阶段完成 | EX阶段结束后立即可用 |
| 可能产生停顿 | 无停顿 |

---

### （三）如何检测需要转发


**核心问题**：如何知道"当前指令需要的寄存器正好是前序指令要写的寄存器"？

**检测方法**：**比较寄存器编号**

**必要前提**：寄存器编号需要随流水线传递

**关键信号**：

| 信号 | 来源 | 含义 |
|-----|------|------|
| `ID/EX.RegisterRs1` | ID/EX寄存器 | 当前EX阶段指令的第一个源寄存器编号 |
| `ID/EX.RegisterRs2` | ID/EX寄存器 | 当前EX阶段指令的第二个源寄存器编号 |
| `EX/MEM.RegisterRd` | EX/MEM寄存器 | 刚完成EX阶段指令的目标寄存器编号 |
| `MEM/WB.RegisterRd` | MEM/WB寄存器 | 刚完成MEM阶段指令的目标寄存器编号 |

**转发检测逻辑**：

图中用大括号分组展示了检测条件：

```
情况1：从 EX/MEM 转发（前一条指令的结果）
├─ 1a: EX/MEM.RegisterRd = ID/EX.RegisterRs1 → 转发到 ForwardA
└─ 1b: EX/MEM.RegisterRd = ID/EX.RegisterRs2 → 转发到 ForwardB

情况2：从 MEM/WB 转发（前前一条指令的结果）
├─ 2a: MEM/WB.RegisterRd = ID/EX.RegisterRs1 → 转发到 ForwardA
└─ 2b: MEM/WB.RegisterRd = ID/EX.RegisterRs2 → 转发到 ForwardB
```

**图中的灰蓝色圆角矩形**：
- 右上角的框"Fwd from EX/MEM pipeline reg"对应情况1
- 右下角的框"Fwd from MEM/WB pipeline reg"对应情况2

### （四）转发的额外条件

**问题**：不是所有"寄存器编号匹配"都需要转发！

**条件1：该指令确实要写寄存器**

**为什么需要？**
- 如果前序指令是 Store 指令，它不写寄存器（RegWrite=0）
- 此时 Rd 字段的值是"无意义"的，不应该用它来转发

**条件2：目标寄存器不是x0**

图中的代码：`EX/MEM.RegisterRd ≠ 0, MEM/WB.RegisterRd ≠ 0`

**为什么需要？**
- x0 是硬连线到0的寄存器，写入x0不会改变任何值
- 很多指令用 x0 作为"丢弃结果"的方式，比如 `add x0, x1, x2` 只是为了设置条件码
- 此时不需要转发

**完整的转发检测条件**：

```
EX转发条件：
if (EX/MEM.RegWrite = 1) 
   and (EX/MEM.RegisterRd ≠ 0) 
   and (EX/MEM.RegisterRd = ID/EX.RegisterRs1 或 Rs2):
    执行转发

MEM转发条件：
if (MEM/WB.RegWrite = 1) 
   and (MEM/WB.RegisterRd ≠ 0) 
   and (MEM/WB.RegisterRd = ID/EX.RegisterRs1 或 Rs2):
    执行转发
```

### （六）双重数据冒险

![[幻灯片70.png]]

**问题场景**：

```assembly
add x1, x1, x2    # x1 被写入（第一次）
add x1, x1, x3    # x1 又被写入（第二次），同时需要读取 x1
add x1, x1, x4    # x1 又被写入（第三次），同时需要读取 x1
```

**图中关键信息**：
- `x1` 以浅灰色显示，表示这是"问题寄存器"
- 两条前序指令都要写入 x1
- 第三条指令需要 x1，应该转发哪个？

**问题分析**：

```
时钟周期：   CC1     CC2     CC3     CC4     CC5     CC6
add x1,x1,x2   [IF]→[ID]→[EX]→[MEM]→[WB]
                           ↑ 第一次写入x1
add x1,x1,x3         [IF]→[ID]→[EX]→[MEM]→[WB]
                                ↑ 第二次写入x1，同时需要x1
add x1,x1,x4               [IF]→[ID]→[EX]→...
                                     ↑ 需要x1，应该用哪个？
```

**解决原则**：

| 原则 | 说明 |
|-----|------|
| **使用最新的值** | 转发距离当前指令最近的那个结果 |
| **MEM转发要加条件** | 只有当EX转发条件不成立时，才进行MEM转发 |

**修改后的MEM转发条件**：

```
原条件：MEM/WB.RegisterRd = ID/EX.RegisterRs1 或 Rs2

修正条件：
if (MEM/WB.RegisterRd = ID/EX.RegisterRs1 或 Rs2)
   and NOT (EX/MEM.RegisterRd = ID/EX.RegisterRs1 或 Rs2):  ← 新增条件
    执行MEM转发
```

![[幻灯片72.png]]

这张图是经典的 **带转发（Forwarding/Bypassing）的流水线数据通路**。

## Forwarding unit（转发单元）的输入

转发单元需要比较"**当前指令要用到的寄存器**"和"**前面指令要写入的寄存器**"是否冲突，所以输入包括：

| 输入信号 | 来源 | 含义 |
|----------|------|------|
| **ID/EX.RegisterRs1** | ID/EX 流水线寄存器 | 当前指令的源操作数 1 的寄存器编号 |
| **ID/EX.RegisterRs2** | ID/EX 流水线寄存器 | 当前指令的源操作数 2 的寄存器编号 |
| **EX/MEM.RegisterRd** | EX/MEM 流水线寄存器 | **上一条**指令的目标寄存器编号 |
| **EX/MEM.RegWrite** | EX/MEM 控制信号 | **上一条**指令是否写回寄存器 |
| **MEM/WB.RegisterRd** | MEM/WB 流水线寄存器 | **上上条**指令的目标寄存器编号 |
| **MEM/WB.RegWrite** | MEM/WB 控制信号 | **上上条**指令是否写回寄存器 |

> 简单说：转发单元接收 **当前指令的两个源寄存器号**，以及 **前面两条指令的目标寄存器号和写使能信号**。

---

## Forwarding unit 的输出

输出是**两个 2 位的控制信号**，分别控制 ALU 输入端的两个多路选择器（Mux）：

| 输出信号 | 控制对象 | 含义 |
|----------|----------|------|
| **ForwardA**（2位） | ALU 上面的 Mux | 选择 ALU 第一个操作数的来源 |
| **ForwardB**（2位） | ALU 下面的 Mux | 选择 ALU 第二个操作数的来源 |

---

## Mux 的选择逻辑（ForwardA / ForwardB）

| 信号值 | 选择来源 | 场景 |
|--------|----------|------|
| **00** | 寄存器堆读出的值 | 正常情况，无数据冒险 |
| **10** | **EX/MEM 流水线寄存器**（ALU result） | 上一条指令的结果刚算出来，直接前递 |
| **01** | **MEM/WB 流水线寄存器**（Write Data） | 上上条指令的结果，从 MEM/WB 前递 |

---

## 一句话总结

> **转发单元输入"当前指令要读谁"和"前面指令要写谁"，判断是否有 RAW（写后读）冲突；输出两个 Mux 控制信号，让 ALU 直接使用前面指令的运算结果，而不必等到写回寄存器堆。**
---

## 四、数据冒险总结

| 冒险类型 | 检测方法 | 解决方案 | 性能影响 |
|---------|---------|---------|---------|
| ALU指令数据冒险 | 寄存器编号匹配 | 前向转发 | 无停顿 |
| Load-Use冒险 | Load后紧跟使用 | 插入气泡 + 转发 | 1周期停顿 |
| 双重数据冒险 | 同一寄存器多次写入 | 选择最新的值 | 无停顿 |

## 七、Load-Use 冒险

### （一）什么是 Load-Use 冒险

**为什么 Load-Use 冒险不能用转发解决？**

普通的数据冒险（如 `sub` 后跟 `and`）可以用转发解决，因为：
- `sub` 在 EX 阶段就计算出了结果
- 可以在同一个周期内转发给下一条指令的 EX 阶段

但 Load 指令不一样：

```
时钟周期：   CC1     CC2     CC3     CC4     CC5
ld x2,20(x1)  [IF] → [ID] → [EX] →[MEM] → [WB]
                              ↑ 计算地址
                                    ↑ 数据从这里才能读取！
and x4,x2,x5         [IF] → [ID] → [EX] → ...
                                   ↑ 这里就需要x2，但数据还在内存中！
```

**问题本质**：
- Load 指令要到 **MEM 阶段** 才能从内存读取数据
- 下一条指令的 **EX 阶段** 就需要这个数据
- 即使转发，也需要"未来"的数据（不可能实现）

**检测条件**（在 ID 阶段检测）：

检测条件：

```
Load-Use冒险检测：
ID/EX.MemRead = 1                                    ← 前一条是Load指令
且 (ID/EX.RegisterRd = IF/ID.RegisterRs1             ← 当前指令的源寄存器1 = Load的目标寄存器
    或 ID/ID.RegisterRd = IF/ID.RegisterRs2)         ← 或源寄存器2 = Load的目标寄存器
```

**关键信息**：
- 检测在 **ID 阶段** 进行（"Check when using instruction is decoded in ID stage"）
- 需要比较 Load 指令的目标寄存器和当前指令的源寄存器
- 如果检测到冒险，"stall and insert bubble"——暂停并插入气泡

---

### （三）Load-Use 冒险示例

![[幻灯片75.png]]

**这张图非常清晰地展示了 Load-Use 冒险的处理过程**

**代码示例**：

```assembly
ld x2, 20(x1)    # Load指令：从内存地址(x1+20)读取数据到x2
and x4, x2, x5   # 紧跟使用x2
or x8, x2, x6
add x9, x4, x2
```

**从图中观察流水线状态**：

时间轴（CC1 - CC10）显示：

```
            CC1   CC2   CC3   CC4   CC5   CC6   CC7   CC8   CC9   CC10
ld x2,20    [IF]  [ID]  [EX] [MEM]  [WB]
                        ↑ 计算地址
                              ↑ 数据从内存读出
                                    ↑ 转发给and的EX阶段

and x4,x2         [IF]  [ID]  气泡   [IF]  [ID]  [EX] [MEM]  [WB]
                              ↑ "Stall inserted here"
```

**图中的关键标注**：

1. **浅蓝色云状图形**：标注"bubble"，表示这是插入的气泡（停顿周期）
2. **箭头指向气泡**："Stall inserted here"——指出停顿的位置
3. **第二条指令的 IF 阶段重复**：`and` 的 IF 阶段在 CC3 和 CC5 都出现了（中间是气泡）

**为什么气泡在第4列？**

```
CC3: ld 在 EX 阶段，and 在 ID 阶段
     此时检测到 Load-Use 冒险
     强制 ID/EX 控制信号 = 0

CC4: ld 在 MEM 阶段（正在读内存）
     ID/EX 的控制信号是 0，所以 EX 阶段不工作 → 气泡
     PC 和 IF/ID 没有更新

CC5: ld 在 WB 阶段
     and 重新进入 ID 阶段
     此时 x2 的数据已经可用
```

### （四）带冒险检测的完整数据通路

![[幻灯片76.png]]

**这张图展示了什么？**

这是带冒险检测机制的完整CPU数据通路。与之前"带转发的数据通路"相比，新增了：

| 新增组件 | 功能 | 位置 |
|---------|------|------|
| **Hazard Detection Unit** | 检测 Load-Use 冒险 | ID 阶段，Control 单元旁边 |
| **PCWrite** 信号 | 控制 PC 是否更新 | 从 Hazard Unit 连到 PC |
| **IF/IDWrite** 信号 | 控制 IF/ID 是否更新 | 从 Hazard Unit 连到 IF/ID |
| **Stall 信号** | 强制 ID/EX 控制信号为 0 | 从 Hazard Unit 连到 ID/EX |

**冒险检测单元的工作流程**：

```
1. 读取输入信号：
   - ID/EX.MemRead：前一条指令是否是 Load
   - ID/EX.RegisterRd：Load 的目标寄存器
   - IF/ID.RegisterRs1、Rs2：当前指令的源寄存器

2. 检测条件：
   if (ID/EX.MemRead = 1) and 
      (ID/EX.RegisterRd = IF/ID.RegisterRs1 或 Rs2):
      检测到 Load-Use 冒险！

3. 执行暂停：
   - PCWrite = 0        → PC 不更新
   - IF/IDWrite = 0     → IF/ID 不更新
   - Stall = 1          → ID/EX 控制信号强制为 0
```

**图中的蓝色控制线**：
- 从 Hazard Detection Unit 发出的蓝色线连接到 PC、IF/ID 和 ID/EX
- 这些就是暂停信号，用于控制流水线的"冻结"

## 八、停顿与性能

![[幻灯片77.png]]

**这张图的核心观点**：

图中用不同颜色的方块项目符号展示三个观点：

| 观点 | 说明 |
|-----|------|
| **Stalls reduce performance**（停顿降低性能） | 每次停顿都浪费了流水线的吞吐能力 |
| **But are required to get correct results**（但必须保证正确性） | 停顿是为了获得正确的结果 |
| **Compiler can arrange code to avoid hazards and stalls**（编译器可以优化） | 通过代码重排避免冒险和停顿 |

**编译器优化示例**：

```assembly
# 原始代码（有 Load-Use 冒险）

ld x2, 20(x1)

add x4, x2, x5    # 现在可以正常执行，无需停顿
```

**性能对比**：

| 情况 | 周期数 | 说明 |
|-----|-------|------|
| 原始代码 | 3 + 1(停顿) = 4 | 浪费 1 个周期 |
| 编译器重排后 | 3 | 无停顿，吞吐量更高 |

## 九、控制冒险（分支冒险）

### （一）分支冒险的问题

![[幻灯片78.png]]

**什么是控制冒险？**

控制冒险发生在**分支指令**执行时：分支的目标地址要等到流水线后期才能确定，但此时已经取指了后续指令。

**问题场景**：

```assembly
40: beq x1, x0, 16    # 分支指令：如果 x1=0，跳转到 40+16*2=72
44: and x12, x2, x5   # 顺序执行的下一条指令
48: or x13, x6, x2
52: add x14, x2, x2
...
72: ld x4, 100(x7)    # 分支目标地址
```

**时序分析**：

```
时钟周期：   CC1     CC2     CC3     CC4     CC5
beq (40)      [IF] → [ID] → [EX] →[MEM] → [WB]
                                   ↑ 分支结果在这里才确定！
and (44)             [IF] → [ID] → [EX] → ...
or (48)                    [IF] → [ID] → ...
add (52)                         [IF] → ...
```

**问题**：
- 分支指令要到 **MEM 阶段** 才能确定是否跳转
- 但此时 44、48、52 处的指令已经进入流水线
- 如果分支成立，这些指令不该执行 → 需要**冲刷（Flush）**

**图中的关键标注**：

右侧用浅蓝色方框标注："Flush these instructions (Set control values to 0)"

大括号框住了三条不该执行的指令（44、48、52 处的 and、or、add），表示这些指令需要被冲刷。

---

### （二）减少分支延迟的方法

![[幻灯片79.png]]

**核心思路**：将分支判断**提前到 ID 阶段**

**具体方法**：

图中列出两个硬件改进：

| 方法 | 说明 | 效果 |
|-----|------|------|
| **Target address adder** | 在 ID 阶段计算分支目标地址 | 目标地址提前到 ID 阶段可知 |
| **Register comparator** | 在 ID 阶段比较寄存器值 | 分支条件提前到 ID 阶段可判断 |

**改进后的效果**：

```
改进前：分支在 MEM 阶段判断 → 冲刷 3 条指令 → 损失 3 个周期

改进后：分支在 ID 阶段判断 → 只需冲刷 1 条指令 → 损失 1 个周期
```

**图中的代码示例**：

```assembly
36: and x12, x2, x5
40: beq x1, x0, 16    # PC-relative branch，跳转到 40+16*2=72
44: sub x10, x4, x8
...
72: ld x4, 100(x7)    # to 40+16*2=72
```

### （四）动态分支预测

![[幻灯片82.png]]

**为什么需要动态分支预测？**

图中第一行说明："In deeper and superscalar pipelines, branch penalty is more significant"

| 处理器类型 | 流水线深度 | 分支惩罚 |
|-----------|-----------|---------|
| 简单流水线 | 5 级 | 冲刷 1-3 条指令 |
| 深度流水线 | 10-20 级 | 冲刷 10+ 条指令 |
| 超标量处理器 | 多发射 | 每周期完成多条指令，惩罚更大 |

**动态预测的核心思想**：

"Use dynamic prediction" —— 根据历史行为预测分支走向

**Branch History Table（分支历史表）**：

| 组件 | 功能 |
|-----|------|
| **存储结构** | 一个小型缓存/表 |
| **索引方式** | 用分支指令地址作为索引 |
| **存储内容** | taken（跳转）或 not taken（不跳转） |

**工作流程**：

图中详细列出了三个步骤：

| 步骤 | 操作 | 说明 |
|-----|------|------|
| **1. Check table** | 查询该分支的历史预测 | 分支指令地址作为索引查表 |
| **2. Start fetching** | 根据预测从目标或顺序地址取指 | 预测 taken → 取目标地址；预测 not taken → 顺序取指 |
| **3. If wrong** | 冲刷流水线，翻转预测 | 预测错误时，更新历史表 |

**预测准确率**：

现代 CPU 的分支预测准确率可达 **95% 以上**，意味着：
- 95% 的分支判断正确 → 无惩罚
- 5% 预测错误 → 需要冲刷流水线

**与静态预测对比**：

| 预测方式 | 策略 | 准确率 |
|---------|------|-------|
| 静态预测 | 总是预测跳转/不跳转 | 50-70% |
| 动态预测 | 根据历史预测 | 95%+ |

## 十一、关键公式汇总

### 转发检测条件

```
EX冒险：
EX/MEM.RegisterRd = ID/EX.RegisterRs1 或 Rs2
且 EX/MEM.RegWrite = 1
且 EX/MEM.RegisterRd ≠ 0
→ ForwardA/B = 10

MEM冒险：
MEM/WB.RegisterRd = ID/EX.RegisterRs1 或 Rs2
且 MEM/WB.RegWrite = 1
且 MEM/WB.RegisterRd ≠ 0
且 EX冒险不成立
→ ForwardA/B = 01
```

### Load-Use冒险检测

```
ID/EX.MemRead = 1
且 (ID/EX.RegisterRd = IF/ID.RegisterRs1 
    或 ID/EX.RegisterRd = IF/ID.RegisterRs2)
→ 插入气泡，暂停1周期
```

### 流水线暂停操作

```
if Load-Use冒险:
    ID/EX控制信号 = 0      # EX/MEM/WB 执行 nop
    PCWrite = 0            # PC 不更新
    IF/IDWrite = 0         # IF/ID 不更新
```

## 十二、动态分支预测详解

### （一）为什么需要动态预测？


**核心问题**：在更深的流水线和超标量处理器中，分支惩罚更严重。

| 处理器类型 | 流水线深度 | 分支惩罚 |
|-----------|-----------|---------|
| 简单流水线 | 5级 | 冲刷1-3条指令 |
| 深度流水线 | 10-20级 | 冲刷10+条指令 |
| 超标量处理器 | 多发射 | 每周期多条指令，惩罚翻倍 |

**动态预测的核心思想**：根据分支的历史行为来预测未来行为。

**分支历史表（Branch History Table）**：

| 特性 | 说明 |
|-----|------|
| **存储结构** | 小型缓存/表 |
| **索引方式** | 用分支指令地址作为索引 |
| **存储内容** | taken（跳转）或 not taken（不跳转）|

**工作流程**：

| 步骤 | 操作 |
|-----|------|
| **1. Check table** | 用分支指令地址查表，获取历史预测 |
| **2. Start fetching** | 根据预测从目标地址或顺序地址取指 |
| **3. If wrong** | 冲刷流水线，翻转预测值 |

**打个比方**：

想象你在猜一个人明天会不会带伞：
- 静态预测：不管三七二十一，总是猜"会带"或"不会带"
- 动态预测：看他过去一周带伞的记录，发现他晴天不带、雨天带，然后预测

---

### （二）1位预测器的缺点

![[幻灯片83.png]]

**问题场景**：嵌套循环

```assembly
outer: ...
inner: ...
       beq x1, x2, inner   # 内层循环的分支
       ...
       beq x3, x4, outer   # 外层循环的分支
```

**1位预测器的逻辑**：

| 当前预测 | 实际行为 | 下次预测 |
|---------|---------|---------|
| taken | taken | taken（正确） |
| taken | not taken | not taken（错误） |
| not taken | taken | taken（错误） |
| not taken | not taken | not taken（正确） |

**问题在哪？**

内层循环的分支会：
1. 循环结束时：实际 not taken，但预测 taken → **预测错误**
2. 下次进入内层循环：实际 taken，但预测变成 not taken → **又预测错误**

**结果**：内层循环每次迭代会预测错误**两次**！

**具体分析**：

```
内层循环第1-N次迭代：
  分支预测：taken
  实际结果：taken
  预测正确 ✓

内层循环第N+1次（最后一次）：
  分支预测：taken
  实际结果：not taken（退出循环）
  预测错误 ✗ → 翻转预测为 not taken

下次进入内层循环：
  分支预测：not taken
  实际结果：taken
  预测错误 ✗ → 翻转预测为 taken
```

---

### （三）2位预测器

![[幻灯片84.png]]

**核心改进**：只有**连续两次预测错误**才改变预测。

**状态机解释**：

图中有4个状态，分为两列：

| 左列（预测 taken） | 右列（预测 not taken） |
|------------------|---------------------|
| **强 taken**（深灰蓝色） | **强 not taken**（深灰色） |
| **弱 taken**（浅灰蓝色） | **弱 not taken**（浅灰色） |

**状态转换规则**：

```
                    Not taken                    Not taken
    强 taken ───────────────────→ 弱 taken ───────────────────→ 弱 not taken
       ↑                              │                              │
       │ Taken                        │ Taken                        │ Not taken
       │                              ↓                              ↓
       └──────────────────────────────┴──────────────────────────────┘
                                                                    │
                                                                Taken
                                                                    │
                                                                    ↓
    强 not taken ←─────────────── 弱 not taken ←─────────────────────┘
              Not taken                  Not taken
```

**用表格理解**：

| 当前状态 | 预测 | 实际taken | 实际not taken |
|---------|------|----------|--------------|
| 强 taken | taken | 保持强 taken | 变成弱 taken |
| 弱 taken | taken | 变成强 taken | 变成弱 not taken |
| 弱 not taken | not taken | 变成弱 taken | 变成强 not taken |
| 强 not taken | not taken | 变成弱 not taken | 保持强 not taken |

**为什么能解决嵌套循环问题？**

```
内层循环最后一次：taken → not taken
  强 taken → 弱 taken（预测仍然是 taken）

下次进入内层循环：not taken → taken
  弱 taken → 强 taken（预测正确！）
```

**预测错误次数对比**：

| 预测器类型 | 内层循环每次迭代错误次数 |
|-----------|----------------------|
| 1位预测器 | 2次 |
| 2位预测器 | 0次（大部分情况） |

### （四）计算分支目标地址

![[幻灯片85.png]]

**问题**：即使有了预测器，仍然需要计算目标地址，会有1个周期的惩罚。

**解决方案：Branch Target Buffer（分支目标缓冲）**

| 特性 | 说明 |
|-----|------|
| **存储内容** | 目标地址的缓存 |
| **索引方式** | 用PC（程序计数器）索引 |
| **查找时机** | 指令取指时 |

**工作流程**：

```
1. 用当前PC查分支目标缓冲
2. 如果命中，且该指令是分支指令，预测taken
3. 直接从缓存的目标地址取指，无需计算
```

**效果**：

| 情况 | 周期惩罚 |
|-----|---------|
| 没有分支目标缓冲 | 1周期（计算目标地址） |
| 有分支目标缓冲，命中 | 0周期 |
| 有分支目标缓冲，未命中 | 1周期 |

## 十三、异常和中断

### （一）什么是异常和中断


**定义**：需要改变控制流的"意外"事件。

**异常 vs 中断**：

| 类型 | 来源 | 例子 |
|-----|------|-----|
| **异常（Exception）** | CPU内部产生 | 未定义操作码、系统调用、除零错误 |
| **中断（Interrupt）** | 外部I/O控制器产生 | 键盘输入、定时器中断、网络数据到达 |

**为什么要处理？**

- 用户按了Ctrl+C → 需要终止程序
- 程序访问非法地址 → 需要报错
- 定时器到期 → 需要切换进程

**挑战**：在不牺牲性能的情况下处理它们。

---

### （二）如何处理异常

**三个关键步骤**：

| 步骤 | 操作 | RISC-V中的实现 |
|-----|------|---------------|
| **1. 保存PC** | 保存引发异常（或被中断）的指令地址 | SEPC（Supervisor Exception Program Counter） |
| **2. 保存原因** | 记录是什么问题 | SCAUSE（Supervisor Exception Cause Register） |
| **3. 跳转到处理程序** | 转移控制权 | 默认地址：0x1C090000 |

**SCAUSE寄存器结构**：

```
┌────────────────────────────────────────────────────────────┐
│ 63        ...         8  │  7  │  6  │  5  │  4  │ 3-0    │
│       未使用             │  Interrupt  │  Exception Code  │
└────────────────────────────────────────────────────────────┘
```

**常见的异常代码**：

| 异常代码 | 含义 |
|---------|-----|
| 2 | 未定义操作码 |
| 12 | 硬件故障 |
| 8 | 环境调用（系统调用） |

---

### （三）向量中断


**核心思想**：不同的异常/中断跳转到不同的处理程序地址。

**向量地址计算**：

```
处理程序地址 = 向量表基址 + 异常向量地址
```

**例子**：

| 异常类型 | 向量地址（二进制） |
|---------|------------------|
| 未定义操作码 | 00 0100 0000 |
| 硬件故障 | 01 1000 0000 |
| ... | ... |

---

#### 为什么能"精准跳到"对应的异常处理程序？

**核心答案：这是硬件设计者人为规定好的映射关系，硬件和软件遵循同一个规范。**

##### 1. 向量表的本质

向量表就是一张**预先约定好的"异常-地址"映射表**：

```
向量表基址 = 0x00000000（或由 mtvec/stvec 寄存器指定）

偏移量      异常类型              该位置存放的内容
─────────────────────────────────────────────────────
0x00       Reset               → 跳转到复位处理程序的指令
0x04       Undefined           → 跳转到未定义指令处理程序的指令
0x08       SVC/系统调用        → 跳转到系统调用处理程序的指令
0x0C       Prefetch Abort      → 跳转到预取指中止处理程序的指令
0x10       Data Abort          → 跳转到数据中止处理程序的指令
0x14       Reserved            → （保留）
0x18       IRQ                 → 跳转到IRQ中断处理程序的指令
0x1C       FIQ                 → 跳转到FIQ快速中断处理程序的指令
```

##### 2. 向量表里实际存的是什么？

**有两种设计方式：**

| 方式 | 向量表存放的内容 | 典型架构 |
|-----|----------------|---------|
| **方式A** | 跳转指令（如 `LDR PC, =Handler`） | ARM |
| **方式B** | 处理程序的入口地址（直接地址值） | RISC-V |

**方式A（ARM）的向量表内容示例**：

```assembly
; 向量表内容（内存地址 0x00 开始）
0x00:  LDR PC, =Reset_Handler      ; 跳转到复位处理程序
0x04:  LDR PC, =Undefined_Handler  ; 跳转到未定义指令处理程序
0x08:  LDR PC, =SVC_Handler        ; 跳转到软中断处理程序
0x0C:  LDR PC, =Prefetch_Handler   ; 跳转到预取指中止处理程序
0x10:  LDR PC, =Abort_Handler      ; 跳转到数据中止处理程序
0x14:  NOP                         ; 保留
0x18:  LDR PC, =IRQ_Handler        ; 跳转到IRQ处理程序
0x1C:  LDR PC, =FIQ_Handler        ; 跳转到FIQ处理程序
```

**方式B（RISC-V）的向量表内容示例**：

```
mtvec（异常向量基址寄存器）= 0x80000000

偏移量      内容（处理程序入口地址）
─────────────────────────────────────
0x00       0x80001234（异常处理程序入口）
0x04       0x80005678（中断处理程序入口）
...
```

##### 3. 完整的跳转流程示例

假设发生了 **IRQ 中断**，使用方式A（ARM）：

```
步骤1: 硬件检测到 IRQ 信号
步骤2: 硬件自动计算: 地址 = 0x00000000 + 0x18 = 0x00000018
步骤3: 硬件跳转到 0x00000018 执行
步骤4: 该地址存放的是: LDR PC, =IRQ_Handler
步骤5: 执行这条指令，PC 被赋值为 IRQ_Handler 的真实地址
步骤6: CPU 开始执行 IRQ_Handler 的代码
```

假设发生了 **IRQ 中断**，使用方式B（RISC-V）：

```
步骤1: 硬件检测到 IRQ 信号
步骤2: 硬件自动计算: 地址 = mtvec + 0x04
步骤3: 硬件从该地址读取处理程序入口地址（如 0x80005678）
步骤4: PC = 0x80005678
步骤5: CPU 开始执行中断处理程序
```

##### 4. 类比理解

想象一个**医院分诊台**：

```
你肚子疼（异常类型）
→ 护士查表：肚子疼 → 3楼消化科（偏移量 → 科室）
→ 你按地址去 3楼（跳转）

为什么知道是消化科？
→ 因为医院规定好了"肚子疼 → 消化科"这个映射！
```

向量表就是这个"分诊规则表"，硬件是护士，自动帮你查表跳转。

##### 5. 谁来保证正确性？

| 角色 | 职责 |
|-----|------|
| **硬件设计者** | 规定每种异常对应的偏移量（如 IRQ = 0x18） |
| **编译器/操作系统** | 在向量表对应位置填入正确的跳转指令/地址 |
| **CPU** | 异常发生时自动计算地址并跳转 |

**一句话总结**：偏移量是硬件设计时规定好的，向量表内容由软件初始化，两边对上就行了。

---

**优点**：

| 方式 | 特点 |
|-----|------|
| 非向量中断 | 所有异常跳转到同一个地址，再判断原因 |
| 向量中断 | 直接跳转到对应的处理程序，更快 |

**指令的两种选择**：
1. 直接处理中断
2. 跳转到真正的处理程序

### （五）流水线中的异常

![[幻灯片91.png]]

**核心观点**：异常是另一种形式的控制冒险。

**示例场景**：

```assembly
add x1, x2, x1    # 在EX阶段发生硬件故障
```

**处理步骤**：

| 步骤 | 操作 |
|-----|------|
| **1. 防止x1被破坏** | 不能让错误结果写入寄存器 |
| **2. 完成前序指令** | 让流水线中更早的指令正常完成 |
| **3. 冲刷add及后续指令** | 清空流水线 |
| **4. 设置SEPC和SCAUSE** | 保存异常信息 |
| **5. 转移控制权** | 跳转到异常处理程序 |

**与分支预测错误的相似性**：

| 特点 | 分支预测错误 | 异常 |
|-----|------------|-----|
| 需要冲刷流水线 | ✓ | ✓ |
| 需要保存PC | ✓ | ✓（SEPC） |
| 需要跳转到新地址 | ✓ | ✓（处理程序） |
| 可复用硬件 | - | ✓ |

**硬件复用**：

异常处理可以使用分支预测错误的大部分硬件：
- 冲刷逻辑
- PC更新逻辑
- 控制信号生成

### （六）带异常处理的流水线架构
![[Pasted image 20260415101416.png]]
**新增的硬件组件**：

| 组件 | 功能 |
|-----|------|
| **SCAUSE** | 保存异常原因 |
| **SEPC** | 保存异常指令地址 |
| **IF.Flush** | 冲刷取指阶段 |
| **ID.Flush** | 冲刷译码阶段 |
| **EX.Flush** | 冲刷执行阶段 |

**与普通流水线的区别**：

| 特性 | 普通流水线 | 带异常处理的流水线 |
|-----|-----------|------------------|
| Flush信号 | 只有分支预测错误 | 分支错误 + 异常 |
| PC来源 | 顺序/分支目标 | + 异常处理程序地址 |
| 寄存器 | 无 | SCAUSE、SEPC |

### （七）可恢复异常

**可恢复异常的定义**：

异常处理后可以返回原程序继续执行。

**处理流程**：

```
1. 流水线冲刷该指令
2. 处理程序执行
3. 返回该指令重新执行
```

**关键点**：PC保存在SEPC寄存器中，用于标识引发异常的指令。

### （八）异常处理示例

![[幻灯片93.png]]

**问题代码**：

```assembly
40: sub x11, x2, x4
44: and x12, x2, x5
48: or  x13, x2, x6
4c: add x1, x2, x1    ← 在这条指令发生异常！
50: sub x15, x6, x7
54: ld  x16, 100(x7)
```

**异常处理程序地址**：`0x1C090000`

```assembly
1c090000: sd x26, 1000(x10)
1c090004: sd x27, 1008(x10)
...
```

**时钟周期分析**：

| 时钟 | 处理内容 |
|-----|---------|
| CC6 | 检测到异常，设置SEPC=0x4c，SCAUSE=异常原因 |
| CC7 | 冲刷流水线，跳转到0x1C090000 |

---

#### Clock 7 时刻的流水线状态详解
![[Pasted image 20260415101800.png]]
**上图的流水线状态**：

```
Clock 7 时刻：
┌────────┬────────┬────────┬────────┬────────┐
│   IF   │   ID   │   EX   │  MEM   │   WB   │
├────────┼────────┼────────┼────────┼────────┤
│sd x26, │ bubble │ bubble │ bubble │ or x13 │
│1000(x0)│ (nop)  │ (nop)  │ (nop)  │  ...   │
└────────┴────────┴────────┴────────┴────────┘
    ↑                              ↑
    │                              └─ 异常指令之后的指令被冲刷
    └─ 异常处理程序的第一条指令
```

**关键硬件操作**：

| 操作 | 信号 | 效果 |
|-----|------|------|
| **冲刷IF阶段** | `IF.Flush = 1` | 清空IF/ID寄存器，防止错误指令进入 |
| **冲刷ID阶段** | `ID.Flush = 1` | 清空ID/EX寄存器，用nop替代 |
| **冲刷EX阶段** | `EX.Flush = 1` | 清空EX/MEM寄存器，用nop替代 |
| **PC更新** | PC = 0x1C090000 | 跳转到异常处理程序入口 |

**蓝色控制线的作用**：

```
                    ┌─────────────────────────────────────────────┐
                    │          Hazard Detection Unit              │
                    │         （冒险检测单元/异常检测）              │
                    └─────────────────────────────────────────────┘
                           │          │          │
                           ↓          ↓          ↓
                      IF.Flush   ID.Flush   EX.Flush
                           │          │          │
                           ↓          ↓          ↓
                      ┌────────┐ ┌────────┐ ┌────────┐
                      │ IF/ID  │ │ ID/EX  │ │ EX/MEM │
                      │ 清空   │ │ 清空   │ │ 清空   │
                      └────────┘ └────────┘ └────────┘
```

**异常处理程序执行**：

```assembly
# 异常处理程序从 0x1C090000 开始
1c090000: sd x26, 1000(x10)    # 保存寄存器 x26
1c090004: sd x27, 1008(x10)    # 保存寄存器 x27
1c090008: ...                  # 其他处理
```

**为什么有3个 bubble？**

```
异常发生在 CC6 的 MEM 阶段（add x1, x2, x1）

CC6 时流水线：
  IF: sub x15  ← 还没执行
  ID: ld x16   ← 还没执行
  EX: (空)
  MEM: add x1  ← 异常在这里检测到！
  WB: or x13   ← 前序指令，正常完成

CC7 时（冲刷后）：
  IF: sd x26   ← 异常处理程序第一条指令
  ID: bubble   ← 原来的 sub x15 被冲刷
  EX: bubble   ← 原来的 ld x16 被冲刷
  MEM: bubble  ← 原来的空被填充
  WB: or x13   ← 前序指令已完成
```

**与分支预测错误的对比**：

| 特性 | 分支预测错误 | 异常处理 |
|-----|------------|---------|
| 触发条件 | 分支方向预测错误 | 硬件故障/非法操作 |
| 冲刷范围 | 预测错误路径的指令 | 异常指令及后续所有指令 |
| PC更新 | 分支目标地址 | 异常处理程序地址 |
| 额外操作 | 无 | 设置 SEPC、SCAUSE |
| 可恢复性 | 不可恢复（预测错误） | 可能可恢复（如缺页异常） |

**一句话总结**：异常发生时，硬件自动冲刷流水线中异常指令之后的所有指令，插入 bubble，并跳转到异常处理程序执行

---

### （九）多重异常

**问题**：流水线重叠执行多条指令，可能同时发生多个异常。

**简单处理方法**：处理最早指令的异常。

```
流水线中的指令：
  IF    ID    EX    MEM   WB
  └───┴───┴───┴───┴───┘
        指令A    指令B    指令C
                 ↑异常1   ↑异常2

处理顺序：先处理指令B的异常，后续指令冲刷
```

**复杂流水线的挑战**：

| 情况 | 问题 |
|-----|------|
| 多发射 | 每周期发射多条指令 |
| 乱序完成 | 指令完成顺序不确定 |
| 精确异常 | 难以维护 |

---

### （十）不精确异常

**定义**：异常发生时，不保证之前的指令都已完成，之后的指令都未开始。

**处理方法**：

```
1. 立即停止流水线，保存状态
2. 让处理程序判断：
   - 哪些指令发生了异常
   - 哪些指令应该完成或冲刷
3. 可能需要"手动"完成某些指令
```

**优缺点**：

| 方面 | 说明 |
|-----|------|
| 硬件 | 简化 |
| 软件 | 处理程序更复杂 |
| 适用 | 复杂多发射乱序流水线 |

## 十四、指令级并行（ILP）

### （一）什么是ILP

**定义**：同时执行多条指令的能力。

**提升ILP的方法**：

| 方法 | 原理 | 效果 |
|-----|------|-----|
| **更深流水线** | 每阶段工作量减少 | 时钟周期更短 |
| **多发射** | 复制流水线阶段 | 每周期多条指令 |

**多发射的效果**：

```
CPI < 1，使用IPC（每周期指令数）衡量

例子：4GHz 4路多发射
- 峰值IPC = 4
- 峰值性能 = 16 BIPS（每秒十亿条指令）
- 峰值CPI = 0.25
```

---

### （二）静态多发射 vs 动态多发射

| 特性 | 静态多发射 | 动态多发射 |
|-----|-----------|-----------|
| **谁来决定** | 编译器 | CPU硬件 |
| **发射包** | 编译器打包指令 | CPU动态选择 |
| **冒险检测** | 编译器避免 | CPU运行时解决 |
| **复杂度** | 编译器复杂 | 硬件复杂 |

**静态多发射（VLIW）**：

```
编译器将可以并行执行的指令打包成一个"发射包"
发射包 = 一条超长指令，指定多个并行操作
```

**动态多发射（超标量）**：

```
CPU检查指令流，每周期决定发射0、1、2...条指令
自动避免结构和数据冒险
```

---

### （三）推测执行

**核心思想**：猜测指令执行结果，提前开始执行。

**推测的类型**：

| 推测类型 | 猜测内容 | 回滚条件 |
|---------|---------|---------|
| **分支推测** | 分支是否跳转 | 猜错了回滚 |
| **Load推测** | Load的地址或值 | 地址被更新 |

**工作流程**：

```
1. 猜测要做什么
2. 尽早开始操作
3. 检查猜测是否正确
   - 正确：完成操作
   - 错误：回滚，做正确的事
```

---

### （四）编译器 vs 硬件推测

| 方式 | 编译器推测 | 硬件推测 |
|-----|-----------|---------|
| **方法** | 重排指令 | 缓存结果 |
| **例子** | 把Load移到分支前 | 缓存执行结果 |
| **恢复** | 插入"修复"指令 | 错误时冲刷缓冲 |

**推测与异常的问题**：

推测执行的指令发生异常怎么办？

| 方法 | 处理 |
|-----|------|
| **静态推测** | 添加ISA支持，延迟异常 |
| **动态推测** | 缓存异常，直到确认指令需要执行 |

---

## 十五、静态多发射（VLIW）

### （一）VLIW原理

**核心概念**：

```
Issue Packet（发射包）= 一组可以同时发射的指令

编译器负责：
1. 将指令分组打包
2. 检测并避免冒险
3. 必要时填充nop
```

**VLIW = Very Long Instruction Word（超长指令字）**：

一条"指令"实际上包含多个并行操作。

---

### （二）编译器调度

**编译器的任务**：

| 任务 | 说明 |
|-----|------|
| **重排指令** | 将指令重新排列到发射包中 |
| **消除包内依赖** | 同一包内的指令不能有依赖 |
| **处理包间依赖** | 不同ISA处理方式不同 |
| **填充nop** | 无法填满时用nop占位 |

---

### （三）RISC-V静态双发射

![[幻灯片105.png]]

**发射包格式**：

| 发射槽 | 指令类型 |
|-------|---------|
| **槽1** | ALU/分支指令 |
| **槽2** | Load/Store指令 |

**约束**：
- 64位对齐
- ALU/分支在前，Load/Store在后
- 未使用的槽填充nop

**示例调度**：

| Address | ALU/branch | Load/store | cycle |
|---------|------------|------------|-------|
| | nop | ld x31, 0(x20) | 1 |
| | addi x20, x20, -8 | nop | 2 |
| | add x31, x31, x21 | nop | 3 |
| | blt x22, x20, Loop | sd x31, 8(x20) | 4 |

**性能**：IPC = 5/4 = 1.25（峰值IPC = 2）

---

### （四）双发射中的冒险

**核心问题**：一个周期两条指令 → 同一时刻竞争资源 + 依赖关系翻倍

---

#### 1. 包内冒险

**问题**：同一个发射包里的两条指令有依赖

```
发射包：
┌─────────────────┬─────────────────┐
│  add x1, x2, x3 │  sub x4, x1, x5 │  ← x1 有依赖！
└─────────────────┴─────────────────┘
        槽1                槽2
```

**解决**：编译器保证包内无依赖，或者拆成两个包

---

#### 2. 包间冒险

**问题**：当前包依赖上一个包的结果

```
周期1：add x1, x2, x3  │  nop
周期2：sub x4, x1, x5  │  ld x6, 0(x4)  ← x1、x4 都依赖上周期
```

**解决**：转发机制仍然有效，但需要更复杂的检测逻辑

---

#### 3. Load-Use 冒险更严重

**问题**：Load 指令要到 MEM 阶段才能读到数据

```
周期1：nop           │  ld x1, 0(x10)
周期2：add x2, x1, x3 │  nop           ← x1 还没读到！

必须拆成：
周期1：nop           │  ld x1, 0(x10)
周期2：nop           │  nop           ← 强制等一周期
周期3：add x2, x1, x3 │  ...
```

**代价**：双发射下 Load-Use 冒险浪费的是**2个发射槽**（相当于浪费2条指令的吞吐量）

---

#### 4. 双发射冒险总结

| 冒险类型 | 原因 | 解决方案 | 代价 |
|---------|------|---------|------|
| **包内数据冒险** | 同包指令有依赖 | 编译器调度避免 | 无（编译期解决） |
| **包间数据冒险** | 跨包依赖 | 转发机制 | 无（硬件解决） |
| **Load-Use冒险** | 数据还没读到 | 插入气泡 | **浪费2个槽** |
| **结构冒险** | 两个ALU冲突 | 复制硬件资源 | 硬件成本 |

---

#### 5. RISC-V 双发射的设计约束

```
┌─────────────┬─────────────┐
│   槽1       │    槽2      │
│ ALU/分支    │ Load/Store  │
└─────────────┴─────────────┘

约束：
- ALU指令和访存指令才能并行
- 两条ALU指令 → 不行
- 两条访存指令 → 不行
```

**这样设计的原因**：避免资源冲突（一个ALU + 一个访存单元）

**一句话总结**：双发射 = 冒险翻倍 + 浪费翻倍 + 编译器要更聪明

---

### （五）循环展开

**目的**：暴露更多并行性。

**方法**：复制循环体，使用不同的寄存器。

**好处**：

| 好处 | 说明 |
|-----|------|
| 减少循环控制开销 | 分支次数减少 |
| 寄存器重命名 | 避免反依赖 |
| 更多调度空间 | 编译器有更多选择 |

**反依赖（Anti-dependency）**：

```
Store后跟Load同一寄存器 → 名称依赖
通过寄存器重命名消除
```

**循环展开示例**：

| cycle | ALU/branch | Load/store |
|-------|------------|------------|
| 1 | addi x20,x20,-32 | ld x28, 0(x20) |
| 2 | nop | ld x29, 24(x20) |
| 3 | add x28,x28,x21 | ld x30, 16(x20) |
| 4 | add x29,x29,x21 | ld x31, 8(x20) |
| 5 | add x30,x30,x21 | sd x28, 32(x20) |
| 6 | add x31,x31,x21 | sd x29, 24(x20) |
| 7 | nop | sd x30, 16(x20) |

#### 深度剖析：保留站与重排序缓冲的工作原理

##### 1. 为什么需要保留站和ROB？

**问题场景**：

```assembly
ld x1, 0(x10)      # Load，需要等待（可能Cache Miss）
add x2, x1, x3     # 依赖x1
sub x4, x5, x6     # 无依赖
and x7, x8, x9     # 无依赖
```

**顺序执行的问题**：
- `ld` 等待内存时，`sub` 和 `and` 也被阻塞
- 即使它们没有依赖，也不能提前执行

**乱序执行的挑战**：
- 如何让 `sub` 和 `and` 提前执行？
- 如何保证结果按顺序写回？
- 如果 `ld` 发生异常怎么办？

**解决方案**：
- **保留站**：暂存指令，等待操作数就绪
- **ROB**：缓存执行结果，保证按序提交

##### 2. 保留站（Reservation Station）详解

**结构**：

```
┌──────────────────────────────────────────────────────────────┐
│                    保留站（多个条目）                          │
├────────┬────────┬────────┬────────┬────────┬────────────────┤
│ Entry  │ Op     │ Vj     │ Vk     │ Qj     │ Qk     │ Busy  │
├────────┼────────┼────────┼────────┼────────┼────────┼────────┤
│ 0      │ add    │ x1的值  │ 10     │ -      │ -      │ 0     │
│ 1      │ sub    │ x5的值  │ x6的值  │ -      │ -      │ 1     │
│ 2      │ ld     │ x10的值 │ 0      │ -      │ -      │ 1     │
│ ...    │ ...    │ ...    │ ...    │ ...    │ ...    │ ...   │
└────────┴────────┴────────┴────────┴────────┴────────┴────────┘
```

**字段说明**：

| 字段 | 含义 |
|-----|------|
| **Entry** | 保留站条目编号 |
| **Op** | 操作类型（add, sub, ld等） |
| **Vj, Vk** | 已就绪的操作数值 |
| **Qj, Qk** | 操作数来源（哪个ROB条目会提供） |
| **Busy** | 该条目是否被占用 |

**工作流程**：

```
1. 指令发射到保留站
   - 如果操作数就绪（寄存器中有值）：存入Vj/Vk
   - 如果操作数未就绪：记录来源到Qj/Qk

2. 等待操作数
   - 当ROB写入结果时，广播ROB编号
   - 保留站检查Qj/Qk是否匹配
   - 匹配则将结果存入Vj/Vk，清除Qj/Qk

3. 执行
   - 所有操作数就绪（Qj=Qk=空）
   - 发送到功能单元执行
```

**例子**：

```assembly
div x1, x2, x3     # x1 = x2/x3，需要10周期
add x4, x1, x5     # 依赖x1
sub x6, x7, x8     # 无依赖
```

**保留站状态变化**：

```
周期1（发射）：
┌────────┬────────┬────────┬────────┬────────┬────────┬────────┐
│ Entry  │ Op     │ Vj     │ Vk     │ Qj     │ Qk     │ Busy   │
├────────┼────────┼────────┼────────┼────────┼────────┼────────┤
│ 0      │ div    │ x2     │ x3     │ -      │ -      │ 1      │ ← 立即执行
│ 1      │ add    │ x5     │ -      │ ROB#0  │ -      │ 1      │ ← 等待x1
│ 2      │ sub    │ x7     │ x8     │ -      │ -      │ 1      │ ← 立即执行
└────────┴────────┴────────┴────────┴────────┴────────┴────────┘

周期2-11：
- div 在执行
- sub 执行完成，结果写入ROB#2
- add 仍在等待

周期11（div完成）：
- div结果写入ROB#0，广播ROB#0
- add的Qj=ROB#0匹配，从广播总线获取x1的值
- add 的 Vj = x1（div的结果），Qj 清空
- add 准备执行
```

##### 3. 重排序缓冲（ROB）详解

**结构**：

```
┌──────────────────────────────────────────────────────────────┐
│                    重排序缓冲（环形队列）                       │
├────────┬────────┬────────┬────────┬────────┬────────────────┤
│ Entry  │ Inst   │ Dest   │ Value  │ Ready  │ Exception      │
├────────┼────────┼────────┼────────┼────────┼────────────────┤
│ 0      │ div    │ x1     │ 42     │ 1      │ 0              │
│ 1      │ add    │ x4     │ 100    │ 1      │ 0              │
│ 2      │ sub    │ x6     │ -5     │ 1      │ 0              │
│ ...    │ ...    │ ...    │ ...    │ ...    │ ...            │
└────────┴────────┴────────┴────────┴────────┴────────┴────────┘
              ↑
           头指针（下一个要提交的）
```

**字段说明**：

| 字段 | 含义 |
|-----|------|
| **Entry** | ROB条目编号 |
| **Inst** | 指令类型 |
| **Dest** | 目标寄存器编号 |
| **Value** | 执行结果 |
| **Ready** | 结果是否就绪 |
| **Exception** | 是否发生异常 |

**工作流程**：

```
1. 指令发射时
   - 在ROB尾部分配一个条目
   - 记录指令信息
   - 尾指针前进

2. 执行完成时
   - 结果写入ROB对应条目
   - 设置Ready=1
   - 广播结果给等待的保留站

3. 提交时
   - 检查头部条目是否Ready
   - 如果Ready：写入寄存器堆，头指针前进
   - 如果Exception：触发异常处理
```

**为什么ROB保证顺序？**

```
发射顺序：ld → add → sub
执行顺序：sub → add → ld（乱序）
ROB顺序： ld → add → sub（保持发射顺序）

提交顺序：
- 只有头部的指令可以提交
- 即使sub先执行完，也要等ld和add先提交
- 保证程序员视角的"顺序执行"语义
```

##### 4. 保留站 + ROB 完整流程示例

```assembly
ld x1, 0(x10)      # 指令1
add x2, x1, x3     # 指令2，依赖x1
sub x4, x5, x6     # 指令3，无依赖
and x7, x8, x9     # 指令4，无依赖
```

**详细时间线**：

```
周期1：发射指令1
  ROB: [ld x1, -, -, 0, 0]
  保留站: [ld, x10, 0, -, -, 1]
  状态：ld开始执行

周期2：发射指令2
  ROB: [ld x1, -, -, 0, 0], [add x2, -, -, 0, 0]
  保留站: [ld, ...], [add, x3, -, ROB#0, -, 1]  ← 等待x1
  状态：ld在执行，add等待

周期3：发射指令3
  ROB: [...], [...], [sub x4, -, -, 0, 0]
  保留站: [...], [...], [sub, x5, x6, -, -, 1]  ← 立即可执行
  状态：ld在执行，add等待，sub执行

周期4：发射指令4
  ROB: [...], [...], [...], [and x7, -, -, 0, 0]
  保留站: [...], [...], [...], [and, x8, x9, -, -, 1]
  状态：ld在执行，add等待，sub完成，and执行

周期5：sub完成，写入ROB#2
  ROB: [ld, -, -, 0, 0], [add, -, -, 0, 0], [sub, x4, -5, 1, 0], ...
  状态：sub结果就绪，但还不能提交（前面有未完成的）

...（假设ld需要100周期）

周期100：ld完成，写入ROB#0
  ROB: [ld, x1, 42, 1, 0], [add, -, -, 0, 0], ...
  广播：ROB#0结果=42

周期101：add获取x1，执行
  保留站: [add, x3, 42, -, -, 1]  ← x1就绪

周期102：add完成，写入ROB#1
  ROB: [ld, x1, 42, 1, 0], [add, x2, 100, 1, 0], ...

周期103：提交阶段
  - 检查ROB#0：Ready=1，提交！写入x1=42
  - 检查ROB#1：Ready=1，提交！写入x2=100
  - 检查ROB#2：Ready=1，提交！写入x4=-5
  - 检查ROB#3：Ready=1，提交！写入x7=...
```

##### 5. 异常处理的精确性

**问题**：乱序执行时，如何保证异常的正确处理？

```assembly
ld x1, 0(x10)      # 可能发生异常（非法地址）
add x2, x1, x3     # 依赖x1
div x4, x5, x0     # 除零异常！
```

**时序**：
- `div` 先执行（没有依赖）
- `ld` 后执行（需要访存）
- 但 `ld` 在程序顺序上更早

**不精确异常的问题**：
- 如果 `div` 先触发异常，处理程序会认为 `div` 是第一条出错的指令
- 但实际上 `ld` 可能也会出错，且 `ld` 更早

**ROB 保证精确异常**：

```
执行顺序：div → ld → add
异常检测：div 先检测到除零错误

ROB 中的处理：
1. div 的异常标记写入ROB
2. 不立即触发异常
3. 等待 div 成为 ROB 头部
4. 提交时检查到异常，触发异常处理
5. 此时所有之前的指令都已提交
```

**关键**：异常只在**提交阶段**触发，保证精确性。

##### 6. 预约站 vs ROB 的分工

| 组件 | 职责 | 数据流向 |
|-----|------|---------|
| **预约站** | 缓冲指令，等待操作数 | 寄存器/ROB → 预约站 |
| **ROB** | 缓存结果，保证顺序提交 | 功能单元 → ROB → 寄存器 |

**类比**：
- 预约站 = 餐厅的"等位区"，顾客（指令）等待食材（操作数）
- ROB = 餐厅的"取餐区"，菜品（结果）按顺序给顾客

##### 7. 硬件开销

| 组件 | 数量 | 面积占比 | 功耗占比 |
|-----|------|---------|---------|
| 预约站 | 16-64条目 | ~5% | ~8% |
| ROB | 64-256条目 | ~10% | ~15% |
| 寄存器重命名表 | 64-128条目 | ~3% | ~5% |

**复杂度**：乱序执行的硬件开销是顺序执行的 **3-5倍**。

##### 8. 性能收益

| 场景 | 顺序执行CPI | 乱序执行CPI | 提升 |
|-----|-----------|-----------|------|
| 无依赖 | 1.0 | 1.0 | 无 |
| Load后跟依赖指令 | 1.0 + Miss惩罚 | ~1.0（掩盖延迟） | 显著 |
| 多个独立操作 | 串行执行 | 并行执行 | 显著 |

**一句话总结**：保留站让指令"等得起"，ROB让结果"排好队"，两者配合实现乱序执行、顺序提交。

#### 深度剖析：Cache Miss 对流水线的毁灭性影响

##### 1. 为什么 Cache Miss 是"性能杀手"？

**时间尺度对比**：

| 操作 | 延迟（周期数） | 类比 |
|-----|-------------|------|
| ALU运算 | 1 周期 | 1秒 |
| L1 Cache命中 | 3-4 周期 | 3-4秒 |
| L2 Cache命中 | 10-12 周期 | 10-12秒 |
| L3 Cache命中 | 30-40 周期 | 半分钟 |
| **内存访问（Cache Miss）** | **100-300 周期** | **2-5分钟** |

**Cache Miss 的本质**：CPU在等待内存时**什么都不做**。

##### 2. Cache Miss 对流水线的连锁反应

**场景1：Load指令发生 Cache Miss**

```assembly
ld x1, 0(x10)      # Cache Miss！需要200周期
add x2, x1, x3     # 依赖x1，必须等待
sub x4, x5, x6     # 无依赖，但...
and x7, x8, x9     # 无依赖，但...
```

**问题**：
- `ld` 发生 Cache Miss，整个流水线**停顿200周期**
- 后续指令即使无依赖，也会被**预约站塞满**
- 预约站满了之后，**无法发射新指令**

**时间线**：
```
周期1-3:   ld 在流水线中前进
周期4:     Cache Miss 检测到
周期5-204: ld 卡在 MEM 阶段等待内存
           预约站逐渐填满（假设8个槽位）
周期12:    预约站满了，无法取新指令
周期204:   内存数据返回
           流水线开始恢复
```

**连锁反应**：
```
Load指令Miss
    ↓
预约站被后续指令填满
    ↓
无法发射新指令
    ↓
取指单元被迫停顿
    ↓
整个流水线空转
```

##### 3. 多发射下 Cache Miss 的代价翻倍

**单发射 vs 双发射的对比**：

| 场景 | 单发射 | 双发射 |
|-----|-------|-------|
| 1次 Cache Miss | 浪费 200 条指令的吞吐量 | 浪费 **400** 条指令的吞吐量 |
| 预约站大小 | 8条指令 | 需要16条指令 |
| 恢复时间 | 较短 | 更长（更多指令在等待） |

**例子**：

```assembly
# 双发射流水线
周期1:  nop          │  ld x1, 0(x10)    # L1 Miss → L2 Miss → Memory Miss
周期2:  add x2,x1,x3 │  ld x4, 4(x10)    # 依赖x1，且第二个Load也在等待
周期3:  sub x5,x6,x7 │  ld x8, 8(x10)    # 三个Load同时Miss！
...
```

**问题**：多个 Cache Miss 同时发生 → **内存带宽饱和** → 所有Load都变慢。

##### 4. Cache Miss 对乱序执行的影响

**预约站的作用**：

| 正常情况 | Cache Miss 情况 |
|---------|----------------|
| 等待操作数时，其他指令可以先执行 | 操作数迟迟不到，预约站被占满 |
| 乱序执行掩盖延迟 | 预约站满了，无法接受新指令 |

**问题本质**：

```
预约站容量 = 有限的"缓冲空间"
Cache Miss = 需要200周期 = 需要缓冲200周期的指令
如果预约站容量 < 200周期内能发射的指令数 → 流水线停顿
```

**计算示例**：

```
假设：
- 预约站容量 = 16 条指令
- 发射宽度 = 4 条/周期
- Cache Miss 延迟 = 200 周期

问题：
预约站能在 16/4 = 4 周期内填满
但 Cache Miss 需要等 200 周期
差距 = 196 周期 的流水线停顿！
```

##### 5. 内存墙（Memory Wall）问题

**定义**：CPU速度增长 >> 内存速度增长 → CPU大部分时间在等内存。

**历史趋势**：

| 年份 | CPU频率 | 内存延迟（周期） |
|-----|--------|----------------|
| 1980 | 5 MHz | ~10 周期 |
| 1990 | 50 MHz | ~20 周期 |
| 2000 | 1 GHz | ~100 周期 |
| 2020 | 4 GHz | ~200-300 周期 |

**核心矛盾**：
- CPU每年性能提升 ~50%
- 内存延迟每年只改善 ~7%
- 差距越来越大

**对流水线的影响**：

```
流水线深度增加 → 分支惩罚增加
发射宽度增加   → Cache Miss 代价翻倍
CPU速度提升   → 内存延迟（周期数）增加
```

##### 6. 解决 Cache Miss 影响的技术

**技术1：非阻塞Cache（Non-blocking Cache）**

```
传统Cache：Miss时整个Cache停顿
非阻塞Cache：Miss时继续服务其他请求
```

| 类型 | 说明 |
|-----|------|
| **Hit under Miss** | Miss时仍可处理其他Hit |
| **Miss under Miss** | 可以同时处理多个Miss |

**效果**：
- 多个Load可以并发执行
- 避免一个Miss阻塞整个Cache

**技术2：预取（Prefetching）**

```
硬件预取：CPU检测访问模式，提前加载数据
软件预取：程序员插入 prefetch 指令
```

```assembly
# 软件预取示例
prefetch 64(x10)    # 提前预取下一个缓存行
ld x1, 0(x10)       # 当前数据（应该命中）
```

**技术3：增大预约站和指令窗口**

| 参数 | 作用 | 代价 |
|-----|------|------|
| 预约站大小 | 缓冲更多等待的指令 | 面积、功耗增加 |
| 重排序缓冲大小 | 支持更多in-flight指令 | 复杂度增加 |
| 加载/存储队列 | 允许更多内存操作并发 | 硬件开销 |

**技术4：多线程（SMT/Hyper-Threading）**

```
一个线程Cache Miss时，切换到另一个线程执行
利用等待时间做有用的工作
```

| 技术 | 说明 |
|-----|------|
| **SMT** | 同时多线程，一个核心同时执行多个线程 |
| **超线程** | Intel的SMT实现，一个物理核心模拟两个逻辑核心 |

**效果**：
- 一个线程等内存时，另一个线程用CPU
- 提高CPU利用率

##### 7. Cache Miss 的量化分析

**CPI分解公式**：

$$CPI_{实际} = CPI_{理想} + CPI_{停顿}$$

其中：
$$CPI_{停顿} = CPI_{数据冒险} + CPI_{控制冒险} + CPI_{内存停顿}$$

**内存停顿的计算**：

$$CPI_{内存} = \text{Miss次数/指令} \times \text{Miss惩罚}$$

**例子**：

```
假设：
- L1 Miss率 = 5%
- L1 Miss惩罚 = 10 周期（访问L2）
- L2 Miss率 = 20%（相对于L1 Miss）
- L2 Miss惩罚 = 200 周期（访问内存）

计算：
CPI_L1 = 0.05 × 10 = 0.5
CPI_L2 = 0.05 × 0.20 × 200 = 2.0
CPI_内存 = 0.5 + 2.0 = 2.5

如果理想CPI = 0.5，则实际CPI = 3.0
内存停顿占总CPI的 83%！
```

##### 8. 不同程序的 Cache Miss 特性

| 程序类型 | Cache Miss率 | CPI影响 | 原因 |
|---------|-------------|---------|------|
| 科学计算 | 低（1-2%） | 小 | 规则访问模式，预取有效 |
| 数据库 | 高（10-20%） | 大 | 指针追踪，随机访问 |
| 图算法 | 很高（>20%） | 很大 | 不规则访问，无法预取 |
| 编译器 | 中等（5-10%） | 中等 | 混合访问模式 |

**mcf程序为什么CPI=8.56？**

```
mcf = 网络流优化程序
- 大量指针追踪
- 极高的Cache Miss率
- 内存带宽成为瓶颈
- 流水线大部分时间在等内存
```

##### 9. 编译器优化对 Cache Miss 的帮助

**循环分块（Loop Tiling）**：

```c
// 原始代码（Cache不友好）
for (i = 0; i < N; i++)
    for (j = 0; j < N; j++)
        A[i][j] = B[i][j] + C[i][j];

// 分块后（Cache友好）
for (ii = 0; ii < N; ii += BLOCK)
    for (jj = 0; jj < N; jj += BLOCK)
        for (i = ii; i < ii+BLOCK; i++)
            for (j = jj; j < jj+BLOCK; j++)
                A[i][j] = B[i][j] + C[i][j];
```

**效果**：
- 数据在Cache中复用
- Miss率显著降低

**数据布局优化**：

```c
// AoS（Array of Structures）
struct Point { float x, y, z; };
Point points[N];  // 访问x时，y、z占用Cache

// SoA（Structure of Arrays）
float x[N], y[N], z[N];  // 只加载需要的数据
```

##### 10. Cache Miss 与流水线设计的权衡

| 设计选择 | 对Cache Miss的影响 | 代价 |
|---------|-------------------|------|
| 增大Cache | 降低Miss率 | 面积、功耗 |
| 增加Cache层级 | 降低Miss惩罚 | 复杂度 |
| 增大预约站 | 容忍更长延迟 | 面积、功耗 |
| SMT | 提高利用率 | 单线程性能下降 |

**一句话总结**：Cache Miss 是现代CPU性能的最大瓶颈，流水线再深、发射再宽，都要等内存。

### 常见面试题

**Q1: 为什么Load-Use冒险不能只用转发解决？**

A: 因为Load指令要到MEM阶段才能从内存读到数据，而需要使用该数据的指令在EX阶段就需要它。即使转发，也需要"未来"的数据，物理上不可能。

**Q2: 2位预测器为什么比1位预测器好？**

A: 对于嵌套循环，1位预测器每次循环结束和下次循环开始都会预测错误。2位预测器需要连续两次错误才改变预测，避免了这个问题。

**Q3: 异常处理和分支预测错误有什么共同点？**

A: 都需要：
1. 冲刷流水线
2. 保存当前状态（PC、原因）
3. 跳转到新地址
4. 可以复用大部分硬件

**Q4: 为什么说流水线提高了吞吐量但不减少延迟？**

A: 
- **吞吐量**：每周期完成的指令数增加（理想情况1条/周期）
- **延迟**：单条指令从开始到完成的时间不变（5个周期）
- 流水线是"并行处理多条指令"，不是"加速单条指令"

**Q5: 什么是精确异常？为什么重要？**

A: 
- **精确异常**：异常发生时，之前的指令都已完成，之后的指令都未执行
- **重要性**：简化异常处理程序的逻辑，保证程序语义正确
- **实现难度**：乱序执行、多发射等优化使得精确异常难以维护

## 二十一、公式速查总表

### 数据冒险

```
EX转发：EX/MEM.Rd = ID/EX.Rs 且 RegWrite=1 且 Rd≠0
MEM转发：MEM/WB.Rd = ID/EX.Rs 且 RegWrite=1 且 Rd≠0 且 EX转发不成立
Load-Use：ID/EX.MemRead=1 且 ID/EX.Rd = IF/ID.Rs
```

### 性能指标

```
CPI = 理想CPI + 流水线停顿 + 内存停顿
IPC = 1/CPI（当CPI < 1时使用IPC）
ILP = 指令级并行度
```

### 流水线效率

```
吞吐量 = IPC × 时钟频率
效率 = 实际IPC / 峰值IPC
```
