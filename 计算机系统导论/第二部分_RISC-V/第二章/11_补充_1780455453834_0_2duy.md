# 0x0B 逻辑操作的应用场景

## 1. AND 操作 —— 掩码（Mask）

**核心用途**：提取特定比特位，将不需要的位清零。

**原理**：1 & x = x（保留），0 & x = 0（清除）。

**示例**：
```asm
andi x9, x10, 0xFF   # 只保留低8位，高位清零（提取一个字节）
and  x9, x10, x11    # x11 作为掩码，x10 中对应 x11 为1的位保留
```

**典型场景**：
- 提取寄存器的低 n 位
- 判断某个位是否为 1（掩码后判零）
- 对齐地址（如清除低3位实现 8 字节对齐）

---

## 2. OR 操作 —— 置位（Set Bits）

**核心用途**：将特定比特位置 1，其他位保持不变。

**原理**：0 | x = x（不变），1 | x = 1（置位）。

**示例**：
```asm
ori  x9, x10, 0x01   # 将最低位置 1
or   x9, x10, x11    # x11 中为1的位，在 x9 中也置1
```

**典型场景**：
- 设置标志位（Flag）
- 合并两个位域（Bit Field）
- 权限控制中开启某些权限位

---

## 3. XOR 操作 —— 差分 / 取反

**核心用途**：
1. **差分比较**：两个数 XOR，结果为 0 表示完全相同。
2. **按位取反**：与全 1 异或实现 NOT。

**原理**：
- 相同位 → 0
- 不同位 → 1

**示例**：
```asm
xori x9, x10, -1     # NOT 操作：所有位取反（-1 即全1）
xor  x9, x10, x11    # 差分：x9 中为1的位表示 x10 和 x11 在此处不同
```

**典型场景**：
- 快速判断两数是否相等（判零）
- 数据加密/校验（简单异或校验）
- 切换标志位（Toggle：x ^ 1 实现 0↔1 翻转）

---

## 4. 移位操作的应用

| 指令 | 用途 | 数学意义 |
|:---:|:---|:---|
| `slli` | 左移 i 位 | 乘以 2^i |
| `srli` | 逻辑右移 i 位 | 无符号除以 2^i |
| `srai` | 算术右移 i 位 | 有符号除以 2^i（符号位保留） |

**注意**：RISC-V 中移位指令的立即数只取低6位（64位架构），最多移 63 位。

---

## 5. 总结口诀

- **AND**：想保留谁，就给谁戴"1"面具。
- **OR**：想点亮谁，就给谁送"1"火种。
- **XOR**：找不同，全1异或即取反。

# 0x0C 字符数据与字符串操作

## 1. 字符编码体系

| 编码 | 位数 | 字符数 | 说明 |
|:---:|:---:|:---:|:---|
| **ASCII** | 7-bit | 128 | 95个可打印图形字符 + 33个控制字符（如 `\n`, `\t`） |
| **Latin-1** | 8-bit | 256 | ASCII + 96个扩展图形字符（支持西欧语言） |
| **Unicode** | 32-bit | 百万级 | 涵盖全球几乎所有文字、符号、表情 |
| **UTF-8** | 变长 (1~4字节) | - | Unicode 的变长实现，兼容 ASCII，最常用 |
| **UTF-16** | 变长 (2或4字节) | - | Java/C++ wide char 常用 |

> 💡 **RISC-V 视角**：内存中存储的是字节序列，解释方式取决于程序。RISC-V 提供 `lb`/`lbu` 来按字节访问。

---

## 2. 字符串在 RISC-V 中的表示

C 风格字符串：**以空字符 `\0`（0x00）结尾的字节数组**。

```c
void strcpy(char x[], char y[]) {
    size_t i = 0;
    while ((x[i] = y[i]) != '\0')
        i += 1;
}
```

---

## 3. RISC-V 汇编实现：`strcpy`

**假设**：`x10` = `x[]` 基地址，`x11` = `y[]` 基地址。

```asm
strcpy:
    addi sp, sp, -8      # 开辟栈帧
    sd   x19, 0(sp)      # 保存 x19（被调用者保存寄存器）

    add  x19, x0, x0     # i = 0

L1:
    add  x5, x19, x11    # x5 = y[i] 的地址
    lbu  x6, 0(x5)       # x6 = y[i]（无符号加载字节）

    add  x7, x19, x10    # x7 = x[i] 的地址
    sb   x6, 0(x7)       # x[i] = y[i]

    beq  x6, x0, L2      # 如果 y[i] == '\0'，结束循环

    addi x19, x19, 1     # i = i + 1
    jal  x0, L1          # 继续循环（无条件跳转）

L2:
    ld   x19, 0(sp)      # 恢复 x19
    addi sp, sp, 8       # 恢复栈指针
    jalr x0, 0(x1)       # 返回
```

### 关键点解析

1. **`lbu` 而非 `lb`**：字符/字节通常视为无符号数（0~255），`lbu` 零扩展到 64 位，避免负数误判。
2. **`sb` 存储**：只存低 8 位到内存，符合字符串逐字节拷贝语义。
3. **空字符检测**：`beq x6, x0` 判断是否读到 `\0`。
4. **寄存器保存**：`x19` 是保存寄存器（s-reg），被调用者必须在栈上保存并恢复。

---

## 4. 其他字符操作指令速查

| 指令 | 功能 | 场景 |
|:---:|:---|:---|
| `lbu rd, offset(rs1)` | 无符号加载字节 | 读取 ASCII/UTF-8 字符 |
| `lhu rd, offset(rs1)` | 无符号加载半字 | 读取 UTF-16 编码单元 |
| `lw rd, offset(rs1)` | 加载字 | 读取 UTF-32 码点 |
| `sb rs2, offset(rs1)` | 存储字节 | 写入字符到字符串 |
| `sh rs2, offset(rs1)` | 存储半字 | 写入 UTF-16 |

# 0x0D AUIPC、SLT 与 32 位操作指令

## 1. AUIPC —— PC 相对高位立即数

**指令格式**：`auipc rd, imm`

**功能**：
```plaintext
rd = PC + (imm << 12)
```

**作用**：将 20 位立即数左移 12 位后，与当前 PC 相加，结果存入 `rd`。

**经典组合 —— 长跳转（32 位绝对地址）**：
```asm
auipc x10, 0x12345    # x10 = PC + 0x12345000
jalr  x1, 0x678(x10)  # 跳转到 x10 + 0x678 = PC + 0x12345678
```

**应用场景**：
- **位置无关代码（PIC）**：动态链接、共享库中，代码不依赖绝对地址。
- **访问全局变量**：`auipc` 计算数据段大致地址，再用 `jalr`/`addi` 精确寻址。

---

## 2. SLT 系列 —— Set Less Than

RISC-V 原生支持 `blt`/`bge` 等分支，但某些场景需要**将比较结果作为数据保存**（如 C 的 `(a < b)` 表达式），此时用 `slt`。

### 2.1 寄存器版本

| 指令 | 功能 | 说明 |
|:---:|:---|:---|
| `slt rd, rs1, rs2` | `rd = (rs1 < rs2) ? 1 : 0` | **有符号**比较 |
| `sltu rd, rs1, rs2` | `rd = (rs1 < rs2) ? 1 : 0` | **无符号**比较 |

### 2.2 立即数版本

| 指令 | 功能 | 说明 |
|:---:|:---|:---|
| `slti rd, rs1, imm` | `rd = (rs1 < imm) ? 1 : 0` | **有符号**与 12 位立即数比较 |
| `sltiu rd, rs1, imm` | `rd = (rs1 < imm) ? 1 : 0` | **无符号**与 12 位立即数比较 |

**示例**：
```asm
slt  x5, x10, x11    # x5 = 1 if x10 < x11 (signed)
sltu x6, x10, x11    # x6 = 1 if x10 < x11 (unsigned)
```

**与 MIPS 的对比**：
- MIPS 没有原生 `blt`，只能用 `slt` + `beq/bne` 组合实现分支。
- RISC-V 直接硬件支持 `blt`/`bge`/`bltu`/`bgeu`，更高效。

---

## 3. 32 位操作指令（RV64I 特有）

在 **RV64I**（64 位 RISC-V）中，寄存器是 64 位的，但有时需要执行 **32 位运算**（如兼容 32 位数据、C 语言 `int` 类型）。

### 3.1 算术指令

| 指令 | 功能 | 结果处理 |
|:---:|:---|:---|
| `addw rd, rs1, rs2` | 32 位加法 | 低 32 位相加，**符号扩展**到 64 位 |
| `subw rd, rs1, rs2` | 32 位减法 | 同上 |
| `addiw rd, rs1, imm` | 32 位立即数加法 | 同上 |

### 3.2 移位指令

| 指令 | 功能 |
|:---:|:---|
| `sllw rd, rs1, rs2` | 32 位逻辑左移 |
| `srlw rd, rs1, rs2` | 32 位逻辑右移 |
| `sraw rd, rs1, rs2` | 32 位算术右移 |
| `slliw rd, rs1, imm` | 32 位立即数逻辑左移 |
| `srliw rd, rs1, imm` | 32 位立即数逻辑右移 |
| `sraiw rd, rs1, imm` | 32 位立即数算术右移 |

**关键特性**：
- 运算只取寄存器的低 32 位。
- 结果写入 `rd` 时，**bit 31 符号扩展到 bit 63**，保证 64 位寄存器中保存的是正确的 32 位有符号值。

**示例**：
```asm
addw x5, x10, x11    # x5 = (int32_t)(x10 + x11)
sllw x6, x10, x11    # x6 = (int32_t)(x10 << x11)
```

---

## 4. 总结速查表

| 场景 | 推荐指令 |
|:---|:---|
| 加载 32 位绝对地址 | `auipc + jalr` 或 `auipc + addi` |
| 比较结果保存到寄存器 | `slt` / `sltu` / `slti` / `sltiu` |
| 64 位系统执行 32 位 `int` 运算 | `addw` / `subw` / `addiw` |
| 64 位系统执行 32 位移位 | `sllw` / `srlw` / `sraw` / `*iw` |

# 0x0E 谬误与陷阱 (Fallacies & Pitfalls)

## 1. 谬误：复杂指令 ⇒ 更高性能

**错误认知**：指令越强大（一条指令干越多活），程序跑得越快。

**真相**：
- 复杂指令（CISC 风格）实现难度大，会拖慢**所有指令**的时钟周期。
- 现代编译器擅长用**简单指令组合**生成高效代码。
- RISC-V 的设计哲学：**简单指令 + 规整格式** → 更容易流水线化 → 更高频率。

> 💡 **结论**：指令数量少 ≠ 性能差，ISA 简洁往往意味着实现更快。

---

## 2. 陷阱：顺序字不在顺序地址

**错误认知**：数组下标递增 1，内存地址也递增 1。

**真相**：
- RISC-V 是**字节编址**，但一个 **64 位双字（doubleword）占 8 字节**。
- 访问下一个数组元素，地址应 **+8**，不是 +1。
- 同理，32 位字（word）占 4 字节，地址应 **+4**。

**示例**：
```c
long long int A[10];
// A[i] 的地址 = base + i * 8
```

```asm
slli x6, x5, 3       # x6 = i * 8（左移3位 = 乘8）
add  x7, x10, x6     # x7 = &A[i]
ld   x8, 0(x7)       # 读取 A[i]
```

> ⚠️ **常见错误**：写成 `addi x10, x10, 1` 去取下一个元素，结果只移动了 1 字节，数据完全错乱。

---

## 3. 陷阱：返回后保留指向自动变量的指针

**错误代码（C）**：
```c
void foo(int **pp) {
    int local = 10;
    *pp = &local;   // 把栈上局部变量的地址传出去
}                   // 函数返回后，栈帧被释放

void bar() {
    int *p;
    foo(&p);
    // p 现在指向已失效的栈空间！（悬垂指针）
}
```

**RISC-V 层面解释**：
- 局部变量（`local`）存放在**栈帧**中，由 `sp` 指针管理。
- `foo` 返回时执行 `addi sp, sp, N` 恢复栈指针，该栈帧空间被标记为"可重用"。
- 虽然旧数据可能暂时还在，但后续函数调用会**覆盖**这片内存。
- 读取 `*p` 得到的是垃圾值，甚至导致段错误。

> ✅ **正确做法**：如果需要返回指针，数据应放在 **堆（Heap）** 上（`malloc`/`new`），而非栈上。

---

## 4. 补充：其他常见误区

| 误区 | 纠正 |
|:---|:---|
| "加载指令 `lw` 和 `lwu` 没区别" | RV64I 中 `lw` 符号扩展，`lwu` 零扩展，处理无符号 32 位数必须用 `lwu` |
| "分支指令 `beq` 的 PC 是当前指令地址" | `beq` 使用 **PC+4**（下一条指令）作为基准，不是当前 PC |
| "移位立即数可以任意大" | `slli` 等指令的立即数只取低 6 位（64 位架构），最大移 63 位 |

---

## 5. 设计原则回顾

PPT 本章强调的 RISC-V 设计原则，也是避开陷阱的指南针：

1. **Simplicity favors regularity**（简洁造就规整）—— 所有指令 32 位，格式统一。
2. **Smaller is faster**（越小越快）—— 32 个寄存器，访问速度远快于内存。
3. **Good design demands good compromises**（好的设计需要好的权衡）—— 立即数位数有限，用 `lui`/`auipc` 补偿。
4. **Make the common case fast**（让常见情况更快）—— 小常数用 `addi`，避免访存。

# 0x0E+ 大端字节序与小端字节序 (Endianness)

## 1. 问题本质：一个字在内存中怎么摆？

多字节对象（如 32 位 `int`、64 位 `long`）在内存中占据连续多个字节，但这些字节**谁放低地址、谁放高地址**，不同架构有不同约定——这就是字节序（Endianness）。

以 32 位数 `0x01020304` 为例，从低地址到高地址：

| 字节序 | 地址 +0 | +1 | +2 | +3 | 口诀 |
|:---:|:---:|:---:|:---:|:---:|:---|
| **大端 (Big-Endian)** | `01` | `02` | `03` | `04` | 高位先来（人类阅读顺序） |
| **小端 (Little-Endian)** | `04` | `03` | `02` | `01` | 低位先来（最低有效字节在低地址） |

> 💡 **记忆法**：Big-Endian = "大端先"，最高有效字节（MSB）放在最低地址，跟人写数字的习惯一致；Little-Endian = "小端先"，最低有效字节（LSB）放在最低地址。

---

## 2. 名字由来

出自《格列佛游记》：两个阵营因为**鸡蛋应该从大头还是小头打开**而爆发战争。计算机领域借用了这个比喻——本质就是"谁放前面"的偏好之争。

---

## 3. 各架构的选择

| 架构 | 字节序 |
|:---:|:---:|
| **RISC-V** | **小端**（默认，RV64I） |
| x86 / x86-64 | 小端 |
| ARM | 双端（默认小端，可切换） |
| MIPS | 大端（传统），现代也有小端模式 |
| SPARC | 大端 |
| PowerPC | 双端 |

> ⚠️ RISC-V 默认小端，但规范允许大端实现。实际几乎所有 RISC-V 芯片都是小端。

---

## 4. 用 RISC-V 指令看字节序

假设内存地址 `0x1000` 处存放 32 位数 `0x01020304`，在小端 RISC-V 上：

```
地址:    0x1000  0x1001  0x1002  0x1003
内容:    0x04    0x03    0x02    0x01
```

执行 `lw x5, 0(x10)`（`x10 = 0x1000`）后，**`x5 = 0x01020304`**——CPU 自动按小端规则把字节拼回正确值。

执行 `lb x6, 0(x10)` 则 `x6 = 0x04`（最低地址处的字节 = 最低有效字节）。

---

## 5. 字节序什么时候会出问题？

### 5.1 网络通信

网络协议（TCP/IP）规定用**大端**传输（称为"网络字节序"）。小端机器发送数据前必须转换：

```c
uint32_t host_val = 0x01020304;
uint32_t net_val = htonl(host_val);  // 小端→大端
// net_val = 0x04030201（字节翻转后发送）
```

| 函数 | 作用 |
|:---:|:---|
| `htonl` | Host to Network, 32-bit |
| `htons` | Host to Network, 16-bit |
| `ntohl` | Network to Host, 32-bit |
| `ntohs` | Network to Host, 16-bit |

### 5.2 文件/数据跨平台

不同字节序的机器读取同一二进制文件，如果文件格式没规定字节序，就会解析出错。

经典案例：BMP 图片格式规定小端，JPEG 规定大端，PNG 用大端。

### 5.3 类型强转

```c
int x = 0x01020304;
char *p = (char *)&x;

// 小端机器上：
// p[0] = 0x04, p[1] = 0x03, p[2] = 0x02, p[3] = 0x01

// 大端机器上：
// p[0] = 0x01, p[1] = 0x02, p[2] = 0x03, p[3] = 0x04
```

通过字节指针逐个访问，不同字节序的结果不同——这类代码**不可移植**。

---

## 6. 对齐与字节序的关系

上一节（0x0E）讨论了"顺序字不在顺序地址"，核心是**字节编址**下，一个字占多个地址。

字节序则进一步回答：**占多个地址的字，内部字节怎么排列？**

```
数组 long A[2] = {0x1122334455667788, 0x99AABBCCDDEEFF00};

小端 RISC-V 内存布局：
地址:   0x00   0x01   0x02   0x03   0x04   0x05   0x06   0x07
A[0]:   88     77     66     55     44     33     22     11
地址:   0x08   0x09   0x0A   0x0B   0x0C   0x0D   0x0E   0x0F
A[1]:   00     FF     EE     DD     CC     BB     AA     99
```

注意：`ld` 读取 `A[0]` 时，CPU 从地址 `0x00~0x07` 读取 8 字节，按小端规则拼回 `0x1122334455667788`——程序员无需手动处理。

---

## 7. 总结

| 问题 | 答案 |
|:---|:---|
| 字节序决定什么？ | 多字节对象的**字节在内存中的排列顺序** |
| RISC-V 用哪种？ | **小端**（LSB 在低地址） |
| 什么时候需要关心？ | 网络通信、跨平台二进制数据、`char*` 强转访问 |
| 什么时候不需要？ | 用 `lw`/`ld`/`sw`/`sd` 正常读写，CPU 自动处理 |

> 💡 **一句话**：大端像人写字（高位在前），小端像堆栈（低位打底）。RISC-V 是小端，`lw`/`ld` 自动搞定，只有"绕过类型系统逐字节操作"时才需要操心。

# 0x0F 数组 vs. 指针遍历优化

## 1. 问题背景

C 语言中，遍历数组有两种写法：

```c
// 版本1：数组索引
void clear1(long long int array[], int size) {
    int i;
    for (i = 0; i < size; i += 1)
        array[i] = 0;
}

// 版本2：指针遍历

        *p = 0;
}
```

两者语义等价，但生成的 RISC-V 汇编**效率不同**。

## 2. 版本 1：数组索引（clear1）

**核心特征**：每次循环都需计算 `base + i * 8`。

```asm
clear1:
    li   x5, 0          # i = 0
loop1:
    slli x6, x5, 3      # x6 = i * 8（左移3位 = 乘8）
    add  x7, x10, x6    # x7 = array[i] 的地址
    sd   x0, 0(x7)      # array[i] = 0
    addi x5, x5, 1      # i++
    blt  x5, x11, loop1 # if (i < size) continue
```

**开销分析**：
- 每轮循环执行一次 `slli`（移位）和一次 `add`（地址计算）。
- 索引计算在循环内部，无法提前消除。

## 3. 版本 2：指针遍历（clear2）

**核心特征**：指针 `p` 直接就是地址，每次只需 `+8`。

```asm
clear2:
    mv   x5, x10        # p = &array[0]
    slli x6, x11, 3     # x6 = size * 8（提前计算，循环外）
    add  x7, x10, x6    # x7 = &array[size]（循环边界）
loop2:
    sd   x0, 0(x5)      # *p = 0
    addi x5, x5, 8      # p = p + 8（直接地址递增）
    bltu x5, x7, loop2  # if (p < &array[size]) continue
```

**优势分析**：
- **无循环内乘法/移位**：地址计算简化为 `+8`。
- **边界一次性计算**：`size * 8` 在循环前完成。
- **更少的指令/周期**：每轮循环省掉 `slli` 和 `add`（数组基址+偏移）。

---

## 4. 编译器优化：强度削弱

**强度削弱（Strength Reduction）**：将昂贵的运算（乘法）替换为便宜的运算（加法/移位）。

- 数组版本中的 `i * 8` 可被编译器优化为**归纳变量（Induction Variable）**：
  - 维护一个额外寄存器保存 `i * 8`，每次循环 `+8` 而非重新计算。
- 优化后的数组版本可与指针版本性能相当。

**归纳变量消除（Induction Variable Elimination）**：
- 编译器检测到 `i` 只用于计算地址时，可能直接删除 `i`，只用指针寄存器。

## 5. 对比总结

| 维度 | 数组索引 (clear1) | 指针遍历 (clear2) |
|:---:|:---|:---|
| **代码清晰度** | 高，语义直观 | 稍低，需理解指针算术 |
| **循环内计算** | `i*8 + base`（2条指令） | `p+8`（1条指令） |
| **编译器优化空间** | 大（强度削弱、归纳变量消除） | 本身已接近最优 |
| **RISC-V 指令数/轮** | 6 条（未优化） | 4 条 |
| **推荐写法** | 日常编码（相信编译器） | 手写汇编或极致优化场景 |

> 💡 **结论**：现代编译器（如 GCC）非常聪明，**手写指针遍历不一定比数组版本快**。优先写清晰的代码（数组索引），让编译器做优化。只有在手写汇编或编译器无法优化时，才手动使用指针技巧。

# 0x10 冒泡排序综合示例

## 1. C 语言源码

```c
void swap(long long int v[], long long int k) {
    long long int temp;
    temp = v[k];
    v[k] = v[k+1];
    v[k+1] = temp;
}

void sort(long long int v[], size_t n) {
    size_t i, j;
    for (i = 0; i < n; i += 1) {
        for (j = i - 1;
             j >= 0 && v[j] > v[j + 1];
             j -= 1) {
            swap(v, j);
        }
    }
}
```

---

## 2. Swap 过程（Leaf Procedure）

**参数**：`x10` = `v[]` 基地址，`x11` = `k`

```asm
swap:
    slli x6, x11, 3      # x6 = k * 8（字节偏移）
    add  x6, x10, x6     # x6 = &v[k]

    ld   x5, 0(x6)       # temp = v[k]
    ld   x7, 8(x6)       # x7 = v[k+1]

    sd   x7, 0(x6)       # v[k] = v[k+1]
    sd   x5, 8(x6)       # v[k+1] = temp

    jalr x0, 0(x1)       # return
```

**关键点**：
- **叶过程**：不调用其他函数，无需保存 `ra`。
- **地址计算**：`k * 8` 因为元素是 64 位双字。
- `v[k+1]` 的地址就是 `v[k]` 地址 `+8`。

---

## 3. Sort 过程（Non-Leaf Procedure）

**参数**：`x10` = `v[]`，`x11` = `n`  
**局部变量**：`x19` = `i`，`x20` = `j`

### 3.1 栈帧布局（需要保存 5 个寄存器）

排序过程是**非叶过程**（调用 `swap`），必须保存：
- `x1`（ra）：返回地址
- `x19` ~ `x22`：保存寄存器（s-reg），其中 `x21`/`x22` 用于传参备份

```asm
sort:
    addi sp, sp, -40     # 开辟 40 字节栈空间（5 * 8）
    sd   x1,  32(sp)     # 保存返回地址
    sd   x22, 24(sp)     # 保存 x22
    sd   x21, 16(sp)     # 保存 x21
    sd   x20, 8(sp)      # 保存 x20（j）
    sd   x19, 0(sp)      # 保存 x19（i）
```

### 3.2 外层循环（i 从 0 到 n-1）

```asm
for1tst:

exit1:
```

### 3.3 内层循环（j 从 i-1 递减到 0）

```asm
    addi x20, x19, -1    # j = i - 1
for2tst:
    blt  x20, x0, exit2  # if (j < 0) exit inner loop

    slli x5, x20, 3      # x5 = j * 8
    add  x5, x10, x5     # x5 = &v[j]
    ld   x6, 0(x5)       # x6 = v[j]
    ld   x7, 8(x5)       # x7 = v[j+1]

    ble  x6, x7, exit2   # if (v[j] <= v[j+1]) exit inner loop

    # 准备调用 swap(v, j)
    mv   x21, x10        # 备份 x10（v）到 x21
    mv   x22, x11        # 备份 x11（n）到 x22
    mv   x10, x21        # 第一个参数：v
    mv   x11, x20        # 第二个参数：j
    jal  x1, swap        # 调用 swap

    mv   x10, x21        # 恢复 x10（v）
    mv   x11, x22        # 恢复 x11（n）

    addi x20, x20, -1    # j--
    j    for2tst         # 继续内层循环
exit2:
```

### 3.4 恢复寄存器并返回

```asm
    ld   x19, 0(sp)      # 恢复 x19
    ld   x20, 8(sp)      # 恢复 x20
    ld   x21, 16(sp)     # 恢复 x21
    ld   x22, 24(sp)     # 恢复 x22
    ld   x1,  32(sp)     # 恢复返回地址
    addi sp, sp, 40      # 释放栈帧
    jalr x0, 0(x1)       # return
```

| **参数传递** | x10~x11 传参，调用前备份旧参数 |

| **条件分支** | `bge`, `blt`, `ble` 控制双层循环 |
