# RISC-V 高级特性

> 本文档整合了 RISC-V 的原子操作、逻辑操作、字符串处理、常见陷阱等高级主题。

---

## 一、同步与原子操作

### 1.1 共享内存与数据竞争

- **场景**：两个处理器（P1、P2）共享同一块内存区域，例如 P1 先写入数据，P2 再读取该数据
- **风险**：如果 P1 和 P2 之间没有同步机制，就会产生**数据竞争（Data race）**
- **后果**：程序的最终结果完全依赖于内存访问的执行顺序，导致行为不可预测

### 1.2 硬件支持的必要性

解决数据竞争需要**硬件层面的原子操作支持**：

- **原子读写内存操作**：将"读-改-写"这类复合操作封装为不可分割的最小执行单元
- **互斥保证**：在原子操作的读与写步骤之间，不允许其他处理器访问该内存位置

### 1.3 原子操作的实现形式

- **单指令实现**：通过一条专用指令完成原子操作，例如寄存器 ↔ 内存的原子交换（Atomic Swap）
- **指令对实现**：通过一对特殊指令实现，例如"加载保留/条件存储"（Load-Reserve/Store-Conditional），同样能保证操作的原子性

---

### 1.4 具体指令

#### A. 加载保留指令 `lr.d rd, (rs1)`

- **功能**：
    1. 从 `rs1` 寄存器指向的内存地址加载一个双字（doubleword）到目标寄存器 `rd`
    2. 同时在硬件层面为该内存地址**设置一个保留标记（Reservation）**，监控后续对该地址的写操作

- **本质**：开启一段"乐观读取"，并标记要保护的内存位置

#### B. 条件存储指令 `sc.d rd, (rs1), rs2`

- **功能**：
    1. 尝试将 `rs2` 寄存器的值存储到 `rs1` 指向的内存地址
    2. **成功条件**：自上次 `lr.d` 执行后，该内存地址**未被其他处理器或中断修改**
        - 成功时：存储完成，并在 `rd` 中写入 **0**
    3. **失败条件**：该内存地址在 `lr.d` 之后被修改过
        - 失败时：存储操作被**放弃**，并在 `rd` 中写入**非零值**

- **本质**：只有在"读取后未被篡改"的前提下，写入才会生效，从而保证了"读-改-写"操作的原子性

---

### 1.5 具体示例

![[370f3e7c-2aea-4f16-bbac-eb8180271c7c.jpg]]

#### Example 1: 原子交换（Atomic Swap）

用于测试并设置锁变量，实现原子的"读旧值-写新值"操作。

```asm
again:
    lr.d  x10, (x20)      # 从x20指向的地址加载旧值到x10，并标记保留
    sc.d  x11, (x20), x23 # 尝试将x23的新值存入x20地址；x11接收状态(0=成功)
    bne   x11, x0, again  # 若存储失败(x11≠0)，则重试
    addi  x23, x10, 0     # 将读取到的旧值(x10)保存到x23
```

- **核心逻辑**：通过循环重试，保证"加载-存储"操作的原子性
- **结果**：`x23` 保存了交换前的旧值，`x20` 地址被更新为 `x23` 原来的新值

#### Example 2: 自旋锁（Lock）与解锁（Unlock）

实现一个经典的**测试并设置（Test-and-Set）自旋锁**，`x20` 指向锁变量地址（0 = 未锁，1 = 已锁）。

##### 加锁流程：

```asm
    addi  x12, x0, 1      # 将锁定值(1)存入x12
again:
    lr.d  x10, (x20)      # 读取锁变量到x10，并标记保留
    bne   x10, x0, again  # 若锁已被占用(x10≠0)，则循环等待（自旋）
    sc.d  x11, (x20), x12 # 尝试将锁置为1；x11接收状态
    bne   x11, x0, again  # 若存储失败，重试
```

- **流程**：先检查锁是否为 0（未锁），若为 0 则尝试原子地将其置为 1；若失败则重试，直到成功获取锁
- **特点**：在锁被占用时，CPU 会不断循环检查，属于**忙等待（Busy Waiting）**

##### 解锁流程：

```asm
    sd    x0, 0(x20)      # 将0写入锁变量地址，释放锁
```

- **逻辑**：直接将锁变量写回 0，无需原子操作（因为只有锁持有者会执行解锁）

## 二、逻辑操作的应用场景

### 2.1 AND 操作 —— 掩码（Mask）

**核心用途**：提取特定比特位，将不需要的位清零。

**原理**：1 & x = x（保留），0 & x = 0（清除）。

**示例**：
```asm
andi x9, x10, 0xFF   # 只保留低8位，高位清零（提取一个字节）
and  x9, x10, x11    # x11 作为掩码，x10 中对应 x11 为1的位保留
```

**典型场景**：
- 提取寄存器的低 n 位
- 判断某个位是否为 1（掩码后判零）
- 对齐地址（如清除低3位实现 8 字节对齐）

---

### 2.2 OR 操作 —— 置位（Set Bits）

**核心用途**：将特定比特位置 1，其他位保持不变。

**原理**：0 | x = x（不变），1 | x = 1（置位）。

**示例**：
```asm
ori  x9, x10, 0x01   # 将最低位置 1
or   x9, x10, x11    # x11 中为1的位，在 x9 中也置1
```

**典型场景**：
- 设置标志位（Flag）
- 合并两个位域（Bit Field）
- 权限控制中开启某些权限位

---

### 2.3 XOR 操作 —— 差分 / 取反

**核心用途**：
1. **差分比较**：两个数 XOR，结果为 0 表示完全相同
2. **按位取反**：与全 1 异或实现 NOT

**原理**：
- 相同位 → 0
- 不同位 → 1

**示例**：
```asm
xori x9, x10, -1     # NOT 操作：所有位取反（-1 即全1）
xor  x9, x10, x11    # 差分：x9 中为1的位表示 x10 和 x11 在此处不同
```

**典型场景**：
- 快速判断两数是否相等（判零）
- 数据加密/校验（简单异或校验）
- 切换标志位（Toggle：x ^ 1 实现 0↔1 翻转）

---

### 2.4 移位操作的应用

| 指令 | 用途 | 数学意义 |
|------|------|----------|
| `slli` | 左移 i 位 | 乘以 2^i |
| `srli` | 逻辑右移 i 位 | 无符号除以 2^i |
| `srai` | 算术右移 i 位 | 有符号除以 2^i（符号位保留） |

**注意**：RISC-V 中移位指令的立即数只取低6位（64位架构），最多移 63 位。

---

### 2.5 总结口诀

- **AND**：想保留谁，就给谁戴"1"面具
- **OR**：想点亮谁，就给谁送"1"火种
- **XOR**：找不同，全1异或即取反

## 三、谬误与陷阱 (Fallacies & Pitfalls)

### 3.1 谬误：复杂指令 ⇒ 更高性能

**错误认知**：指令越强大（一条指令干越多活），程序跑得越快。

**真相**：
- 复杂指令（CISC 风格）实现难度大，会拖慢**所有指令**的时钟周期
- 现代编译器擅长用**简单指令组合**生成高效代码
- RISC-V 的设计哲学：**简单指令 + 规整格式** → 更容易流水线化 → 更高频率

> 💡 **结论**：指令数量少 ≠ 性能差，ISA 简洁往往意味着实现更快

---

### 3.2 陷阱：顺序字不在顺序地址

**错误认知**：数组下标递增 1，内存地址也递增 1。

**真相**：
- RISC-V 是**字节编址**，但一个 **64 位双字（doubleword）占 8 字节**
- 访问下一个数组元素，地址应 **+8**，不是 +1
- 同理，32 位字（word）占 4 字节，地址应 **+4**

**示例**：
```c
long long int A[10];
# A[i] 的地址 = base + i * 8
```

```asm
slli x6, x5, 3       # x6 = i * 8（左移3位 = 乘8）
add  x7, x10, x6     # x7 = &A[i]
ld   x8, 0(x7)       # 读取 A[i]
```

> ⚠️ **常见错误**：写成 `addi x10, x10, 1` 去取下一个元素，结果只移动了 1 字节，数据完全错乱

---

### 3.3 陷阱：返回后保留指向自动变量的指针

**错误代码（C）**：
```c
void foo(int **pp) {
    int local = 10;
    *pp = &local;   # 把栈上局部变量的地址传出去
}                   # 函数返回后，栈帧被释放

void bar() {
    int *p;
    foo(&p);
    # p 现在指向已失效的栈空间！（悬垂指针）
}
```

**RISC-V 层面解释**：
- 局部变量（`local`）存放在**栈帧**中，由 `sp` 指针管理
- `foo` 返回时执行 `addi sp, sp, N` 恢复栈指针，该栈帧空间被标记为"可重用"
- 虽然旧数据可能暂时还在，但后续函数调用会**覆盖**这片内存
- 读取 `*p` 得到的是垃圾值，甚至导致段错误

> ✅ **正确做法**：如果需要返回指针，数据应放在 **堆（Heap）** 上（`malloc`/`new`），而非栈上

---

### 3.4 设计原则回顾

PPT 本章强调的 RISC-V 设计原则，也是避开陷阱的指南针：

1. **Simplicity favors regularity**（简洁造就规整）—— 所有指令 32 位，格式统一
2. **Smaller is faster**（越小越快）—— 32 个寄存器，访问速度远快于内存
3. **Good design demands good compromises**（好的设计需要好的权衡）—— 立即数位数有限，用 `lui`/`auipc` 补偿
4. **Make the common case fast**（让常见情况更快）—— 小常数用 `addi`，避免访存

## 四、大端字节序与小端字节序 (Endianness)

### 4.1 问题本质：一个字在内存中怎么摆？

多字节对象（如 32 位 `int`、64 位 `long`）在内存中占据连续多个字节，但这些字节**谁放低地址、谁放高地址**，不同架构有不同约定——这就是字节序（Endianness）。

以 32 位数 `0x01020304` 为例，从低地址到高地址：

| 字节序 | 地址 +0 | +1 | +2 | +3 | 口诀 |
|--------|---------|-----|-----|-----|------|
| **大端 (Big-Endian)** | `01` | `02` | `03` | `04` | 高位先来（人类阅读顺序） |
| **小端 (Little-Endian)** | `04` | `03` | `02` | `01` | 低位先来（最低有效字节在低地址） |

> 💡 **记忆法**：Big-Endian = "大端先"，最高有效字节（MSB）放在最低地址，跟人写数字的习惯一致；Little-Endian = "小端先"，最低有效字节（LSB）放在最低地址

---

### 4.2 各架构的选择

| 架构 | 字节序 |
|------|--------|
| **RISC-V** | **小端**（默认，RV64I） |
| x86 / x86-64 | 小端 |
| ARM | 双端（默认小端，可切换） |
| MIPS | 大端（传统），现代也有小端模式 |
| SPARC | 大端 |
| PowerPC | 双端 |

> ⚠️ RISC-V 默认小端，但规范允许大端实现。实际几乎所有 RISC-V 芯片都是小端

---

### 4.3 字节序什么时候会出问题？

#### A. 网络通信

网络协议（TCP/IP）规定用**大端**传输（称为"网络字节序"）。小端机器发送数据前必须转换：

```c
uint32_t host_val = 0x01020304;
uint32_t net_val = htonl(host_val);  # 小端→大端
# net_val = 0x04030201（字节翻转后发送）
```

| 函数 | 作用 |
|------|------|
| `htonl` | Host to Network, 32-bit |
| `htons` | Host to Network, 16-bit |
| `ntohl` | Network to Host, 32-bit |
| `ntohs` | Network to Host, 16-bit |

#### B. 文件/数据跨平台

不同字节序的机器读取同一二进制文件，如果文件格式没规定字节序，就会解析出错。

经典案例：BMP 图片格式规定小端，JPEG 规定大端，PNG 用大端。

---

### 4.4 总结

| 问题 | 答案 |
|------|------|
| 字节序决定什么？ | 多字节对象的**字节在内存中的排列顺序** |
| RISC-V 用哪种？ | **小端**（LSB 在低地址） |
| 什么时候需要关心？ | 网络通信、跨平台二进制数据、`char*` 强转访问 |
| 什么时候不需要？ | 用 `lw`/`ld`/`sw`/`sd` 正常读写，CPU 自动处理 |

> 💡 **一句话**：大端像人写字（高位在前），小端像堆栈（低位打底）。RISC-V 是小端，`lw`/`ld` 自动搞定，只有"绕过类型系统逐字节操作"时才需要操心。
