### 📌 RISC-V 内存布局（Memory Layout）详解

这张图展示了典型 RISC-V 程序的虚拟内存分段结构，从低地址到高地址依次分为 **Reserved、Text、Static data、Dynamic data（Heap）、Stack** 五个区域，各段用途和特性如下：

---

#### 1. 各段核心功能

表格

| 段名称                        | 位置             | 核心用途        | 典型存储内容                               |
| -------------------------- | -------------- | ----------- | ------------------------------------ |
| **Reserved（保留区）**          | 最低地址（0 开始）     | 系统保留，不可访问   | 操作系统 / 硬件预留空间，用户程序无法使用               |
| **Text（代码段）**              | Reserved 之上    | 存储可执行指令     | 程序机器码（如 `.text` 段）、只读常量（部分实现）        |
| **Static data（静态数据段）**     | Text 之上        | 存储全局 / 静态变量 | C 语言全局变量、`static` 变量、字符串常量、只读数组      |
| **Dynamic data（堆 / Heap）** | Static data 之上 | 动态内存分配      | `malloc`/`new` 申请的内存，**向上生长**（地址递增）  |
| **Stack（栈）**               | 最高地址           | 自动存储管理      | 函数调用栈帧、局部变量、返回地址、临时数据，**向下生长**（地址递减） |

---

#### 2. 关键细节与寄存器关联

- **Text 段**：程序计数器 `PC` 指向该段地址，用于取指执行。
- **Static data 段**：全局指针寄存器 `x3`（`gp`）被初始化为该段基地址，通过 `±offset` 快速访问全局变量，避免长地址计算。
- **Stack 段**：栈指针 `SP`（`x2`）指向栈顶，随着函数调用 / 返回自动增减，实现自动变量管理。
- **Heap 与 Stack 相向生长**：堆向上、栈向下，中间预留大量空闲地址，避免两者地址冲突。

---

#### 3. 直观图示

- **PC**：指向 Text 段（程序代码）的当前指令地址。
- **SP**：指向 Stack 段的栈顶地址，栈向下扩展。
- 地址从下到上递增：Reserved → Text → Static data → Dynamic data → Stack。

---

#### 4. 与高级语言的对应

- **Text 段**：对应编译后的函数、语句机器码。
- **Static data 段**：对应全局变量、静态变量、字符串常量。
- **Heap 段**：对应动态分配的对象 / 数组（`malloc`/`new`）。
- **Stack 段**：对应函数局部变量、函数参数、返回地址。

---

### 💡 总结

RISC-V 内存布局是程序运行的基础：

- **代码与数据分离**：Text 段只读，避免指令被意外修改。
- **静态与动态区分**：全局 / 静态变量固定地址，动态内存按需分配。
- **栈与堆协作**：栈管理函数调用生命周期，堆处理长期动态数据。

---

### 1. 三张图的演变过程（从空到满）

#### (a) 函数刚进入：未分配局部数据

- **状态**：`FP` 和 `SP` 指向同一个地址（栈帧底部）。
- **含义**：此时栈上还没有为当前函数的局部变量、保存的寄存器预留空间。
- **对应汇编**：刚执行完 `addi sp, sp, -N` 开辟栈空间之前，或者函数入口处。

#### (b) 函数执行中：已分配局部数据

- **状态**：**`SP` 向下移动（地址变小）**，`FP` 位置固定。
- **栈结构（从上到下 / 高地址到低地址）**：
    
    1. **Saved argument registers (如果有)**：保存传入的参数寄存器（如 `a0-a7`）。
    2. **Saved return address**：保存返回地址 `ra`（`x1`）。
    3. **Saved saved registers (如果有)**：保存被调用者保存的寄存器（如 `s0-s11`）。
    4. **Local arrays and structures (如果有)**：局部数组、结构体、临时变量（栈上动态分配）。
    
- **关键逻辑**：`FP` 固定不动，通过 `FP + offset` 就能稳定访问栈上所有变量（无论 `SP` 怎么动）。

#### (c) 局部数据释放后

- **状态**：**`SP` 回到 `FP` 的位置**。
- **含义**：函数执行结束，栈上的局部数据和保存的寄存器数据被视为无效。此时恢复现场，准备 `ret` 返回。

---

### 2. 核心术语解释

表格

|术语|英文|解释|
|:--|:--|:--|
|**Local Data**|局部数据|函数内的自动变量（C 语言的 `auto`）、局部数组、结构体。**只在函数运行期间有效**。|
|**Procedure Frame**|过程帧 / 激活记录 (Activation Record)|就是这一整块栈空间。它包含了管理函数执行所需的所有信息（返回地址、保存的寄存器、局部变量）。|
|**FP**|Frame Pointer|帧指针（`x8` / `s0`）。**它是固定的锚点**，用来寻址栈上的参数和局部变量。|
|**SP**|Stack Pointer|栈指针。**它是动态的**，随着局部变量的分配（`sub`）和释放（`add`）上下移动。|

---

### 3. 为什么要这么设计？（Callee 的职责）

幻灯片提到 **"Local data allocated by callee"**，意思是：

如果在这个非叶函数（Non-Leaf）里定义了局部变量（比如 `int a[10]`），或者需要使用保存寄存器（`s` 寄存器），**必须由被调用者（callee）自己在栈上预留空间**。

### 4. 实战对应（汇编视角）

假设一个函数 `func`：

```c
void func(int a) {
    int arr[2]; // 局部数组，占 8 字节
    // 这里可能还要用 s0 寄存器
}
```

**汇编伪代码实现栈布局：**

```asm
func:
    # 1. 保存现场（如果用了 FP）
    addi sp, sp, -32     # 开辟栈帧（32字节）
    sd   ra, 24(sp)      # 保存返回地址
    sd   s0, 16(sp)      # 保存 FP (s0)
    addi s0, sp, 32      # 设置 FP = SP + 32 (确立栈底)

    # 2. 分配局部数据
    addi sp, sp, -16     # SP 继续向下移动，为 arr[2] 和其他东西留空间
                         # 此时 FP 依然不动，通过 FP 访问固定地址

    # ... 函数体执行 ...

    # 3. 恢复现场
    addi sp, sp, 16      # 释放局部数据，SP 回到 FP 位置
    ld   s0, 16(sp)      # 恢复 s0
    ld   ra, 24(sp)      # 恢复 ra
    addi sp, sp, 32      # 释放栈帧
    jr   ra              # 返回
```

---

### ✅ 总结

看这张图，只要记住一句话：

**FP 是不动的管家，负责管理整个栈房间；SP 是忙碌的服务员，负责把数据搬进来（分配）或搬出去（释放）**
