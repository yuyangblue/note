# RISC-V 处理器工作原理与程序执行

> 本文档整合了 RISC-V 程序从编译到执行的完整流程，涵盖编译链接过程和处理器流水线工作原理。

---

## 一、程序翻译与启动过程

![[Pasted image 20260331085540.png]]

### 1.1 编译流程总览

| 阶段 | 输入 | 输出 | 核心工作 |
|------|------|------|----------|
| **编译（Compiler）** | C program（C 语言源代码） | Assembly language program（汇编语言代码） | 将高级语言翻译为汇编代码 |
| **汇编（Assembler）** | 汇编语言代码 | Object: Machine language module（机器语言目标模块） | 将汇编翻译为机器码，包含未解析的符号引用 |
| **链接（Linker）** | 目标模块 + 库函数目标模块 | Executable: Machine language program（可执行文件） | 合并代码段/数据段、解析符号引用、重定位地址 |
| **加载（Loader）** | 可执行文件 | 内存中的程序 | 将可执行文件映射到进程地址空间，设置程序入口点 |

**流程简化版**：
```
C源码 → 编译 → 汇编 → 链接（静态） → 加载 → 内存执行
```

### 1.2 关键补充

- **静态链接**：库函数代码被直接打包进最终可执行文件，运行时无需依赖外部库，但文件体积更大
- **现代优化**：很多编译器直接支持 `C → Object`，省略中间汇编文件生成，提升效率

---

### 1.3 目标模块（Object Module）生成与结构解析

#### A. 核心生成逻辑

- **翻译工作**：汇编器（或编译器）将高级语言/汇编代码翻译为机器指令
- **元数据提供**：目标模块不仅包含机器码，还提供了**链接器构建完整程序所需的全部元数据**，用于将多个模块拼接为可执行文件

#### B. 目标模块核心结构

| 段/信息类型 | 作用描述 |
|-------------|----------|
| **Header（文件头）** | 描述目标模块的整体内容，包括段的大小、偏移、架构信息等，是链接器的"目录" |
| **Text segment（代码段）** | 存储翻译后的机器指令，是程序的可执行代码部分，通常为只读 |
| **Static data segment（静态数据段）** | 存储程序生命周期内分配的静态变量和全局变量，在程序运行期间一直存在 |
| **Relocation info（重定位信息）** | 记录依赖于程序加载绝对地址的内容（如函数调用、全局变量访问），供链接器进行地址修正 |
| **Symbol table（符号表）** | 记录全局变量/函数的定义，以及对外部模块符号的引用，是链接器解析符号依赖的核心 |
| **Debug info（调试信息）** | 建立机器码与源代码的映射关系，用于调试器定位源码行号、变量名等 |

#### C. 关键作用总结

目标模块是**编译与链接之间的桥梁**：
- 它将源代码转化为机器可识别的指令
- 同时保留足够的元数据，让链接器能合并多个模块、解析外部依赖、重定位地址，最终生成可执行文件

---

### 1.4 目标模块链接（静态链接）

#### A. 链接器的核心任务（生成可执行镜像）

1. **合并段（Merges segments）**
   - 将多个目标模块中同类型的段（如代码段 `.text`、数据段 `.data`）合并为一个连续的段，形成可执行文件的整体布局

2. **解析标签（Resolve labels）**
   - 为所有符号（函数名、全局变量名）分配最终的虚拟内存地址，确定它们在可执行文件中的位置

3. **修补地址依赖与外部引用（Patch location-dependent and external refs）**
   - 根据解析后的地址，修改机器指令中对局部标签和外部符号的引用，确保程序运行时能正确跳转到目标地址

#### B. 重定位加载器与虚拟内存

- **传统方式**：链接器可以保留部分地址依赖，由 **重定位加载器（relocating loader）** 在程序加载到内存时完成最终地址修正
- **虚拟内存下的优化**：
  - 虚拟内存让每个进程拥有独立的地址空间，程序可直接加载到虚拟内存的**固定绝对地址**
  - 链接器可在链接阶段完成全部地址重定位，加载时无需再修正地址，提升了加载效率

---

### 1.5 动态链接（Dynamic Linking）

#### A. 核心设计思想

- **延迟加载**：仅在库函数被调用时，才进行链接与加载操作，而非在程序启动时一次性加载所有依赖库
- **可重定位代码**：要求库过程代码本身是 **位置无关（Position-Independent Code, PIC）** 的，可在任意内存地址加载执行

#### B. 关键优势

- **减小可执行文件体积**：避免了静态链接将所有依赖库（包括间接依赖）的代码打包进可执行文件，解决了"镜像膨胀（image bloat）"问题
- **库版本自动更新**：库文件更新后，所有依赖它的程序无需重新编译链接，即可在运行时自动使用新版本库，便于 bug 修复和功能升级
- **内存共享**：多个进程可共享同一份库代码在内存中的副本，提升系统内存利用率

#### C. 动态链接 vs 静态链接 对比

| 特性 | 动态链接 | 静态链接 |
|------|----------|----------|
| 链接时机 | 运行时/调用时 | 编译时 |
| 可执行文件大小 | 小（仅存引用） | 大（包含库代码） |
| 库更新 | 自动生效，无需重编译程序 | 需重新链接程序才能使用新版库 |
| 内存占用 | 低（库代码多进程共享） | 高（每个程序副本独立） |
| 加载速度 | 启动快，首次调用库函数略有延迟 | 启动慢，运行时无额外延迟 |

---

### 1.6 惰性链接（Lazy Linkage）

![[Pasted image 20260331090944.png]]

惰性链接是动态链接的一种优化策略，核心思想是**延迟加载库函数**，直到函数第一次被调用时才完成链接和加载，以此加快程序启动速度。

#### A. 核心组件

- **间接表（Indirection table）**：也叫全局偏移表（GOT），存储函数地址的占位符，初始指向 stub 代码，第一次调用后更新为真实函数地址
- **Stub 代码**：加载当前函数的唯一 ID，然后跳转到动态链接器/加载器
- **链接器/加载器代码**：负责解析函数 ID，找到并加载对应的动态库函数，然后将间接表中的地址更新为真实函数地址
- **动态映射代码**：被加载到内存中的库函数实际代码

#### B. 执行流程

**(a) 第一次调用 DLL 函数**

1. 主程序通过 `JAL` 跳转到 stub 代码
2. Stub 加载函数 ID，通过 `JAL` 跳转到动态链接器/加载器
3. 链接器解析函数 ID，加载 DLL 函数到内存，并将间接表中的地址更新为 DLL 函数的真实地址
4. 链接器通过 `JALR` 跳转到 DLL 函数执行

**(b) 后续调用 DLL 函数**

1. 主程序通过 `JAL` 跳转到 stub 代码
2. Stub 执行 `LD` 加载间接表中已更新的真实函数地址，然后通过 `JALR` 直接跳转到 DLL 函数
3. 不再需要调用链接器，执行效率与直接调用本地函数几乎一致

#### C. 核心优势

- **启动更快**：程序启动时无需加载所有依赖库，仅加载主程序代码
- **内存高效**：未被调用的库函数不会被加载到内存
- **调用无额外开销**：第一次调用后，后续调用直接跳转至真实函数地址，性能与静态链接相当

## 二、处理器工作原理

我们可以把 RISC-V 处理器的工作拆解成**取指、译码、执行、访存、写回**五个核心阶段（经典 5 级流水线），再结合寄存器、内存和指令格式，把整个过程讲透。

### 2.1 核心硬件基础

#### 🔧 关键组件

| 组件 | 功能说明 |
|------|----------|
| **通用寄存器（x0~x31）** | 32 个快速存储单元，用于存放运算数据、地址、临时变量等 |
| **程序计数器（PC）** | 记录**下一条要执行的指令地址**，每次执行完一条指令后自动 +4（32 位指令） |
| **算术逻辑单元（ALU）** | 执行加减、与或非、移位等运算 |
| **内存（RAM）** | 存放指令和数据，通过地址访问 |
| **控制单元（CU）** | 解析指令，发出控制信号，指挥各部件协同工作 |

---

### 2.2 指令执行的 5 个阶段（流水线）

RISC-V 采用**精简指令集（RISC）**，所有指令长度固定（32 位），执行流程高度规整：

#### ① 取指（IF, Instruction Fetch）

- **动作**：
    1. PC 指向当前指令地址，从内存中取出 32 位指令
    2. PC 自动 +4，准备取下一条指令（分支/跳转时会修改 PC）

- **目的**：拿到要执行的指令机器码

#### ② 译码（ID, Instruction Decode）

- **动作**：
    1. 控制单元解析指令机器码，识别指令类型（R/I/S/B/J/U 型）
    2. 读取源寄存器（rs1/rs2）的值，解析立即数、目标寄存器（rd）等字段

- **目的**：把机器码翻译成"要做什么、用什么数据、结果放哪"

#### ③ 执行（EX, Execute）

- **动作**：
    - **R 型（运算）**：ALU 对 rs1/rs2 做加减、逻辑运算等
    - **I 型（加载/立即数运算）**：ALU 计算内存地址（rs1+imm）或执行立即数运算
    - **S 型（存储）**：ALU 计算内存地址（rs1+imm）
    - **B 型（分支）**：ALU 比较 rs1/rs2，计算跳转目标地址，决定是否修改 PC
    - **J 型（跳转）**：ALU 计算跳转目标地址，修改 PC

- **目的**：完成指令的核心计算/判断逻辑

#### ④ 访存（MEM, Memory Access）

- **动作**：
    - **加载指令（lb/lh/lw）**：从内存读取数据
    - **存储指令（sb/sh/sw）**：向内存写入数据
    - 其他指令：此阶段无操作，直接传递结果

- **目的**：和内存交互，读写数据

#### ⑤ 写回（WB, Write Back）

- **动作**：
    - 将运算结果/加载的数据写入目标寄存器 rd
    - 分支/存储指令无结果，此阶段无操作

- **目的**：把指令执行结果保存到寄存器，供后续指令使用

---

### 2.3 指令格式与硬件映射

RISC-V 所有指令都遵循固定格式，硬件可以高效译码：

| 格式 | 典型指令 | 核心字段 | 硬件行为 |
|------|----------|----------|----------|
| R-type | add / sub / xor | rd, rs1, rs2, funct3/7 | ALU 运算，结果写回 rd |
| I-type | addi / lw / jalr | rd, rs1, imm[11:0] | 立即数运算/加载地址计算/间接跳转 |
| S-type | sw | rs1, rs2, imm[11:5]/[4:0] | 计算存储地址，写入内存 |
| B-type | beq / blt | rs1, rs2, imm[12], imm[10:5], imm[4:1], imm[11] | 比较后决定是否跳转 |
| J-type | jal | rd, imm[20], imm[10:1], imm[11], imm[19:12] | 保存返回地址并跳转 |
| U-type | lui / auipc | rd, imm[31:12] | 构建 20 位高位立即数 |

---

### 2.4 函数调用与寄存器协作

以 `jal`/`jalr` 为例，展示处理器如何实现函数调用：

1. **调用**：`jal x1, func`
    - 把 `PC+4`（返回地址）写入 `x1`（ra）
    - PC 跳转到 `func` 地址，开始执行函数

2. **函数执行**：
    - 若使用保存寄存器（s-reg），先把旧值压栈保存
    - 用临时寄存器（t-reg）做计算，无需保存

3. **返回**：`jalr x0, 0(x1)`
    - 跳转到 `x1` 保存的返回地址，恢复 PC
    - 恢复保存寄存器的值，弹栈

---

### 2.5 流水线与性能优化

RISC-V 处理器通常采用**流水线设计**，让 5 个阶段重叠执行：

- 第 1 条指令在执行（EX）时，第 2 条在译码（ID），第 3 条在取指（IF）
- 理想情况下，每个周期都能完成一条指令，大幅提升吞吐量
- 遇到分支/跳转时，流水线会被"冲刷"，需要重新取指，这是性能瓶颈，现代 CPU 用**分支预测**来缓解

## 三、总结

### 3.1 程序执行全流程

```
C源码 → 编译 → 汇编 → 链接 → 加载 → 内存 → 取指 → 译码 → 执行 → 访存 → 写回 → (循环)
```

### 3.2 一句话总结 RISC-V 工作逻辑

**处理器不断从内存取指令 → 解析指令要做什么 → 用寄存器/ALU/内存完成计算 → 把结果写回寄存器 → 重复这个过程**，所有复杂程序都是由这一套简单、规整的指令流驱动的。
