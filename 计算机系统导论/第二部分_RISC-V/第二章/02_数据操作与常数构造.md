
# RISC-V 数据操作与常数构造

> 本文档整合了 RISC-V 加载存储指令和常数构造方法，涵盖从内存访问到大常数生成的完整数据操作流程。

---

## 一、加载与存储指令详解

### 1.1 核心逻辑速查表

| 指令 | 英文全称 | 数据大小 | 填充方式 (扩展) | 作用 |
|------|----------|----------|-----------------|------|
| **lb** | Load Byte | 8 bit | **有符号扩展 (Sign Extend)** | 取 1 字节，高位补符号位 |
| **lh** | Load Halfword | 16 bit | **有符号扩展** | 取 2 字节，高位补符号位 |
| **lw** | Load Word | 32 bit | **有符号扩展** | 取 4 字节，高位补符号位 |
| **lbu** | Load Byte Unsigned | 8 bit | **零扩展 (Zero Extend)** | 取 1 字节，高位补 0 |
| **lhu** | Load Halfword Unsigned | 16 bit | **零扩展** | 取 2 字节，高位补 0 |
| **lwu** | Load Word Unsigned | 32 bit | **零扩展** | 取 4 字节，高位补 0 |
| **sb** | Store Byte | 8 bit | **直接存储** | 只存最低 8 位 |
| **sh** | Store Halfword | 16 bit | **直接存储** | 只存最低 16 位 |
| **sw** | Store Word | 32 bit | **直接存储** | 存 32 位 |

---

### 1.2 加载指令 (Load) —— 从内存读数据到寄存器

RISC-V 是 64 位架构（RISC-V 64），寄存器是 64 位宽的，但内存数据可能只有 8/16/32 位。读进来后必须处理**高位**。

#### A. 有符号加载 (`lb`, `lh`, `lw`)

- **原理**：将读取的数据的 **符号位（最高位）** 复制到寄存器的所有高位上
- **目的**：保持负数的正确性。例如 `0xFF` (byte) 代表 -1，加载到 64 位寄存器应变为 `0xFFFFFFFFFFFFFFFF`
- **例子**：
```assembly
lb  t0, 0(a0)   # 从内存取 1 字节，如果最高位是 1，t0 高位全填 1
```

#### B. 无符号加载 (`lbu`, `lhu`, `lwu`)

- **原理**：直接把寄存器的**高位全部补 0**
- **目的**：用于处理字符、数组索引、地址等非负数
- **例子**：
```assembly
lbu t1, 1(a0)   # 取 1 字节，高位全填 0 (0~255)
lwu t2, 4(a0)   # 取 4 字节，高位全填 0 (0~4294967295)
```

---

### 1.3 存储指令 (Store) —— 从寄存器存数据到内存

存储时不需要扩展，直接截断。

#### 存储 (`sb`, `sh`, `sw`)

- **原理**：只取寄存器**最右边**对应的位数写入内存，其余位忽略
- **例子**：
```assembly
sb  ra, 0(sp)  # 存 ra 寄存器的最低 8 位到内存
sh  ra, 0(sp)  # 存 ra 寄存器的最低 16 位到内存
sw  ra, 0(sp)  # 存 ra 寄存器的最低 32 位到内存
```

---

### 1.4 为什么要区分 `lw` 和 `lwu`？（易错点）

在 **RISC-V 64** 中：

- **`lw`**：加载 32 位并**符号扩展**到 64 位。如果你的数据本来是无符号的 32 位大数，用 `lw` 会导致符号位误判（变成负数）
- **`lwu`**：加载 32 位并**零扩展**到 64 位。**这是访问 32 位无符号整数的正确指令**

---

### 1.5 指令格式

所有这些指令都属于 **I-type (加载)** 或 **S-type (存储)** 格式：

- **加载 (I-type)**: `lb rd, offset(rs1)`
    - `rs1`: 基地址寄存器
    - `rd`: 目标寄存器（结果存这里）
    - `offset`: 12 位立即数偏移

- **存储 (S-type)**: `sb rs2, offset(rs1)`
    - `rs1`: 基地址寄存器
    - `rs2`: 源寄存器（要存的数据）
    - `offset`: 12 位立即数偏移

## 二、构造大常数

![[Pasted image 20260330204015.png]]

### 2.1 为什么需要 `lui`？

RISC-V 的常规 I-type 指令只有 **12 位立即数**，范围仅为 ±2048，根本放不下完整的 32 位常数。

因此，构造任意 32 位常数必须分两步：

- **`lui`**：负责填充高 20 位
- **`addi`**：负责填充低 12 位

**20 + 12 = 32 位**

---

### 2.2 `lui` 指令（Load Upper Immediate）

```asm
lui rd, immediate
```

**作用：**

- 把 **20 位无符号立即数** 放到寄存器的 **bit[31:12]**
- **bit[11:0] 全部清零**
- 64 位架构下，自动将 **bit[31]** 符号扩展到 **bit[63:32]**

**一句话：`lui` = 把常数放到高 20 位，低 12 位归零。**

---

### 2.3 示例：`lui` 单独执行

```asm
lui x19, 0x3D0       # 0x3D0 = 976
```

执行后，寄存器 `x19` 的值为：

```plaintext
0x00000000_003D0000
```

---

### 2.4 配合 `addi` 构建完整 32 位常数

```asm
lui  x19, 0x3D0       # x19 = 0x003D0000
addi x19, x19, 0x80   # 0x80 = 128
```

现在：

```plaintext
x19 = 0x003D0000 + 0x80 = 0x003D0080
```

最终效果：寄存器 `x19` 中存放了完整的 32 位常数 `0x0003D0080`。

---

### 2.5 注意：`addi` 的立即数是有符号的

`addi` 的 12 位立即数是**有符号数**（范围：-2048 ～ +2047）。

如果目标常数的低 12 位 **≥ 0x800**（例如 `0xF00`），直接相加会产生符号扩展错误：

```asm
lui  x19, 0x3D0       # x19 = 0x003D0000
addi x19, x19, 0xF00   # 错误！0xF00 被当作 -256，结果变成 0x003CF000
```

**正确做法**：调整 `lui` 的高位，让 `addi` 使用一个正数立即数：

```asm
lui  x19, 0x3D1       # x19 = 0x003D1000
addi x19, x19, -0x100  # 0x100 = 256，最终 0x003D1000 - 0x100 = 0x003D0F00
```

或者更简单地，直接使用伪指令 **`li`**，让汇编器自动处理：

```asm
li x19, 0x3D0F00      # 汇编器自动拆成合适的 lui + addi 组合
```

## 三、数据操作总结

### 3.1 加载存储口诀

- **加载看后缀**：带 `u` 的高位补 0，不带 `u` 的高位补符号
- **存储看宽度**：`sb`(8) / `sh`(16) / `sw`(32)，存多少取多少

### 3.2 常数构造步骤

| 步骤 | 指令 | 作用 | 覆盖位 |
|------|------|------|--------|
| 1 | `lui rd, imm20` | 填充 bit[31:12]，低 12 位清零 | 高 20 位 |
| 2 | `addi rd, rd, imm12` | 填充 bit[11:0] | 低 12 位 |
| 结果 | — | 完整 32 位常数 | 32 位 |

**核心原因**：单条指令的 12 位立即数太小，必须 `lui + addi` 配合才能构造任意 32 位常数。

### 3.3 实际应用场景

| 场景 | 推荐指令组合 | 说明 |
|------|--------------|------|
| 加载小常数 | `li rd, imm` | 汇编器自动选择最优方式 |
| 加载大常数 | `lui + addi` | 手动控制高位和低位 |
| 加载全局变量地址 | `lui + lw` | 基址+偏移访问 |
| 函数调用 | `auipc + jalr` | 位置无关代码 |
| 动态链接 | `auipc` | 相对地址计算 |
