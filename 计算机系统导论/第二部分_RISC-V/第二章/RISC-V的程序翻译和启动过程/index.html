    1. 编译阶段（Compiler）¶

    • 输入：C program（C 语言源代码）
    • 输出：Assembly language program（汇编语言代码）
    • 补充：现代编译器（如 GCC）也可以直接生成目标文件（Object module），跳过显式的汇编步骤。

    2. 汇编阶段（Assembler）¶

    • 输入：汇编语言代码
    • 输出：Object: Machine language module（机器语言目标模块，包含未解析的符号引用）

    3. 链接阶段（Linker）—— 静态链接（Static linking）¶

    • 输入：

      • 自定义代码生成的目标模块
      • 库函数目标模块（Object: Library routine，如标准库）

    • 输出：Executable: Machine language program（可执行文件，包含完整的机器码和 resolved 符号）

    • 核心工作：合并代码段 / 数据段、解析符号引用、重定位地址。

    4. 加载阶段（Loader）¶

    • 输入：可执行文件
    • 输出：将可执行文件加载到 Memory（内存）中，准备让 CPU 执行。
    • 核心工作：将可执行文件的段映射到进程地址空间，设置程序入口点。

    💡 关键补充¶

    • 静态链接：库函数代码被直接打包进最终可执行文件，运行时无需依赖外部库，但文件体积更大。

    • 流程简化版：

      1
      C源码 → 编译 → 汇编 → 链接（静态） → 加载 → 内存执行

    • 现代优化：很多编译器直接支持 C → Object，省略中间汇编文件生成，提升效率。

    a.目标模块（Object Module，编译生成的）生成与结构解析¶

    1. 核心生成逻辑¶

    • 翻译工作：汇编器（或编译器）将高级语言 / 汇编代码翻译为机器指令。
    • 元数据提供：目标模块不仅包含机器码，还提供了链接器构建完整程序所需的全部元数据，用于将多个模块拼接为可执行文件。

    2. 目标模块核心结构¶

    段 / 信息类型 作用描述
    Header（文件头） 描述目标模块的整体内容，包括段的大小、偏移、架构信息等，是链接器的 “目录”。
    Text segment（代码段） 存储翻译后的机器指令，是程序的可执行代码部分，通常为只读。
    Static data segment（静态数据段） 存储程序生命周期内分配的静态变量和全局变量，在程序运行期间一直存在。
    Relocation info（重定位信息） 记录依赖于程序加载绝对地址的内容（如函数调用、全局变量访问），供链接器进行地址修正。
    Symbol table（符号表） 记录全局变量 / 函数的定义，以及对外部模块符号的引用，是链接器解析符号依赖的核心。
    Debug info（调试信息） 建立机器码与源代码的映射关系，用于调试器定位源码行号、变量名等。

    3. 关键作用总结¶

    目标模块是编译与链接之间的桥梁：

    • 它将源代码转化为机器可识别的指令。
    • 同时保留足够的元数据，让链接器能合并多个模块、解析外部依赖、重定位地址，最终生成可执行文件。

    b.目标模块链接（Linking Object Modules）(静态)¶

    . 链接器的核心任务（生成可执行镜像）¶

    1. 合并段（Merges segments）

      将多个目标模块中同类型的段（如代码段 .text、数据段 .data）合并为一个连续的段，形成可执行文件的整体布局。 2. 解析标签（Resolve labels）

      为所有符号（函数名、全局变量名）分配最终的虚拟内存地址，确定它们在可执行文件中的位置。 3. 修补地址依赖与外部引用（Patch location-dependent and external refs）

      根据解析后的地址，修改机器指令中对局部标签和外部符号的引用，确保程序运行时能正确跳转到目标地址。

    2. 重定位加载器与虚拟内存¶

    • 传统方式：链接器可以保留部分地址依赖，由 重定位加载器（relocating loader） 在程序加载到内存时完成最终地址修正。
    • 虚拟内存下的优化：
      • 虚拟内存让每个进程拥有独立的地址空间，程序可直接加载到虚拟内存的固定绝对地址。
      • 链接器可在链接阶段完成全部地址重定位，加载时无需再修正地址，提升了加载效率。

    另一种链接方式---动态链接（Dynamic Linking）¶

    1. 核心设计思想¶

    • 延迟加载：仅在库函数被调用时，才进行链接与加载操作，而非在程序启动时一次性加载所有依赖库。
    • 可重定位代码：要求库过程代码本身是 位置无关（Position-Independent Code, PIC 的，可在任意内存地址加载执行。

    2. 关键优势¶

    • 减小可执行文件体积：避免了静态链接将所有依赖库（包括间接依赖）的代码打包进可执行文件，解决了 “镜像膨胀（image bloat）” 问题。
    • 库版本自动更新：库文件更新后，所有依赖它的程序无需重新编译链接，即可在运行时自动使用新版本库，便于 bug 修复和功能升级。
    • 内存共享：多个进程可共享同一份库代码在内存中的副本，提升系统内存利用率。

    📊 动态链接 vs 静态链接 对比¶

    表格

    特性 动态链接 静态链接
    链接时机 运行时 / 调用时 编译时
    可执行文件大小 小（仅存引用） 大（包含库代码）
    库更新 自动生效，无需重编译程序 需重新链接程序才能使用新版库
    内存占用 低（库代码多进程共享） 高（每个程序副本独立）
    加载速度 启动快，首次调用库函数略有延迟 启动慢，运行时无额外延迟

    惰性链接（Lazy Linkage）¶

    ![[Pasted image 20260331090944.png]]

    惰性链接是动态链接的一种优化策略，核心思想是延迟加载库函数，直到函数第一次被调用时才完成链接和加载，以此加快程序启动速度。

    核心组件¶

    • 间接表（Indirection table）：也叫全局偏移表（GOT），存储函数地址的占位符，初始指向 stub 代码，第一次调用后更新为真实函数地址。
    • Stub 代码：加载当前函数的唯一 ID，然后跳转到动态链接器 / 加载器。
    • 链接器 / 加载器代码：负责解析函数 ID，找到并加载对应的动态库函数，然后将间接表中的地址更新为真实函数地址。
    • 动态映射代码：被加载到内存中的库函数实际代码。

    执行流程¶

    (a) 第一次调用 DLL 函数¶
    1. 主程序通过 JAL 跳转到 stub 代码。
    2. Stub 加载函数 ID，通过 JAL 跳转到动态链接器 / 加载器。
    3. 链接器解析函数 ID，加载 DLL 函数到内存，并将间接表中的地址更新为 DLL 函数的真实地址。
    4. 链接器通过 JALR 跳转到 DLL 函数执行。
    (b) 后续调用 DLL 函数¶
    1. 主程序通过 JAL 跳转到 stub 代码。
    2. Stub 执行 LD 加载间接表中已更新的真实函数地址，然后通过 JALR 直接跳转到 DLL 函数。
    3. 不再需要调用链接器，执行效率与直接调用本地函数几乎一致。

    核心优势¶

    • 启动更快：程序启动时无需加载所有依赖库，仅加载主程序代码。
    • 内存高效：未被调用的库函数不会被加载到内存。
    • 调用无额外开销：第一次调用后，后续调用直接跳转至真实函数地址，性能与静态链接相当。
