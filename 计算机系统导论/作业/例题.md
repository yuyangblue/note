
## 相似例题练习

### 例题1：矩阵局部性（类似5.1）

以下C代码中，同一行元素连续存储，每个字为64位整数：

```c
for (I=0; I<16; I++)
  for (J=0; J<4000; J++)
    A[I][J] = B[J][I] + A[I][J];
```

**问题：**
1. 在一个32字节的Cache块中可以存放多少个64位整数？
2. 对哪个变量的访问显示了时间局部性？对哪个显示了空间局部性？
3. 如果改用Matlab（列优先存储），对哪个变量的访问显示了空间局部性？
4. 假设Cache块大小为16字节，存储所有被访问的矩阵元素需要多少个Cache块？（C存储 vs Matlab存储）

---

**参考答案：**

1. 32B ÷ 8B = **4个**
2. **空间局部性**：`A[I][J]`——内层循环J递增，C行优先，A[I][J]和A[I][J+1]在同一行连续存储，同一Cache块
   **时间局部性**：不明显（此题没有反复访问同一元素的变量；如果外层循环有复用才算）
3. Matlab列优先：`B[J][I]`显示空间局部性——J递增时B[J][I]和B[J+1][I]在同一列，列优先下连续
4. 被访问的元素：
   - A: 16×4000 = 64000个元素（整个矩阵都被访问）
   - B: 4000×16 = 64000个元素（整个矩阵都被访问）
   - 总元素 = 128000个，每块放2个（16B÷8B=2）
   - 需要的块数 = 128000÷2 = **64000块**（C存储和Matlab存储访问的元素总数相同，只是布局不同）
   - ⚠️ 如果题目问的是"覆盖所有被访问地址所需的Cache块数"而非"同时存储"，则C行优先和Matlab列优先结果一样，因为访问的总元素集合相同

### 例题2：直接映射Cache地址分析（类似5.2）

字地址序列：0x05, 0x12, 0x07, 0x04, 0x13, 0x06, 0x14, 0x05

**5.2-A**：16块、块大小1字的直接映射cache

| 字地址 | 二进制(8位) | Tag | Index | Hit/Miss |
|--------|------------|-----|-------|----------|
| 0x05 | 0000 0101 | 0 | 5 | Miss |
| 0x12 | 0001 0010 | 1 | 2 | Miss |
| 0x07 | 0000 0111 | 0 | 7 | Miss |
| 0x04 | 0000 0100 | 0 | 4 | Miss |
| 0x13 | 0001 0011 | 1 | 3 | Miss |
| 0x06 | 0000 0110 | 0 | 6 | Miss |
| 0x14 | 0001 0100 | 1 | 4 | Miss（Index=4被0x04的Tag=0占，Tag不同→替换） |
| 0x05 | 0000 0101 | 0 | 5 | **Hit** |

Miss率 = 7/8 = 87.5%

**5.2-B**：8块、块大小2字的直接映射cache（Index = 块地址 mod 8，块地址 = 字地址÷2）

| 字地址 | 二进制(8位) | 块地址 | Tag | Index | Hit/Miss |
|--------|------------|--------|-----|-------|----------|
| 0x05 | 0000 0101 | 2 | 0 | 2 | Miss |
| 0x12 | 0001 0010 | 9 | 1 | 1 | Miss |
| 0x07 | 0000 0111 | 3 | 0 | 3 | Miss |
| 0x04 | 0000 0100 | 2 | 0 | 2 | **Hit**（与0x05同块，0x05加载时已带入0x04） |
| 0x13 | 0001 0011 | 9 | 1 | 1 | **Hit**（与0x12同块） |
| 0x06 | 0000 0110 | 3 | 0 | 3 | **Hit**（与0x07同块） |
| 0x14 | 0001 0100 | 10 | 1 | 2 | Miss（Index=2被Tag=0占，替换） |
| 0x05 | 0000 0101 | 2 | 0 | 2 | Miss（Index=2被0x14的Tag=1占，又被替换了） |

Miss率 = 5/8 = 62.5%

> 💡 关键观察：块大小从1字→2字，利用了空间局部性（0x05和0x04同块、0x12和0x13同块），命中从1次→3次。但0x05第二次访问因0x14冲突而miss——**块变大也意味着块数变少，冲突变多**。

### 例题3：Cache总位数计算（类似5.3）

直接映射cache，字节寻址，64位地址和字。

**5.3-A**：块大小4个字的16KiB cache

```
块大小 = 4字 × 8B = 32B
块数 = 16KiB ÷ 32B = 16384 ÷ 32 = 512块

地址划分：
Offset = log₂(32) = 5位
Index = log₂(512) = 9位
Tag = 64 - 9 - 5 = 50位

每行 = Tag(50) + Valid(1) + 数据(32×8=256) = 307位
总位数 = 307 × 512 = 157184位
```

**5.3-B**：块大小8个字的32KiB cache

```
块大小 = 8字 × 8B = 64B
块数 = 32KiB ÷ 64B = 32768 ÷ 64 = 512块

地址划分：
Offset = log₂(64) = 6位
Index = log₂(512) = 9位
Tag = 64 - 9 - 6 = 49位

每行 = Tag(49) + Valid(1) + 数据(64×8=512) = 562位
总位数 = 562 × 512 = 287744位
```

> 💡 对比5.3-A和5.3-B：数据量翻倍（16→32KiB），但总位数从157184→287744，只增加了83%而非100%，因为**Tag+Valid的元数据开销被更大的数据块分摊了**（每行数据从256位→512位，但元数据从51位→50位，几乎不变）。

### 例题4：写策略带宽计算（类似5.7）

| 参数 | 值 |
|------|-----|
| 每1000条指令数据读次数 | 300 |
| 每1000条指令数据写次数 | 150 |
| 指令Cache失效率 | 0.5% |
| 数据Cache失效率 | 3% |
| 块大小 | 32字节 |
| 读/写操作宽度 | 32位 |
| 基础CPI | 2 |

**5.7-A**：写直达 + 写分配

```
读内存（取块）：
  I-cache miss: 1000 × 0.5% × 32B = 160B
  D-cache 读miss: 300 × 3% × 32B = 288B
  D-cache 写miss（写分配取块）: 150 × 3% × 32B = 144B
  读内存小计 = 592B

写内存（写直达）：
  每次写直写: 150 × 4B = 600B

总带宽 = 592 + 600 = 1192B
BPC = 1192 ÷ (1000 × 2) = 0.596 BPC
```

**5.7-B**：写回 + 写分配，脏块比例25%

```
读内存（取块）= 592B（同上）

写内存（脏块写回）：
  D-cache 总miss = 300×3% + 150×3% = 9 + 4.5 = 13.5次
  脏块写回 = 13.5 × 25% × 32B = 108B

总带宽 = 592 + 108 = 700B
BPC = 700 ÷ 2000 = 0.35 BPC
```

> 💡 对比：写回比写直达节省约41%带宽（0.596→0.35），因为写直达每次写都打内存（150次×4B=600B），而写回只在驱逐脏块时才写回（108B），差距主要来自这部分。
